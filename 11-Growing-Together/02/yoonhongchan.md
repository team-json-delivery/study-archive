# 달인이 되는 비결
- 자신이 10년 넘게 해온 것 중에 전문성이 실제로 높아진 역량과 변화가 없는 것의 차이는 무엇일까?
##### 동기가 부족하다
- 더 잘하고자 하는 동기가 없기에 실력이 제자리 -> 동기가 존재하더라도 실력이 개선되지 않는 경우도 많음
##### 피드백을 제때 받지 못한다
- 정확한 피드백을 제때 받지 못하기에 실력이 제자리
  - 내가 뭘 잘했고, 잘못했는지 알지 못하고 실력도 늘지 않음
- 꾸준한 반복으로 달인이 되려면?
  - 실력을 개선하려는 동기가 필요
  - 구체적인 피드백을 적절한 시기에 수신

* “더 잘하고 싶다”는 마음이 꺼진 순간은 언제였나? 그 트리거는 무엇이었나?
* 각 팀의 피드백 루프(코드리뷰/배포/사용자 반응) 중 가장 느린 고리는 어디?

# 수십 년 동안 전문가가 안 되는 비결
##### 전문성 형성에서 타당성과 피드백의 중요성
- 믿을 수 있는 직관이 형성되려면 특정 조건이 필요
  - 타당성
    - 타당성이 필요하다는 의미는 직관이 적용되는 영역에 어느정도 인과관계와 규칙성이 존재
  - 피드백
    - 피드백이 필요하다는 의미는 자신이 내린 직관적 판단에 대해 빨리 피드백을 받고 이를 통해 학습할 기회가 주어지는 환경이 갖춤 필요
- 수십년 동안 한가지 일을 하면서 전문가가 안되는 비결은? 타당성과 피드백이 부족한 환경에서 일하는 것
- 소프트웨어 개발은 전문성이 발전될 수 있는 직업과 그렇지 않은 직업 사이
  - 타당성과 피드백이 부족한 환경에서는 오래 일해도 전문성이 신장되지 않음
##### 타당성과 피드백을 높이기
- 일하는 방식, 개발하는 방식을 바꾸면 이 타당성과 피드백을 어느정도 높일 수 있음
  - 타당성을 높이려면 변수를 제한하고 실험하면서 규칙성과 인과관계를 찾으려고 노력 필요
  - 피드백을 높이려면 동료나 상사, 고객에게서, 혹은 내가 개발하는 프로그램에서 직접 피드백을 적극적으로 구하면 됨

* 최근 직관이 빗나간 사례가 있나? 그때 피드백을 언제, 어떻게 받았나?

# 당신이 제자리 걸음인 이유
- 실력을 높이기 위해서는 '의도적 수련'이 중요
  - 양적인 부분(1만 시간 법칙) 외에도 어떤 조건을 갖춰야 의도적 수련인지, 더 효과적인지 알 필요가 있음
##### 의도적 수련의 필수조건, 적절한 난이도
- 의도적 수련이 되려면 나의 실력과 작업의 난이도가 비슷해야함 -> 적절한 난이도
  - A영역: 나의 실력 > 작업 난이도: 지루함을 느낌
  - B영역: 나의 실력 < 작업 난이도: 불안함이나 두려움을 느낌
  - C영역: 나의 실력 == 작업 난이도: 몰입과 함께 학습 능력 최대와 행복감 수반
##### 실력이 늘지 않는 이유
- 의도적 수련의 필수 요건 중 하나가 '적절한 난이도'
  - A영역 또는 B영역 일때는 의도적 수련이 되지 않으며, 실력 향상에 별 도움이 되지 않음
- 업무 시간 중 불안함이나 지루함을 느끼는 때가 대부분이라면. 실력이 도무지 늘지 않는 환경에 있는 것
  - `문제는 익숙해지고 행동이 습관화 되면서, 인식도 잘 되지 않음`
- 뛰어난 선수는 자기 기량보다 어려운 기술을 연마하지만, 그러지 못한 선수는 이미 잘하는 걸 더 연습한다
  - `A 영역이 제공해주는 안전지대 안에 머무르고, 지루함 속에서 자기 실력에 대해 안심하고 그 상태에 익숙해지는 것`
##### 제자리걸음에서 벗어나기
- 나에게 적합한 업무가 들어오지 않으면? 다음과 같은 전략을 써서 C영역(나의 실력 == 작업 난이도) 들어 와야함
##### 지루함을 느끼는 경우: a1 실력 낮추기
- 작업 난이도는 그대로 두고 실력을 낮추는 전략
  - 같은 난이도의 체력 훈련을 하는데, 팔과 다리에 모래주머니를 달고 운동하는 경우
- 평상시 즐겨 쓰던 보조 도구를 일부러 안 쓰는 것
  - 마우스를 즐겨쓰면 키보드로만 개발, 디버그를 늘 쓰는 경우 디버거를 안쓰는 것 -> 실력이 떨어진 느낌
- 지루하던 작업이 몰입하는 작업이 되고 실력도 늘 수 있음
##### 지루함을 느끼는 경우: a2 난이도 높이기
- 실력은 그대로 두고 난이도를 높이는 전략
  - 자신만의 제약을 추가하는 것 -> 같은 기간에 해야할 일의 양 늘리는 것
- 뛰어난 프로그래머들이 이미 많이 쓰고 있음
  - 하루 만에 개발하라고 주어진 업무를 한시간 만에 할 수 있는 방법 고안
  - 공시적으로 안해도 되는 업무를 자신의 의지로 추가 -> 리팩토링 또는 자동화 테스트 등
- 자신만의 도구 만드는게 중요 -> 남들보다 일을 효율적/효과적으로 하기 위해 나만의 도구/방법
##### 불안함을 느끼는 경우: b2 실력 높이기
- 실력을 높여서 몰입 영역으로 들어가는 전략
- 당장 불안감을 느낄때는? 사회적 접근, 도구적 접근 및 내관적 접근 존재
  - 사회적 접근: 나보다 뛰어난 전문가의 도움 받기
    - 잘하는 사람한테 가서 페어 프로그래밍
    - 튜토리얼 문서가 있다면 문서에서 보여주는 진행 순서 따라가 보기
  - 도구적 접근: 다른 도구의 도움을 받는 것
    - 내 능력을 확장시켜 줄 수 있는 도구 활용
  - 내관적 접근: 비슷한 일을 했던 경험을 머리속에서 되살려 보는 것
    - 그 일을 어떻게 했는지 떠올려 보면서 비유적으로 문제를 해결
##### 불안함을 느끼는 경우: b2 난이도 낮추기
- 불안감을 느낄 때 난이도를 낮춰서 몰입 영역으로 들어가기
  - 자신이 맡은 일의 가장 간단하면서 핵심적인 결과물을 첫번째 목표로 삼는 것
- 난이도를 낮춘 결과 학습 효과, 동기 강화, 스트레스 감소, 자기 효능감 증가 등 긍정적 효과 발생
##### 동적인 균형
- 자산의 실력이나 작업의 난이도가 계속 조금씩 요동 침
  - 자신의 엄무 처리속도가 외부적으로 문제가 되지 않는 범위 내에서 적절하게 난이도와 실력을 조정 -> 자신의 감정 상태를 살피어 전략을 활용
- 자신이 어떤 상태 인지 알아차림 필요 -> 메타인지 전략
  - 공부를 잘하는 사람들의 중요한 특징 중 하나로 꼽힘
##### 팀장이 할 수 있는 일
- 다른 사람을 관리하는 사람도 동일 내용을 응용 가능
  - 팀원들이 현재 어떤 상태를 주로 경험하고 있는지 파악하고 적절한 전략을 구사 -> 사람의 실력과 난이도에 맞는 일 분배가 쉽지 않기에
- 안티 패턴
  - 몰입 영역 밖으로 팀원을 몰아내는 행동
    - 실력보다 난이도 낮은 일을 해서 지루함을 느끼는 직원에게 스터디를 시키거나 콘퍼런스에 보내서 실력을 높이게 하는 것
    - 실력보다 높은 난이도의 일을 해서 불안감을 느끼는 직원에게 핀잔을 줘서 인지하는 실력(자기효능감)을 더 떨어뜨리는 것
- 팀장은 팀원의 상태를 파악하고 그들이 몰입으로 가게 도와주기 -> 고도의 의도적 수련 가능
##### 이소룡의 자기 혁신
- 이소룡은 '이긴. 그러나 진' 싸움을 통해 자신의 훈련 방법을 전반을 검토 -> 자신이 고안한 방법이 많은 혁신을 불러 일으킴

* 안티패턴 예시로 팀 규칙으로 바꾸자면? (A에 스터디만 시키지 않기, B에 질책 금지 등)

# Chapter 11. Improvement Loops (개선 루프)

## 1. 개선 루프란 무엇이고 왜 필요한가
- 에이전트는 **한 번 배포로 끝이 아니다**. 실제 사용 데이터를 바탕으로 계속 고쳐야 한다.
- 개선 루프 = **관측 → 문제 발견 → 개선 → 검증 → 재배포**의 순환 구조
- 10장(모니터링)이 "무엇이 잘못됐나"를 보는 단계라면, 11장은 그걸 **"어떻게 고칠까"**로 잇는 단계
- 목표: 프로덕션에서 얻은 신호(피드백·실패 사례)를 **체계적으로 제품 개선에 반영**

## 2. 피드백 파이프라인 (Feedback Pipelines)
사용자·시스템에서 나온 신호를 모아 개선으로 연결하는 흐름.

### 2-1. 이슈 탐지 (Issue Detection)
- 실패·저품질 응답·이상 패턴을 **자동/수동으로 포착**
- 소스: 사용자 피드백(👍/👎), 오류 로그, 평가 점수, 이상치 알림

### 2-2. 사람 검토 (Human Review)
- 탐지된 사례를 **사람이 확인·라벨링**해 진짜 문제인지 판단
- 애매하거나 고위험 케이스는 사람이 최종 판단 (human-in-the-loop)

### 2-3. 개선 (Refinement)
- 원인에 맞는 수정: **프롬프트 개선, 도구 보완, 라우팅 수정, 데이터 추가** 등
- "무엇을 바꿔야 이 실패가 사라지나"를 구체적 액션으로

### 2-4. 우선순위화 (Prioritization)
- 모든 이슈를 다 고칠 순 없음 → **빈도 × 영향도**로 우선순위 결정
- 자주 발생하고 피해가 큰 문제부터 처리

> 흐름: **이슈 탐지 → 사람 검토 → 개선 → 우선순위화 → (재배포)** 반복

## 3. 실험 (Experimentation) ⭐
개선안이 진짜 나은지 **안전하게 검증**하는 방법들.

### 3-1. 섀도 배포 (Shadow Deployment)
- 새 버전을 실제 트래픽에 **그림자처럼 병렬 실행**하되 결과는 사용자에게 노출하지 않음
- 위험 없이 실제 입력에 대한 동작을 비교 관찰

### 3-2. A/B 테스트
- 사용자를 그룹으로 나눠 **기존(A) vs 신규(B)**를 비교
- 지표(성공률·만족도·비용 등)로 통계적으로 우열 판단

### 3-3. 적응형 (Adaptive)
- 성과가 좋은 쪽으로 **트래픽 비중을 동적으로 조정** (예: 멀티암드 밴딧)
- 좋은 변형에 더 빨리 수렴, 손실 최소화

### 3-4. 게이팅 (Gating)
- 일정 **품질 기준(gate)을 통과해야** 다음 단계/전체 배포로 진행
- 점진적 롤아웃(canary)과 결합해 리스크 통제

## 4. 지속 학습 (Continuous Learning)
쌓인 데이터로 에이전트를 실제로 더 똑똑하게 만드는 단계.

### 4-1. 인컨텍스트 학습 (In-Context)
- 모델 가중치는 그대로 두고, **프롬프트/예시/검색 지식(RAG)**을 갱신해 개선
- 가장 빠르고 저렴, 즉시 반영 가능 (7장 비모수적 학습과 연결)

### 4-2. 오프라인 재학습 (Offline Retraining)
- 수집한 데이터로 **파인튜닝/재학습**을 배치로 수행
- 근본적 성능 향상 가능하지만 비용·시간·검증 부담

### 4-3. 온라인 강화학습 (Online Reinforcement)
- 실사용 중 **보상 신호로 지속 업데이트**
- 강력하지만 불안정·안전성 리스크 → 신중한 가드레일 필요

## 5. 핵심 정리 (Key Takeaways)
1. 에이전트는 배포 후 **관측→개선→검증→재배포** 루프로 계속 다듬어야 한다.
2. 피드백 파이프라인: **이슈 탐지 → 사람 검토 → 개선 → 우선순위화**.
3. 개선안은 **섀도/A/B/적응형/게이팅**으로 안전하게 검증 후 반영.
4. 지속 학습은 **인컨텍스트(빠름·저렴) → 오프라인 재학습 → 온라인 RL(강력·고위험)** 순으로 강도가 세진다.
5. 원칙: 우선순위(빈도×영향) 기반으로, **작고 안전한 실험**부터 점진적으로.

### 한눈에 보기
| 실험 방법 | 방식 | 강점 | 주의 |
|---|---|---|---|
| Shadow | 무노출 병렬 실행 | 위험 없이 관찰 | 결과 반영 안 됨 |
| A/B Test | 그룹 비교 | 통계적 검증 | 트래픽·기간 필요 |
| Adaptive | 동적 트래픽 배분 | 빠른 수렴·손실↓ | 구현 복잡 |
| Gating | 기준 통과 시 진행 | 리스크 통제 | 기준 설계 중요 |

| 지속 학습 | 반영 속도 | 비용/위험 |
|---|---|---|
| In-Context | 즉시 | 낮음 |
| Offline Retraining | 느림(배치) | 중간 |
| Online RL | 실시간 | 높음(불안정) |

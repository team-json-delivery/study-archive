# 8. 단일 에이전트에서 멀티 에이전트로
- 툴의 수가 늘어나고 에이전트에 맡기고 싶은 문제의 범위가 넓어 질수록 멀티에이전트 도입 -> 전체적인 성능과 신뢰성 ↑
- 언제, 어떻게 시스템에 에이전트를 추가할지 에이전트를 어떻게 조직하고 관리할지 살펴보기

# 8.1 에이전트는 몇 개나 필요할까?
- 단순 접근부터 시작하고 성능 개선이 필요할 때만 복잡성을 추가하는 것이 좋음
  - 작업의 난이도, 도구의 개수, 환경의 복합성에 따라 다름

##### 8.1.1 단일 에이전트 시나리오
- 단일 에이전트 적합 조건
  - 시스템 난이도가 높지 않은 작업
  - 제한된 수의 도구
  - 복잡성이 낮은 환경
  - 지연 시간이 특히 중요환 상황
- 장점
  - 단순성: 구현과 관리가 더 쉬움
  - 낮은 리소스 요구사항: 계산 오버헤드가 적음
  - 지연시간: 사용자에게 더 빠르게 응답
- 매우 복잡하거나 다면적 작업으로 확장하는데 한계가 있지만, 빠르게 검증하고 효율적으로 반복 가능
- 복잡성, 도구셋, 작업 조율 요구사항이 단일 에이전트가 감당할 수 있는 범위를 넘어설 때, 멀티 에이전트 시스템 전환 고려
- 도구 집합이 16개까지 확장되면 에이전트의 시스템 프롬프트에 모든 가능성을 설명해야하므로 혼란이나 최적 이하의 선택이 발생 가능
  - 멀티 에이전트 분해의 필요성 발생
- 도구와 책임의 개수가 늘어날 때 핵심 병목이 발생
  - 에이전트가 사용할 수 있는 도구수가 늘어남에 따라 성능이 떨어짐
- 멀티 에이전트로 넘어가기 전 우선 싱글 에이전트 프레임워크 안에서 확장하는 방식 고려
  - 여러 도구를 더 큰 그룹으로 캡슐화해 계층적 도구 선택 적용
  - 벡터 데이터 베이스를 활용해 시맨틱 도구 선택

##### 8.1.2 멀티 에이전트 시나리오
- 여러 에이전트가 공통 목표를 달성하기 위해 협력 필요
  - 작업이 복잡하고 다양한 도구 모음, 병렬 처리, 동적인 환경에 대한 적응력이 필요한 문제에 유리
- 구성 방식
  - 도구를 그룹한 후, 각 전문가를 위한 별도의 언어 모델 인스턴스를 바인딩
    - 에이전트 당 컨텍스트 크기가 줄여 집중도와 효율성을 높일 수 있음
    - 벙렬 처리를 가능하게 함
    - 공유 상태는 흐름 속에서 작업 인계가 끊김 없이 이루어지도록 보장
  - 슈퍼바이저 노드는 중앙 조율 역할하며 쿼리를 분석해 적절한 전문가에 라우팅
- 장점
  - 전문화
    - 에이전트에 특정 역할이나 전문 분야를 할당해 각 에이전트 강점 발휘
    - 도구와 책임을 분산하여, 단일 에이전트가 직면한 한계를 해소
  - 적응성
    - 변화하는 조건에 동적으로 대응
    - 복잡하고 예측 불가능한 시나리오에서도 효율성과 효과성을 유지
- 도전 과제
  - 에이전트가 조화롭게 작동하도록 정교한 통신, 동기화 메커니즘 필요 -> 통신 오버헤드 발생 및 추가적인 리소스 부담

##### 8.1.3 스웜
- 많은 수의 단순화 에이전트가 개별적으로 지능은 거의 없지만, 국소적인 상호작용과 단순화된 규칙을 통해 집단적으로 지능적인 창발적 행동 발생
  - 역할 할당, 중앙 집중식 조율 등 방법과 반대로 탈 중앙화와 자기 조직화를 강조
- 장점
  - 확장성
    - 느슨하게 결합 -> 수백, 수천 개의 에이전트 확장 가능
  - 견고성
    - 단일 장애지점 없음
  - 유연성
    - 목표나 환경이 바뀌는 상황에 실시간 대응 가능
  - 분산 문제 해결
    - 탐색, 모니터링, 합의 형성, 분산 검색 같은 작업은 스웜 다이나믹스로 효과적으로 해결 가능
- 집중 제어가 비현실적이거나 바람직하지 않은 환경에 효과적
  - 대규모 데이터 발견 작업
  - 여러 소스를 가로지르는 리서치
  - 분산 의사결정에 유용
- 해결 과제
  - 예측 가능성
  - 관측 가능성
  - 효율성

# 8.2 에이전트 추가 원칙
- 에이전트 기반 설계와 기능 최적화하는 가이드 라인
  - 작업 분해
    - 작업을 분해하면 각 에이전트는 특정 측면에 집중할 수 있어 책임이 단순해지고 효율성이 높아짐
    - 명확한 작업 경계는 중첩과 중복을 줄여 에이전트 기여가 가치 있게 유지 및 낭비 자제
  - 전문화
    - 각 에이전트가 고유 기능에 맞는 활동을 담당 -> 시스템 높은 정밀성과 효과 작동
    - 특정 유형의 작업을 더 잘처리할 수 있어 전체적으로 성능 및 작업속도 향상
  - 파시모니
    - 원하는 기능과 성능을 달성하는데 최소한의 에이전트가 추가하도록 권장 -> 단순성과 효율성 강조
      - 불필요한 에이전트의 난립을 피할 수 있음 -> 유지 관리 부담 증가와 잠재적인 성능 병목 방지
    - 복잡성과 관련된 위험과 비용은 최소화, 기능을 극대화하는 효율적이고 슬림한 멀티 에이전트 시스템 지향
  - 조율
    - 견고한 통신 프로토콜을 구축해 정보 공유를 효율적으로 하고 충돌 위험을 줄여야함
    - 에이전트의 작업이나 리소스 요구가 겹칠 때를 대비한 충돌 해결 프로토콜도 포함
  - 견고성
    - 장애 허용과 탄력성을 높이는데 필수 -> 연속적인 운영을 보장하는 백업을 제공
  - 효율성
    - 에이전트 추가와 그로 인해 발생하는 잠재적 복잡성과 리소스 요구 사이의 트레이프 오프 평가 기준
      - 에이전트 하나 추가 될 때마다 이점과 비용을 비교해 따져보는 것이 중요

# 8.3 멀티 에이전트 조율
- 에이전트 효과적 조율은 멀티 에이전트 시스템 성공에 매우 중요
- 사용할 수 있는 조율 전략은 여러가지 존재
##### 8.3.1 민주적 조율
- 모든 에이전트가 각자 동등한 의사결정 권한을 가지며, 행동과 솔루션에 대한 합의에 도달하는 것을 목표
  - 리더가 없고, 탈중앙화된 제어가 특징
    - 단일 장애 지점이 없음 -> 견고성
    - 에이전트들이 개방적으로 협업하면 집단 입력을 갱신하는 방식 -> 유연성
    - 동등한 발언권을 통해 공정한 결과를 도출
- 고유한 과제
  - 광범위한 통신 필요 -> 통신에 대한 상당한 오버헤드 발생
  - 느린 응답
  - 민주적 조율 프로토콜 구현은 복잡함
- 분산 센서 네트워크나 협업 로보틱스 같은 분야에 적합
##### 8.3.2 관리자 중심 조율
- 중앙집중적 접근 방식, 하나 이상의 에이전트를 관리자로 지정하여 하위 에이전트의 행동을 감독하고 지시하는 책임 부여
  - 민주적 조율보다 의사 결정이 간소화
  - 통신 경로가 단순해지고 조율 복잡성이 줄어듬
- 취약점
  - 관리자 에이전트 장애 시, 전체 시스템 장애로 발생 -> 단일 장애 지점 존재
  - 시스템이 커질수록 확장성 이슈 발생
    - 관리자가 병목현상이 발생
  - 의사결정 구조에서 적응성을 떨어뜨림
    - 하위 에이전트의 환경에서 발생하는 실시간 변화에 대해 최선의 결정을 내리기 어려움
- 제조 시스템이나 고객 지원 센터와 같은 구조적 계층적 환경에 적합
##### 8.3.3 계층형 조율
- 구조화된 계층을 통해 중앙 집중식 제어와 탈중앙화 제어 요소를 결합
  - 상위 수준 에이전트는 하위 수준 에이전트를 감독 지시
  - 하위 에이전트에 일정 수준의 자율성 부여
    - 상당한 확장성 이점 및 장애 허용성 향상
- 고유한 과제
  - 각 계층 간의 조율이 원할하도록 세심한 설계 필요 -> 시스템 설계에 대한 높은 복잡성
  - 여러 수준에 정보가 거쳐야 하기 때문에 통신 지연 및 낮은 반응성 발생
- 공급망 관리나 군사 작전처럼 규모가 크고 복잡한 시스템에 적합
##### 8.3.4 액터-크리틱 접근법
- 평가 주도 반복을 가볍게 적용한 형태
  - 액터는 답변, 계획, 행동 같은 후보 출력 생성
  - 크리틱은 사전에 정의된 평가 기준에 따라 출력을 받아들이거나 거부하는 품질 게이트 역할
  - 액터가 출력을 원하는 품질 임계값에 도달할 때까지 계속해서 후보를 생성, 크리틱이 판정
- 신뢰성과 성능을 옾이기 위해 추가 추론 사이클 사용
  - 연산 비용 증가 <-> 검색 품질 증가
- 유용한 케이스
  - 명확한 평가 기준이나 체크리스트가 존재할 때
  - 더 높은 품질이 주는 이득에 비해 추가 출력을 생성하는 비용 수용 가능할 때
  - 한 번의 시도로는 성능이 떨어지지만, 재랭킹이나 필터링을 거친 접근 방식이 더 잘 작동하는 모호거나 생성적인 작업일 때

# 8.4 에이전틱 시스템의 자동 설계(ADAS)
- 수작업으로 만든 아키텍처에서 벗어나, 스스로 설계하고 평가하며 반복적으로 개선할 수 있는 시스템
  - 에이전트 각 구성요소를 사람이 아닌 상위 수준의 메타 에이전트 탐색 알고리즘을 통해 자동으로 생성/평가/개선
- ADAS의 근본은 코드를 통해 에이전트를 정의한다는 개념
  - 메타 에이전트가 코드를 통해 에이전트를 만들고 해당 에이전트를 평가하는 방식
  - MAS 알고리즘을 통해 메타 에이전트는 설계자로서 새로운 에이전트를 정의하고 코드를 작성 -> 에이전트들에게 다양한 작업 및 테스트 실행
- 환경 세부 조건이 바뀌어도 뛰어난 성능을 발휘함 -> 크로스 도메인 전이 가능성에 작동 설계 탁월
- 상당한 잠재력이 있지만 윤리 및 기술적 측면에 대해 신중한 고려 필요 -> 자율성과 제약 사이의 균형 필요
  - 에이전트가 윤리 기준을 준수 확인 필요
  - 인간의 의도와 어긋날 수 있는 예기치 않은 행동 발생 유의
- 최소한의 인간 개입으로 스스로 적응하고 개선하며 점점 더 넓은 범위의 작업을 다룰 수 있는 미래를 시사

# 8.5 에이전트 통신 기법
- 시스템의 범위, 에이전트의 수, 지리적 분산 등에 따라 에이전트 간의 통신, 조율, 작업 흐름을 관리하는 핵심 기법과 기술 필요
##### 8.5.1 로컬 통신과 분산 통신
- 단일 디바이스나 단일 프로세스 -> 단순하고 효율적이지만 확장성 한계
  - 에이전트가 직접 호출
  - 공유 메모리
  - 메모리내 메시지큐 활용
- 에이전트 분산
  - 통신 방식은 명시적이면서 비동기
  - 장애 허용성 필요
  - 통신과 상태 관리 고도화 필요
##### 8.5.2 A2A 프로토콜
- 내부 로직이나 구현 세부 사항을 노출하지 않고, 서로 탐색하고 협업을 조율하며 구조화된 요청 교환 가능한 표준화된 크로스 플랫폼 메커니즘 제공
  - 더 복잡한 목표를 달성할 수 있도록 돕는 유망한 시도
  - HTTP 기반 전송 계층 위에서 상호 운영 -> 기존 JSON-RPC 2.0이지만 gRPC나 WebSocket 프로토콜 확장 가능
- A2A의 핵심은 에이전트 카드
  - 자신의 신원, 기능, 엔드포인트, 지원 인증 방식을 알리기 위한 json 기술서
  - 해당 카드를 통해 상대 에이전트를 탐색하고 기능을 평가하며 보안 통신 채널 협상 가능
- 더 광범위하고 정교한 문제를 해결할 수 있는 '느슨하게 결합된 동적 생태계' 지원
- 보안 분야에는 여전히 해결해야할 중대한 과제 존재
  - 견고한 권한 부여, 요청 제한, 신뢰 수립, 오남용 방지 등의 해결책은 아직 미존재

# 8.6 메시지 브로커와 이벤트 버스
- 에이전트 기반의 시스템 규모가 확장되면 point to point 통신 방식은 불안정하고 유연성이 떨어짐 -> 대안으로 메시지 브로커 or 이벤트 버스 도입
  - 결합 분리 및 공유 통신 패브릭을 통해 비동기적으로 상호작용 가능
  - 확장 가능
  - 장애 허용성
  - 관측 가능한 워크플로 확립
- 슈퍼 바이저는 공유 토픽(supply-chain-tasks)에 작업을 발행 -> 전문가는 비동기적으로 구독하여 관련 메시지만 처리
  - 독립 확장이 가능(인스턴스 증설)
  - 내결함성 확보(놓친 메시지 재생)
- 메시지 브로커
  - 아파키 카프카
  - 레디스 스트림과 래빗 MQ
  - NATS(Neural Autonomic Transport System)
    - 저지연, 고처리량 통신을 위해 설계된 경량 클라우드 네이티브 메시징 시스템
- 해결할 과제
  - 최종 일관성 이슈
  - 더 복잡한 오류 처리 필요

# 8.7 액터 프레임워크
- 메시징과 연산을 하나의 통합된 모델로 제공 <-> 메시지 버스는 비동기 이벤트 라우팅을 통해 데이터 흐름만 집중
  - 액터는 메시지 교환 및 자체 상태와 기능을 캡슐화함
  - 순차 처리를 보장하기 때문에 스레드 기반에서의 경쟁 상태나 공유 상태를 원천적으로 제거
  - 정교한 분산 처리, 복원력, 동적 확장이 필요한 상황에 탁월
- 도입 시기
  - 에이전트 수가 10~20개가 넘어갈 때
  - 지연시간 요구가 엄격할 때
- 대표 프레임워크
  - 레이
    - 파이썬 네이티브 분산 컴퓨팀 프레임워크
    - 상태 유지 및 확장 가능한 연산을 위한 액터 모델 지원
  - 올리언스
    - 가상 액터 모델 제공 (.Net 중심으로 Windows 생태계에 적합)
    - 수요에 따라 자동 생성, 일시 중단, 복구
  - 아카
    - JVM 생태계에서 널리 쓰이는 프레임워크로 JAVA와 Scala를 지원
    - 매우 높은 성능과 액터의 작업을 세밀하게 제어해야하는 장애 허용 분산 시스템 구축에 적합
# 8.8 오케스트레이션 및 워크플로 엔진
- 메시징과 에이전트 모델이 있어도 오케스트레이션은 필요
  - 작업 순서 정리
  - 재시도 처리
  - 의존성 추적
  - 에이전트 전반의 실패를 관리하는 로직
- 에이전틱 시스템에 지속성과 복구 가능성을 보장
  - 신뢰할 수 없는 외부 의존성, 실패 가능성, 장시간 실행이 프로세스에 포함될 때 탁월
  - `빠르고 위험이 낮은 실험에는 기본 스크립팅만으로도 충분`
- 템포럴은 장기 실행 작업, 재시도, 실패 복구 기능을 갖춘 지속적이고 상태 유지형 워크플로 제공
  - 장애가 발생하더라도 마지막 성공 단계에서 워크플로가 재개되도록 보장
- 아파치 에어플로로 방향성 비순환 그래프를 통해 에이전트 흐름을 조율하는 데도 사용 가능
  - 배치 작업이나 일정 기반 워크플로에 적합
# 8.9 상태와 영속성 관리
- 멀티 에이전트 시스템은 공유 상태, 에이전트 메모리, 작업별 메타데이터도 함께 관리해야함
  - 영속성, 일관성, 접근 패턴 측면에서 상당한 복잡성이 생기고 시스템 확장될 수록 문제가 심화됨

| 접근 방식                          | 장점                       | 단점                   | 적합한 용도                |
|--------------------------------|--------------------------|----------------------|-----------------------|
| 상태 저장 데이터베이스(PostgreSQL/Reids) | 유연하고 질의 가능하며 비용 효율적      | 수동 관리 필요. 일관성 문제 가능성 | 맞춤형 질의가 많은 시스템        |
| 벡터 스토어(pinecone)               | 시멘틱 검색, 확장 가능한 임베딩       | 더 높은 비용, 특수한 설정 필요   | 지식 집약적인 에이전트          |
| 오브젝트 스토리지(s3)                  | 저렴하고 대용량 데이터에 대한 영속성이 높음 | 느린 접근, 기본 제공 인덱싱 없음  | 아카이브용 출력물             |
| 상태 저장 오케이스트레이션 프레임워크           | 자동 복구, 보일러플레이트 코드가 적음    | 프레임워크 종속             | 복원력이 높고 장시간 실행되는 워크플로 |

- 선택은 필요한 메모리와 조율 방식의 특성에 따라 달라짐
  - 에피소드 메모리(수명이 짧은 작업 특화 상태)
    - 메모리 내 저장이나 최소한의 영속성만 가진 일시적 저장소로도 충분
  - 시멘틱 메모리(여러 상호작용에 걸친 장기 지식)
    - 보통 검색이나 벡터 인덱싱 기능을 갖춘 영속 저장소가 필요
  - 워크플로 내구성(프로세스 중간 장애에 대한 복원력)
    - 진행 상황과 상태를 자동으로 체크포인팅하는 템포럴이나 올리언스 같은 통합 엔진에서 가장 큰 이점을 얻음
- 영속성 설계는 개발자 노력, 성능, 영속성, 유연성 사이의 트레이드 오프를 반영

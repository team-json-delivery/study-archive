# Chapter 8. From One Agent to Many (단일 에이전트에서 멀티에이전트로)

## 1. 왜 멀티에이전트인가

- 단일 에이전트로는 한계가 있는 상황이 있다.
  - 작업이 너무 크고 복잡해 **하나의 컨텍스트/역할에 다 담기 어려움**
  - 서로 다른 **전문성(specialization)**이 필요 (예: 리서치 + 코딩 + 검수)
  - 병렬 처리로 **속도·처리량**을 높이고 싶음
- 멀티에이전트 = 여러 에이전트가 **역할을 나눠 협업**해 하나의 목표를 해결
- 핵심 발상: “한 명의 만능 일꾼” 대신 “**전문가 팀**”

## 2. 단일 vs 멀티에이전트 트레이드오프 ⭐

- **단일 에이전트 장점**: 단순·저비용·디버깅 쉬움, 컨텍스트 공유가 자연스러움
- **멀티에이전트 장점**: 전문화, 병렬성, 모듈화, 확장성, 관심사 분리
- **멀티에이전트 비용/리스크**
  - 토큰·호출 비용 증가, 지연(latency) 증가
  - 에이전트 간 **조율 오버헤드(coordination overhead)**
  - 오류 전파(한 에이전트의 실수가 전체로 번짐)
  - 디버깅·관측(observability) 난이도↑
- 원칙: **필요할 때만 멀티에이전트로**. 단일로 충분하면 단일이 낫다.

## 3. 멀티에이전트 아키텍처(토폴로지)

### 3-1. 매니저-워커 / 오케스트레이터 (Manager–Worker, Hierarchical) ⭐

- **매니저(오케스트레이터)** 에이전트가 작업을 분해해 **워커**들에게 분배, 결과를 취합
- 가장 흔하고 통제하기 쉬운 패턴 (supervisor 패턴)
- 장점: 명확한 책임·흐름 / 단점: 매니저가 병목·단일 실패점

### 3-2. 순차(Sequential / Pipeline)

- 에이전트들이 **줄지어** 처리 (A 출력 → B 입력 → C …)
- 예: 초안 작성 → 검토 → 교정
- 단점: 앞 단계 오류가 뒤로 전파

### 3-3. 병렬(Parallel)

- 독립적 하위 작업을 **여러 에이전트가 동시에** 처리 후 병합
- 속도↑, 단 결과 통합·충돌 해결 필요

### 3-4. 분산/네트워크(Decentralized, Peer-to-Peer)

- 중앙 관리자 없이 에이전트들이 **서로 직접 통신·협상**
- 유연·견고하지만 흐름 예측·디버깅이 어려움

### 3-5. 블랙보드 / 공유 메모리(Blackboard)

- 공용 작업공간(공유 상태)에 각자 읽고 쓰며 협업
- 느슨한 결합, 상태 일관성 관리가 관건

## 4. 역할과 전문화 (Roles & Specialization)

- 각 에이전트에 **명확한 역할·책임·도구 집합**을 부여 (예: Researcher, Coder, Critic, Planner)
- 역할별로 프롬프트·도구·권한을 좁히면 정확도·안전성↑
- “한 에이전트가 다 하기”보다 **관심사 분리(separation of concerns)**가 핵심

## 5. 통신과 조율 (Communication & Coordination)

- 에이전트 간 정보 교환 방식
  - **메시지 패싱(message passing)**: 구조화된 메시지 주고받기
  - **공유 상태(shared state/memory)**: 공용 저장소를 통해 간접 협업
  - **A2A(Agent-to-Agent) 프로토콜** 등 표준화된 에이전트 간 통신 규약
- 조율 이슈: 작업 분배, 충돌·중복 방지, 합의(consensus), 종료 조건(언제 끝낼지)
- 핸드오프(handoff): 한 에이전트가 다른 에이전트에게 제어/작업을 넘기는 패턴

## 6. 오케스트레이션 패턴 정리

- **Supervisor(감독)**: 중앙 매니저가 라우팅·분배 (3-1과 연결)
- **Swarm/네트워크**: 에이전트들이 자율적으로 핸드오프
- **Hierarchical**: 매니저 밑에 또 다른 서브-매니저(다층 구조)
- 프레임워크 예: LangGraph, CrewAI, AutoGen 등

## 7. 멀티에이전트의 도전 과제

- **비용·지연**: 에이전트·호출이 늘수록 선형 이상으로 증가 가능
- **오류 전파 / 누적**: 한 단계 오류가 downstream으로 번짐
- **조율 복잡도**: 무한 루프, 교착(서로 기다림), 책임 불명확
- **관측·디버깅**: 어디서 무엇이 잘못됐는지 추적이 어려움 → 로깅·트레이싱 필요
- **일관성**: 공유 상태의 동시 수정 충돌

## 8. 핵심 정리 (Key Takeaways)

1. 멀티에이전트는 **전문화·병렬성·모듈화**가 필요할 때의 해법. 단일로 충분하면 단일이 낫다.
1. 대표 토폴로지: **매니저-워커(가장 흔함)** / 순차 / 병렬 / 분산 / 블랙보드.
1. 각 에이전트에 **명확한 역할·도구·권한**을 주는 관심사 분리가 핵심.
1. 통신은 메시지 패싱·공유 상태·**A2A 프로토콜**, 핸드오프로 제어를 넘긴다.
1. 멀티에이전트는 강력하지만 **비용·지연·오류전파·조율·디버깅** 비용을 동반 → 신중히 도입.

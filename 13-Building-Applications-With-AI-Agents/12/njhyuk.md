# Chapter 12. Protecting Agent Systems (에이전트 시스템 보호)

## 1. 에이전트의 고유 리스크 (Unique Risks)
- 에이전트는 **도구를 실행하고 행동**하므로, 일반 챗봇보다 위험 표면(attack surface)이 넓다.
- 대표 리스크
  - **프롬프트 인젝션(Prompt Injection)**: 악성 입력/외부 콘텐츠가 에이전트 지시를 탈취
  - **간접 인젝션(Indirect Injection)**: 웹페이지·문서·툴 응답 속에 숨은 악성 지시
  - **탈옥(Jailbreak)**: 안전장치 우회
  - **도구 오남용**: 위험한 도구(코드 실행·결제·파일 삭제)를 잘못 호출
  - **과잉 권한/과잉 자율성**: 필요 이상 권한으로 피해 확대
  - **데이터 유출**: 민감정보가 응답·로그·외부 호출로 새어나감
- 자율성이 높을수록 **한 번의 오류가 실제 피해로 직결**됨

## 2. LLM 보안 (Securing LLMs)

### 2-1. 모델 선택 (Model Selection)
- 안전성·정렬(alignment) 수준, 안전 필터, 신뢰할 수 있는 공급자를 고려해 모델 선택
- 용도에 맞는 모델 (민감 작업엔 더 엄격한 모델)

### 2-2. 방어 (Defenses)
- **입력/출력 필터링**, 시스템 프롬프트 강화, 콘텐츠 검열(가드레일)
- 신뢰 경계 구분: 외부에서 들어온 텍스트는 **지시가 아니라 데이터로** 취급

### 2-3. 레드팀 (Red Teaming) ⭐
- 공격자 관점에서 **의도적으로 취약점을 찾는 테스트** (인젝션·탈옥 시도 등)
- 배포 전/후 지속적으로 수행해 방어를 검증·보완

### 2-4. 파인튜닝 (Fine-Tuning)
- 안전 데이터로 미세조정해 위험 응답 성향을 줄임
- 단, 잘못된 파인튜닝은 오히려 안전성 저하 가능 → 검증 필요

## 3. 데이터 보호 (Data Protection)

### 3-1. 프라이버시 (Privacy)
- **민감정보(PII) 최소 수집·마스킹·암호화**, 로그에 민감정보 남기지 않기
- 목적 외 사용 금지, 보관 기간·삭제 정책

### 3-2. 출처/계보 (Provenance)
- 데이터가 **어디서 왔고 어떻게 쓰였는지 추적** 가능하게
- 오염된 데이터·독성 입력의 유입 경로 파악, 신뢰 가능한 소스만 사용

## 4. 에이전트 보안 (Securing Agents)

### 4-1. 세이프가드 (Safeguards)
- 위험 행동에 대한 **가드레일·정책 체크·승인 절차**
- 예: 결제·삭제 등 고위험 도구는 **사람 승인(human approval)** 후 실행

### 4-2. 외부 보호 (External Protections)
- 네트워크 격리, API 게이트웨이, 레이트 리밋, 인증/인가
- 도구 실행은 **샌드박스**에서 (임의 코드 실행 격리)

### 4-3. 내부 보호 (Internal Protections)
- **최소 권한 원칙(least privilege)**: 각 에이전트/도구에 꼭 필요한 권한만
- 액션 검증·한도 설정, 감사 로그(audit log), 이상 행동 탐지

## 5. 거버넌스·컴플라이언스 (Governance & Compliance)
- 조직 차원의 **정책·책임·규제 준수** 체계
- 규제 예: GDPR 등 데이터 보호, 산업별 규정
- 책임 소재(accountability), 승인·감사 프로세스, 사고 대응(incident response) 계획
- "누가 무엇을 할 수 있는가"를 정책으로 명문화

## 6. 핵심 정리 (Key Takeaways)
1. 에이전트는 **행동을 하므로** 프롬프트 인젝션·도구 오남용·과잉권한 등 고유 리스크가 크다.
2. LLM 보안: **모델 선택 → 입출력 방어 → 레드팀 → 안전 파인튜닝**.
3. 데이터 보호: **프라이버시(PII 최소·암호화)** 와 **출처 추적(provenance)**.
4. 에이전트 보안: **세이프가드(승인) + 외부(샌드박스·격리) + 내부(최소 권한·감사)**.
5. 마지막은 **거버넌스·컴플라이언스** — 정책·책임·규제 준수로 마무리.
6. 관통 원칙: **최소 권한, 신뢰 경계 분리, 사람 승인, 지속적 레드팀**.

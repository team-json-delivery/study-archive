# Chapter 6. Knowledge and Memory (지식과 메모리)

## 1. 왜 메모리가 필요한가

- 파운데이션 모델은 기본적으로 **stateless** → 이전 대화/경험을 스스로 기억하지 못한다.
- 에이전트가 일관성 있게 동작하고, 과거 정보를 활용하고, 모델이 모르는 지식에 접근하려면 **외부 메모리/지식**이 필요.
- 메모리는 크게 두 축으로 본다.
  - **단기 기억(Short-Term)**: 현재 작업·대화 맥락 (컨텍스트 윈도우 안)
  - **장기 기억(Long-Term)**: 세션을 넘어 보존되는 지식·경험 (외부 저장소)
- 이 장의 범위: 컨텍스트 윈도우 관리 → 검색(full-text/semantic) → RAG → 경험 메모리 → 지식 그래프/GraphRAG → 노트테이킹

## 2. 메모리의 기초 접근 (Foundational Approaches)

### 2-1. 컨텍스트 윈도우 관리 (Managing Context Windows)

- 모델이 한 번에 볼 수 있는 토큰 양은 유한 → 무엇을 넣고 뺄지 관리가 핵심
- 주요 기법
  - **잘라내기(truncation)**: 오래된/덜 중요한 내용 제거
  - **요약(summarization)**: 긴 히스토리를 압축해 보존
  - **선별 주입**: 지금 단계에 필요한 정보만 동적으로 삽입
- 5장의 컨텍스트 엔지니어링과 직접 연결되는 부분

### 2-2. 전통적 전문 검색 (Traditional Full-Text Search)

- 키워드 기반 검색 (예: BM25, 역색인/inverted index)
- 장점: 정확한 키워드 매칭에 강함, 빠름, 구현 단순, 해석 가능
- 단점: **의미(semantic)**를 모름 → 동의어·맥락·패러프레이즈에 약함
- “정확한 용어를 알 때”는 여전히 강력

## 3. 시맨틱 메모리와 벡터 스토어 (Semantic Memory & Vector Stores)

### 3-1. 시맨틱 검색 개요 (Introduction to Semantic Search)

- 텍스트를 **임베딩(embedding)** 벡터로 변환 → 의미가 비슷하면 벡터 공간에서 가깝다
- 키워드가 달라도 **의미적으로 유사한 내용**을 찾을 수 있음
- 유사도 측정: 코사인 유사도 등

### 3-2. 벡터 스토어로 시맨틱 메모리 구현

- 문서를 **청크(chunk)** 단위로 쪼개 임베딩 → 벡터 DB에 저장
- 질의도 임베딩해서 가장 가까운 청크를 **top-k**로 검색
- 대표 벡터 DB: FAISS, Pinecone, Chroma, Weaviate 등
- 고려사항: 청크 크기, 임베딩 모델 선택, top-k, 메타데이터 필터링

```text
[문서] → 청크 분할 → 임베딩 → 벡터 DB 저장
[질의] → 임베딩 → 유사도 검색(top-k) → 관련 청크 반환
```

## 4. RAG (Retrieval-Augmented Generation) ⭐

- 모델이 답을 생성하기 전에 **외부 지식을 검색해 컨텍스트로 주입**하는 패턴
- 효과: 모델이 모르는/최신/사내 지식 활용, **환각(hallucination) 감소**, 출처 제시 가능
- 기본 흐름: **질의 → 검색(Retrieve) → 컨텍스트 결합 → 생성(Generate)**
- 품질 좌우 요소: 검색 정확도(retrieval quality), 청크 설계, 재순위(re-ranking)

### 4-1. 시맨틱 경험 메모리 (Semantic Experience Memory)

- 단순 문서 지식이 아니라 **과거 경험/상호작용**을 임베딩해 저장·검색
- 비슷한 상황이 다시 오면 과거 사례를 불러와 활용 → 일관성·학습 효과
- 7장(Learning)의 비모수적(nonparametric) 학습과 연결

## 5. 지식 그래프와 GraphRAG (Knowledge Graphs & GraphRAG)

### 5-1. 지식 그래프 사용하기 (Using Knowledge Graphs)

- 정보를 **노드(엔티티) + 엣지(관계)** 구조로 표현
- 벡터 검색이 약한 **관계·다단계 추론(multi-hop)**에 강함
- 예: “A가 다닌 회사의 CEO는?” 같은 연결 질의

### 5-2. 지식 그래프 구축 (Building Knowledge Graphs)

- 텍스트에서 엔티티·관계를 추출(LLM/NER 활용)해 그래프로 구성
- 구조화된 스키마 설계가 품질을 좌우

### 5-3. GraphRAG

- RAG에 **그래프 구조**를 결합한 형태
- 벡터 검색(의미 유사) + 그래프 탐색(관계)을 함께 활용 → 더 풍부한 컨텍스트
- 복잡한 관계형 질의·요약에 강점

### 5-4. 동적 지식 그래프의 약속과 위험 (Promise and Peril)

- **약속**: 그래프가 실시간으로 갱신되면 항상 최신 관계 반영 가능
- **위험**: 잘못된 추출/오염된 관계가 누적되면 오류 전파, 유지보수·일관성 비용↑

## 6. 노트테이킹 (Note-Taking)

- 에이전트가 작업 중 **중요한 정보를 스스로 메모로 남겨** 이후 단계/세션에서 재사용
- 긴 작업에서 컨텍스트 윈도우 한계를 보완하는 실용적 장기기억 기법
- 사람이 회의 중 메모하는 것과 유사한 발상

## 7. 핵심 정리 (Key Takeaways)

1. 모델은 stateless → 일관성·지식 활용을 위해 **외부 메모리(단기/장기)**가 필요.
1. 컨텍스트 윈도우는 유한 → 잘라내기·요약·선별 주입으로 관리.
1. 검색은 **키워드(full-text)** vs **의미(semantic/임베딩)** 두 축, 보통 함께(hybrid) 쓴다.
1. **RAG**: 검색 결과를 주입해 환각을 줄이고 외부/최신 지식을 활용하는 핵심 패턴.
1. **지식 그래프 / GraphRAG**: 관계·다단계 추론에 강함, 단 구축·유지 비용 주의.
1. **노트테이킹**: 에이전트가 스스로 메모를 남겨 장기 작업의 컨텍스트 한계를 보완.

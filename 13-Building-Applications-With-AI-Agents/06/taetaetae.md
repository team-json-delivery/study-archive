# 6. 지식과 메모리 (Knowledge and Memory)

- **지식(Knowledge)**: 모델 가중치 밖의 사실·도메인 정보(기술 사양·정책·로그 등)를 생성 시점에 끌어옴 → 주로 RAG로 구현
- **메모리(Memory)**: 에이전트 자신의 이력(이전 대화·도구 출력·상태 업데이트)을 기억 → 턴·세션 간 연속성 유지
- 예제는 **LangGraph**(노드=모델 호출·메모리·도구, 엣지=제어 흐름의 방향 그래프) 기반. 메모리 메커니즘을 1급 노드로 다뤄 모듈화·체크포인팅이 쉬움

---

## 6.1 메모리 기본 사용법
가장 단순한 접근(롤링 윈도우·키워드 검색)이지만 의외로 많은 사례에 충분함.

### 6.1.1 컨텍스트 윈도 관리
- **컨텍스트 윈도우** = 한 번의 호출에 모델에 넣는 정보 전체(그 요청의 작업 메모리). 수용 가능한 최대 토큰 수가 **컨텍스트 길이**
- 토큰 1개 ≈ 단어 3/4(약 4글자). 현재 모델은 수십만~100만 토큰까지 지원(예: GPT-5·Claude 3.7 Sonnet 272K, Gemini 2.5 100만)
- **롤링 윈도우(FIFO)**: 윈도우가 차면 오래된 것부터 밀어내고 최신으로 교체
    - 장점: 구현 쉽고 복잡도 낮음 / 단점: 중요도와 무관하게 밀려난 정보는 사라짐(큰 프롬프트·긴 응답에서 금방 발생)
- 팁: 모델은 큰 프롬프트에서 중요한 정보를 놓치기 쉬움 → 핵심을 강조하고 프롬프트 끝쪽에 둘수록 잘 활용함

### 6.1.2 전체 텍스트 검색
- **역색인(inverted index)**: 토큰화·정규화·불용어 제거 후 각 용어를 등장하는 문서·청크에 매핑 → 전체를 스캔하지 않고 해당 구절만 빠르게 조회
- **BM25**로 순위 매김 — 3요소: 용어 빈도, 역문서 빈도(희소성), 문서 길이 정규화
- 흐름: 쿼리를 같은 파이프라인으로 분석 → 상위 K개 구절을 프롬프트에 주입(컨텍스트 길이 절약)
- 한계: 정확한 용어엔 강하지만 의미·패러프레이즈·개념 연결은 못 잡음 → 시맨틱 검색으로 이어짐

## 6.2 시맨틱 메모리와 벡터 스토어
일반 지식·개념·경험을 저장했다가 *의미 기반*으로 검색하는 장기 기억. 핵심 수단은 벡터 DB.

### 6.2.1 시맨틱 검색
- 단어 일치가 아니라 **의미**로 검색. 컨텍스트·동의어·관계를 ML로 해석 → 정확한 키워드가 없어도 관련 내용을 찾음
- 기반은 **임베딩**: 대규모 코퍼스의 쓰임에서 단어 의미를 포착한 벡터. 의미가 비슷할수록 벡터 공간에서 가까움
- 대표 모델: Word2Vec, GloVe, BERT (LLM이 임베딩 성능을 더 끌어올림)

### 6.2.2 벡터 스토어로 시맨틱 메모리 구현
- 지식을 임베딩(연속 벡터 공간에 의미·관계를 표현) → **벡터 DB**에 저장(예: VectorDB, FAISS, Annoy)
- 쿼리가 오면 쿼리 임베딩으로 유사도 검색 → 가장 가까운 항목을 꺼내 응답에 활용
- 효과: 대량 정보를 의미 기반으로 빠르게 검색 → 응답 품질 향상

### 6.2.3 RAG: 검색 증강 생성
- **검색 + 생성**을 결합한 기법
- 색인(준비): 문서를 작은 청크로 분할 → 인코더로 임베딩 → 벡터 DB에 색인
- 런타임: 쿼리에 관련된 청크를 검색 → 그 맥락 + 모델의 자체 지식을 종합해 답 생성
- 도메인·회사별 정보나 정책을 답변에 반영할 때 특히 유용

### 6.2.4 시맨틱 경험 메모리
- 문제: ① 에이전트가 매 세션을 백지에서 시작 ② 긴 작업의 맥락이 윈도우에서 점점 빠짐
- 해결: 매 입력을 벡터화 → **과거 상호작용 전체**에 벡터 검색 → 윈도우 일부를 관련 경험에 예약, 나머지를 시스템 메시지·최신 입력에 배분
- 효과: 외부 지식 + 축적된 경험 기반 → 더 적응적·개인화된 행동

## 6.3 그래프RAG (GraphRAG)
- 베이스라인 RAG(청크 검색)의 약점: ① 여러 문서에 흩어진 정보 연결("connecting the dots") ② 전체 데이터의 상위 주제 요약 ③ 크거나 지저분하거나 서사형인 데이터
- **GraphRAG**: 데이터에서 엔티티-관계 **지식 그래프**를 구축 → 멀티홉 추론·관계 연결·구조화 요약이 가능

### 6.3.1 지식 그래프 활용
- 검색 단계가 단순히 문서를 꺼내는 게 아니라, 그래프에서 **노드·엣지를 분석해** 관련 서브그래프를 뽑음
- 구성 3요소:
    - **지식 그래프**: 엔티티(노드)·관계(엣지)를 명시적으로 저장 (연결 데이터·멀티홉 쿼리에 강함)
    - **검색 시스템**: 그래프를 쿼리해 관련 서브그래프·노드 클러스터 추출
    - **생성 모델**: 그래프로 검색된 데이터를 종합해 응답 생성
- 판단: 표준 청킹+벡터 검색이 한계일 때 → 더 비싸고 복잡하지만 결과가 더 좋은 경우가 많음

### 6.3.2 지식 그래프 구축
- 8단계: ① 데이터 수집 → ② 전처리(정제·표준화) → ③ 엔티티 추출(NER) → ④ 관계 추출 → ⑤ 온톨로지 설계(스키마=백본) → ⑥ 그래프 채우기 → ⑦ 통합·검증(중복 해결 등) → ⑧ 유지보수(지속 갱신)
- 보통 **RDF 트리플(주어-술어-목적어)**로 구축. 파운데이션 모델이 트리플 추출을 잘해 대규모 구축 가능
- 도구: **Microsoft GraphRAG**(CLI, Python 없이 색인·쿼리) / **Neo4j**(프로덕션급, index-free adjacency로 수십억 노드에서도 빠르고 ACID·클러스터링 지원). Cypher의 `CREATE`/`MERGE`로 채움
- 로드 후 멀티홉 순회(`shortestPath` 등)로 평면 테이블·벡터 스토어보다 풍부한 관계 추론 가능

### 6.3.3 동적 지식 그래프의 가능성과 위험성
- **동적 그래프**: 새 데이터로 실시간 갱신·적응 학습 → 뉴스·금융 등 변화 잦은 분야에 강점. 단 프로토타입은 쉬워도 프로덕션은 큰 작업
- **대안 흐름(장기 컨텍스트 / 인덱스 프리 RAG)**: 100만 토큰급 모델에 지식을 통째로 넣어 외부 검색을 생략
    - 트레이드오프: 비용·지연 ↑, 큰 맥락에서 정답 하나를 못 짚을 수 있음
- **책의 결론**: 당분간은 RAG + 장기 컨텍스트 **하이브리드**가 우세(특히 최신성·정밀 순위가 중요할 때)
- 위험 4가지 → 완화책:
    - 유지보수 복잡(오류 전파) → 자동 검증
    - 자원 집약 → 분산 DB·클라우드로 확장 설계
    - 보안·프라이버시 → 암호화·접근 제어
    - 과의존 → 중요 결정에 인간 감독 유지

### 6.3.4 노트 작성
- 모델이 답하기 전에 컨텍스트에 **노트**를 먼저 달고(여백 메모처럼), 그 노트를 원문과 섞어 최종 답을 만듦
- 추론 깊이 비교: 바닐라(바로 답) < CoT(추론 후 답) < **셀프 노트**(컨텍스트·질문에 노트 → 답)

## 6.4 마무리
- 메모리는 에이전트의 핵심. 단순 컨텍스트 윈도우로 충분한 경우도 많지만, 어려운 작업엔 더 강건한 접근(시맨틱 메모리·GraphRAG·작업 메모리)이 큰 도움
- **검색 진화 흐름**: 롤링 윈도우 → BM25(키워드) → 임베딩(의미) → RAG → GraphRAG
- 메모리는 단순 저장이 아니라, 에이전트가 환경·사용자와 상호작용하는 방식을 바꾸는 것
- 다음 장: 에이전트가 경험으로 학습해 자동 개선되는 방법

# CHAPTER 7. 에이전틱 시스템의 학습

- 학습 능력은 매우 유용하지만 **필수는 아님** — 추가 설계·평가·모니터링 비용이 들고, 투자 가치는 애플리케이션마다 다름
- **비모수적 학습**: 모델 파라미터를 바꾸지 않고 성능 개선(예시·성찰·인사이트를 프롬프트에 주입)
- **모수적 학습**: 파운데이션 모델의 가중치를 직접 학습·파인튜닝
- 일반 원칙: 더 단순·빠른 비모수적을 먼저, 데이터가 충분히 쌓이면 파인튜닝 검토

---

## 7.1 비모수적 학습
모델 가중치를 건드리지 않고 경험에서 성능을 끌어올리는 기법들. 실용적·저렴·구현 쉬움.

### 7.1.1 비모수적 예시 학습
- 가장 단순한 기법: 작업 결과에 품질 척도를 매기고, 좋은 예시를 **few-shot 예시**로 프롬프트에 넣어 in-context learning
- **고정 few-shot**: 프롬프트에 하드코딩(불변) / **동적 선택**: 런타임에 벡터 DB에서 입력과 가장 관련 있는 예시를 검색해 주입
- 예시가 많아지면 무작정 추가 시 비용·지연 증가 + 모든 예시가 모든 입력에 유용하진 않음 → 타입·텍스트·시맨틱 검색으로 관련 예시만 선별
- 각 경험을 메모리 뱅크에 저장(문제 설명 + 적용한 해결책 + 결과). 새 상황에서 유사 케이스를 찾아 해결책을 적응 → 인간 기억처럼 과거가 미래 결정을 안내
- 성공 예시를 저장·검색해 프롬프트에 제공하면 다양한 작업에서 성능이 크게 향상(여러 도메인에서 확인된 정착된 발견). 전체 작업 또는 부분 작업에 따로 적용 가능

### 7.1.2 리플렉시온 (Reflexion)
- 핵심: 실패할 때마다 **무엇이 잘못됐고 다음에 어떻게 할지** 짧은 성찰을 글로 남기고, 다음 시도 전에 다시 읽어 전략을 조정(재학습 없음 = 모델을 자기 자신의 코치로)
- 루프: ① 행동 수행 → ② 시도 전체를 로그에 기록(JSON·DB) → ③ 실패 시 성찰 프롬프트로 간결한 계획 생성 → ④ `update_memory`로 성찰을 메모리에 저장 → ⑤ 재시도 시 최근 성찰을 프롬프트 앞에 주입
- 매우 경량(20줄 미만으로 구현). 수치 피드백(성공 플래그)과 자유 형식 코멘트 모두 수용. 코드 디버깅~다단계 추론에서 성능 향상 입증
- 성찰 프롬프트 3부분: ① "환경 요약 말고 전략적 실수에 집중, 계획은 'Plan' 뒤에" 프레이밍 → ② 원래 목표 재진술 → ③ 실패한 실행의 Action/Observation 전사. "Plan:" 큐로 진단→처방 전환

### 7.1.3 경험 학습 (Experiential Learning)
- Reflexion을 한 단계 발전: 경험을 모으는 데 그치지 않고 **경험들에 걸쳐 인사이트를 집계·증류**해 미래 정책을 개선. 핵심은 **교차 작업 학습(cross-task learning)** — 전이 가능한 좋은 관행 식별(**ExpeL**)
- 인사이트 목록을 동적으로 관리: 가치 있는 것은 승급(promote), 덜 유용한 것은 강등(demote), 새 경험으로 수정
- 규칙 갱신 연산 4가지: **AGREE**(강하게 관련) / **REMOVE**(모순·중복) / **EDIT**(일반화·보강) / **ADD**(기존과 구별되는 새 규칙). 규칙은 *일반적이고 상위 수준*이어야 다른 질문의 유사 실패를 막음
- 예시가 적으면 단순 성찰로 충분하지만, 많아지면 인사이트를 정기적으로 재평가해 상대적 중요도를 조정 → 작은 핵심 규칙 집합으로 증류
- **비정상(nonstationary) 환경**에 점진적으로 적응하는 데 특히 유용. 샘플이 매우 많아지면 파인튜닝도 고려할 만함

## 7.2 모수적 학습: 파인튜닝
사전학습 모델의 파라미터를 조금 조정해 특정 작업에 적응. 비모수적보다 강력하지만 데이터·연산·유지보수 비용이 큼 → 정말 필요할 때만.

### 7.2.1 대형 파운데이션 모델 파인튜닝
- 대부분 GPT-5·Claude Opus·Gemini 같은 범용 대형 모델로 시작(광범위 사전학습 + 회사들의 post-training). 파인튜닝은 일반 능력을 유지하며 특정 작업·도메인에 맞춤
- **파인튜닝을 고려할 때**:
    - 도메인 특화가 중요(전문 용어·엄격한 스타일·민감 콘텐츠) → SFT/DPO로 전문성 고정
    - 일관된 톤·형식이 필수(재무 공시·법적 면책 등 정확한 템플릿)
    - 도구·API 호출이 정밀해야 함(의료 용량·트레이딩) → function-calling 파인튜닝으로 오호출 감소
    - 충분한 고품질 데이터·예산 보유(수백~수천 예시, GPU 시간; RFT는 전문 채점자 필요)
    - 재학습 빈도가 관리 가능(도메인이 자주 변하면 유지비 > 이득)
- **보류할 때**: 빠른 프로토타이핑·소량 사용 / 새 베이스 모델 출시가 결과를 무효화할 위험(새 GPT-5가 파인튜닝된 GPT-4를 능가) / GPU·어노테이션·추론 속도 제약(→ RAG 등 비모수적 대안)
- **사전학습(pretraining, 수조 토큰 처음부터)은 거대 AI 랩의 영역**. 거의 모든 팀은 적절한 라이선스의 instruction-tuned 오픈소스 모델로 시작이 최선 → 종종 추가 파인튜닝 불필요. **의심스러우면 파인튜닝하지 마라**
- **4가지 레버**(표 7-1):
    - **SFT**: (프롬프트, 응답) 쌍으로 행동 학습 → 분류·구조화 출력·instruction 오류 수정
    - **Vision 파인튜닝**: 이미지-레이블 쌍 → 이미지 분류·멀티모달 강건성
    - **DPO**: "좋음 vs 나쁨" 쌍 → 톤·스타일·요약 우선순위
    - **RFT**: 전문가 채점 + policy-gradient → 복잡한 추론, 법률·의료 고위험 도메인
- 트레이드오프: 수십억 파라미터 → 무거운 GPU·긴 학습·클라우드 비용, 실시간 추론 지연. 수천 개 고품질 데이터 큐레이션·검증 필요, 부주의 시 편향·과적합. 그럼에도 정확도·깊이가 중요한 헬스케어·법률 등에선 작은 특화 모델을 능가 가능

### 7.2.2 소형 모델
- 대형의 대안: 자원 효율적, 응답 시간 중요·연산 제한 환경에 적합. 잘 튜닝하면 좁은 작업에서 놀랍게 효과적
- 장점:
    - **투명성·해석가능성**: 레이어·파라미터가 적어 의사결정 분석·디버그 쉬움 → 금융·헬스케어·규제 등 설명가능성이 필수인 영역에 유리
    - **Agile·실시간**: 빠른 반복, 증분 학습. 임베디드·모바일·IoT 등 저지연 배포 가능
    - **접근성**: 다수가 오픈소스·무료(Llama, Phi 등) → 비용·인프라 장벽 낮춤(ML 민주화)
    - **성능**: 잘 정의된 좁은 작업(예: 도메인 감성 분석)에선 대형 모델 매치/능가, 과적합 위험도 낮음
    - **지속가능성·갱신**: 학습·추론 에너지 적음(green AI), 빠른 재학습·federated learning(엣지·프라이버시 보존)에 적합
- 빠르게 변하는 영역(2025 초 기준): DeepSeek-v3·Llama 3.1(70B)이 MMLU 66%+, Phi-3-mini(3.8B)가 540B PaLM의 60%를 매치(2년 만에 142배 축소) 등. "최고" 모델은 몇 달이면 바뀜 → HELM·Papers With Code·Hugging Face·BigBench 등 리더보드로 최신 확인
- 선택 기준: **8B 미만** = 온디바이스·저비용 최강 / **8B–70B** = 일반 추론 sweet spot / **그 이상** = GPT-5 등 독점 거대 모델이 고위험 정확도 선도

### 7.2.3 SFT: 지도 파인튜닝
- 모수적 접근의 **기초**. 큐레이션된 **(프롬프트, 응답) 쌍**으로 원하는 출력 스타일·구조·행동을 직접 학습 → 분류·구조화 출력·정밀 도구 사용
- 대표 사용 사례: **function calling 파인튜닝** — 호출 형식뿐 아니라 *호출할지 말지*까지 추론 학습. 예: "보스턴 날씨?" → `get_weather` 호출 / "보스턴에 눈 온다 가정하면 뭐 입지?" → 실제 호출 없이 가상 추론
- 견고화 방법: 각 API에 명시적 **스키마**(JSON Schema·Zod 등) 정의 → 파인튜닝 시 예시와 함께 포함, 런타임에 동일 스키마로 검증(Zod/Ajv/Pydantic) → 잘못·악의적 호출 조기 차단. 특수 토큰(`<think>`, `<tool_call>`)으로 생각·행동 구분
- 구현: **LoRA**(타겟 레이어만 효율적 적응) + Hugging Face TRL의 **SFTTrainer**(sequence packing, gradient checkpointing)
- 권장 순서: 먼저 모델 **내장 function-calling + 런타임 스키마 검증**을 쓰고, 프롬프트·표준 API가 부족함을 확인한 뒤에만 SFT 투자. 효과: 에러율↓, 맥락 판단(호출 안 함) 학습, 재시도·잘못된 호출 감소로 토큰 비용↓

### 7.2.4 DPO: 직접 선호 최적화
- SFT 위에 **선호 학습**을 더함. 표준 SFT가 'gold' 출력을 복제한다면, DPO는 **순위 매겨진 쌍(chosen vs rejected)**에서 선호 판단을 내재화 → 추론 시 더 나은 완성을 고름
- 워크플로우: 한 프롬프트로 여러 완성 생성 → 인간이 선호 표시(y_win/y_lose) → DPO로 선호 출력을 향해 직접 최적화. 구현은 LoRA + `DPOTrainer`, `beta`가 선호 우선 강도를 조절
- 단순 복제가 아니라 **출력 품질 형성**이 목표일 때 유용. 톤·스타일·요약 우선순위 등 미묘한 인간 기대에 정렬

### 7.2.5 RLVR: 검증 가능 보상 강화 학습
- 선호 학습 위에 **명시적·측정 가능한 보상 함수**에 대한 정책 최적화를 더함. 자동 메트릭·규칙 검증기·외부 점수 모델·인간 평가자 등 **만들 수 있는 어떤 채점자든** 연결해 직접 최적화
- DPO(쌍별 선호만 직접 최적화)와 차이: 선호 학습 + 강화 학습을 결합 → **값 점수를 예측**해 관찰된 순위 너머로 일반화
- 적합한 경우: 요약 품질·도구 호출 정확성·사실성·안전 준수처럼 **검증 가능한 평가 신호**가 있을 때. 보상이 희소하거나 대규모 인간 레이블링이 너무 비쌀 때 특히 유용
- 의의: RFT를 확장해 선호 출력 *모방*을 넘어 *가장 유용·정확·정렬된 것*을 예측·최적화 → 자기개선·작업 특화 모델

## 7.3 마무리
- **비모수적**(예시·Reflexion·경험 학습): 파라미터 수정 없이 경험에서 빠르게 학습, 단순·저렴·실세계 반응성
- **모수적**(파인튜닝): 가중치를 직접 조정해 깊은 특화 — SFT(구조·function calling), DPO(품질·정렬), RLVR(검증 가능 보상)
- 두 축을 결합하면 변화하는 작업·환경과 함께 진화하는 견고한 에이전트. 단, **모든 학습 투자는 운영 제약·성능 목표에 정렬**되어야 함

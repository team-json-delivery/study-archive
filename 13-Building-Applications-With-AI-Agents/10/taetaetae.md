## Chapter 10. 운영 환경 모니터링

이 장은 오픈소스 모니터링에 집중하며, 레퍼런스 스택은 OpenTelemetry(계측)·Loki(로그)·Tempo(트레이스)·Grafana(시각화/알림)이고, LangGraph 기반 에이전트와의 연동을 예제로 설명합니다.

### 10.1 모니터링: 학습의 출발점
- 에이전트는 확률적으로 동작하고 도구를 엮으며 무한한 입력에 반응하므로 모든 시나리오를 테스트할 수 없어 모니터링이 사실상 신경계 역할
- 에이전트 실패는 미묘 — 도구는 성공해도 오류가 연쇄되거나, LLM 출력이 유창하지만 오도하거나, 계획이 일부만 맞는 식이라 크래시 없이 지나가므로 신속한 노출이 필요
- 실패와 성공 모두 테스트 케이스로 — 실패 트레이스는 회귀 테스트로, 우수 사례는 골든 패스로 전환해 실제 조건을 반영하는 살아있는 CI/CD 코퍼스 구축("shift left")
- 진짜 실패(체계적 버그)와 기대 가능한 변동(허용 범위 내 비결정성)을 구분하는 것이 핵심 과제이며, 의사결정 트리(성공 기준 충족 여부→재현성→신뢰도·분산)로 노이즈 과민 반응 방지
- 인프라 신호(지연·오류율·CPU)와 의미적 행동(의도 파악·도구 선택·할루시네이션·작업 포기)을 함께 관찰
- 별도 스택이 아니라 프로덕션 서비스용 동일 파이프라인(Prometheus·Grafana)을 그대로 활용하는 것이 바람직
- 관측 데이터엔 민감정보가 섞이므로 RBAC·암호화 저장·접근 감사와 PII 마스킹/해싱이 필요하며 OpenTelemetry가 export 시 스크러빙 훅 제공
- 지표는 인프라·워크플로·출력 품질·사용자 피드백의 네 계층으로 분류되며, 다 모으는 게 아니라 의미 있는 변화를 잡아낼 만큼만 수집

### 10.2 모니터링 스택
- 관측성은 전통 인프라 지표뿐 아니라 할루시네이션·도구 효율·분포 변화 같은 의미적 통찰까지 담아야 하며, 현재는 LangGraph·CrewAI·AutoGen 등과 매끄럽게 연동되는 오픈소스 도구가 중심
- 이미 엔터프라이즈 로깅 스택(Splunk·Datadog·New Relic 등)이 있다면 새 솔루션보다 기존 스택 확장이 대체로 합리적

**10.2.1 그라파나 (Grafana + OpenTelemetry/Loki/Tempo)**
- OTel로 스팬·지표를 export, 로그는 Loki, 트레이스는 Tempo로 보내고 Grafana가 양쪽을 끌어와 에이전트 행동과 시스템 건강을 상관 분석
- 의미적 지표 실시간 대시보드, 이상 징후(재시도 급증 등) 알림, AI 확장을 위한 강한 커뮤니티가 강점
- 장점은 유연성과 벤더 종속 없음, 단점은 멀티툴 셋업과 비인프라 팀엔 가파른 학습 곡선 — 기존 인프라 모니터링을 에이전트로 확장하는 기업에 적합

**10.2.2 ELK 스택 (Elasticsearch/Logstash·Fluentd/Kibana)**
- OTel 컬렉터로 트레이스·로그를 Elasticsearch에 보내고 Kibana로 조회·대시보드 구성, Elasticsearch ML 잡으로 이상 탐지
- 강력한 전문 검색·벡터 검색, 예측 알림용 내장 ML, 대용량 로그 확장성이 강점
- 장점은 우수한 검색·분석과 엔터프라이즈급 확장성, 단점은 높은 자원 요구와 복잡한 배포 — 기존 ELK 투자가 있는 팀에 적합

**10.2.3 어라이즈 피닉스 (Arize Phoenix)**
- Python SDK로 LangGraph를 계측해 LLM 호출을 eval과 함께 트레이싱하고 OTel export로 하이브리드 사용 지원
- eval 포함 구조적 트레이싱(RAG 품질·할루시네이션 탐지), Jupyter 연동, 멀티에이전트 협업 지표가 강점
- 장점은 eval·디버깅 특화와 프로토타이핑에 가벼움, 단점은 트레이스·eval에 한정되고 운영보다 개발 지향 — 연구·ML 팀에 적합

**10.2.4 시그노즈 (SigNoz)**
- OTel 데이터를 직접 수집하는 OTel 네이티브 단일 플랫폼으로 Python 자동 계측 지원, 지표·트레이스·로그를 한 도구에 통합
- AI 기반 통찰(트레이스 이상 탐지), LLM 지표 커스텀 대시보드, ClickHouse 백엔드로 가벼운 셀프 호스팅이 강점
- 장점은 단순한 셋업과 낮은 오버헤드, 단점은 덜 확장적인 생태계와 다소 평범한 시각화 — 스타트업·ML 중심 팀에 적합

**10.2.5 랭퓨즈 (Langfuse)**
- SDK로 LangGraph 노드를 래핑해 프롬프트·출력·eval(응집성 스코어러 등)을 캡처하고 트레이스를 회귀 테스트용으로 export
- LLM 네이티브 지표(토큰 비용·프롬프트 A/B 테스트), 세션 리플레이, PostgreSQL 등 셀프 호스팅이 강점
- 장점은 에이전트·LLM 특화와 개발팀 친화성, 단점은 좁은 범위(CPU 등 인프라 지표 약함, Prometheus 병행 권장) — 코어 스택 교체 없이 에이전트 기능을 더하기에 적합

### 10.3 프로젝트에 적합한 스택
- 모든 스택이 동등하게 유효하므로 먼저 현재 셋업을 평가 — 엔터프라이즈 솔루션이 있으면 OTel 계측으로 확장하되, 자동 eval이 필요하면 Langfuse/Phoenix, 고급 검색이 필요하면 ELK
- 그린필드 프로젝트라면 폭넓은 커버리지의 Grafana나 SigNoz가 무난
- 팀 역량·데이터 규모·연동 요구로 평가하며 다수는 하이브리드 가능(OTel을 여러 백엔드로)
- 이 책의 예제는 높은 조합성·널리 쓰이는 채택·LangGraph 연동·벤더 종속 없음을 이유로 OTel + Grafana/Loki/Tempo 사용

### 10.4 오픈텔레메트리 계측
- 모니터링 루프의 첫 단계는 계측이며, 양질의 신호가 런타임에 박혀 있지 않으면 깜깜이 비행
- LangGraph는 비동기 함수 호출 그래프라 각 노드(계획·도구 호출·생성)가 이미 격리·선언되어 있어 OTel 스팬으로 계측하기 수월
- 노드 시작 시 스팬을 열고 메타데이터를 붙임 — 도구 노드엔 도구명·메서드·지연·성공 여부·오류 코드, LLM 노드엔 프롬프트 ID·토큰 수·모델 지연·할루시네이션 위험/신뢰도
- 대규모 구조 변경 없이 가능하며 OTel Python SDK는 시작 시 1회 초기화, 스팬은 컨텍스트 매니저로 열고 닫고 분산 트레이싱 컨텍스트는 async 호출 간 자동 전파
- 스팬엔 이벤트(폴백·재시도), 중첩 서브스팬, 예외 캡처를 담을 수 있고 Tempo·Jaeger로 실시간 export
- 계측은 신중히 범위 설정 — 너무 많으면 노이즈, 너무 적으면 원인 분석 곤란이라 요청 ID·세션 메타데이터·설정 상태·스킬명·평가 신호 등 딱 필요한 만큼만 부착
- Tempo는 트레이스 백엔드로 확장성 있게 저장하고 깊은 쿼리 지원(예: 계획 단계 1.5초 초과 트레이스 필터링)
- Loki는 로그 집계 계층으로 구조적(JSON) 로그를 수집하며 스팬·트레이스 ID로 주석을 달아 같은 세션의 로그와 트레이스를 상관
- Grafana가 양쪽을 단일 화면으로 통합해 시각화·드릴다운·커스텀 알림 제공

### 10.5 시각화와 알림
- Grafana는 단순 대시보드 도구를 넘어 신호를 스토리로, 지표를 행동으로 바꾸는 관측성의 운영 프런트엔드
- Loki·Tempo와 네이티브 연동 — 트레이스는 스팬 계층(쿼리 수신→계획→도구 호출→출력)을 지연·상태·속성으로 필터링, 로그는 에이전트명·세션·오류 유형·트레이스 ID로 조회
- 공통 메타데이터(요청·세션 ID) 덕분에 로그 급증에서 이를 유발한 정확한 트레이스로 바로 점프
- 진짜 힘은 에이전트 의미·성공 기준에 맞춘 대시보드 — 시간당 토큰 사용량, 도구·계획 노드 P95 지연, 워크플로/프롬프트 버전별 작업 성공률, 도구·스킬별 폴백 빈도, 임베딩 유사도 기반 드리프트 지표
- 커스텀 알림으로 임계값을 정해 Slack·이메일·PagerDuty 등으로 트리거(예: 30분간 할루시네이션 5% 초과, 한 세션 재시도 3회 초과, 핵심 도구 응답 시간 50% 이상 증가)
- 알림은 아무도 대시보드를 안 봐도 회귀·이상을 실시간으로 알려 피드백 루프를 빠르게 마감
- PagerDuty로 온콜 에스컬레이션, Sentry로 예외·스택트레이스·릴리스 건강 보강이 가능하고, AgentOps.ai 같은 올인원 대안도 존재(다만 벤더 종속 가능)

### 10.6 모니터링 패턴
- 관측 스택이 갖춰진 뒤의 질문은 "확률적·적응적이라 완전히 예측하기 어려운 에이전트에 변경을 어떻게 안전하게 배포할 것인가"
- 답은 실험 위험을 낮추고 프로덕션 변경에 안전망을 두는 모니터링 인식형 개발 패턴 채택

**10.6.1 섀도 모드 (Shadow Mode)**
- 새 버전을 프로덕션과 나란히 돌리되 같은 입력을 처리하면서 출력은 사용자에게 제공하지 않아 사용자 경험에 영향 없이 행동을 기록·추적
- OTel로 양쪽을 계측하고 공유 요청 ID를 붙여 Loki·Tempo에서 도구 선택·지연·토큰·할루시네이션 빈도 차이를 비교
- 새 모델·계획 전략·프롬프팅을 시험할 때 유용하며 정상 운영과 병렬로 데이터를 지속 수집해 더 안전한 혁신 가능

**10.6.2 카나리 배포 (Canary Deployments)**
- 섀도 모드가 노출 없이 정보를 모은다면, 카나리는 한 발 더 나아가 새 버전을 소수 실사용자(1~5%)에게만 제공
- Grafana에서 버전 태그로 지표·트레이스를 필터링해 카나리와 베이스라인의 성공률·지연·도구 사용·오류를 직접 비교하고 회귀 시 알림
- 잘 동작하면 점진 확대, 아니면 즉시 롤백해 사용자 영향을 최소화하며 빠른 반복에 필요한 운영 안전성 확보

**10.6.3 회귀 트레이스 수집 (Regression Trace Collection)**
- 프로덕션 실패(할루시네이션·계획 오류·도구 오용)는 학습 기회이므로 Tempo 실패 트레이스나 Loki 로그 스냅샷을 테스트 스위트로 자동 export해 지속 갱신되는 회귀 코퍼스 구축
- 실패한 도구 호출·어긋난 출력이 새 테스트 케이스가 되고 수정 후 같은 트레이스를 재실행하면 통과해야 함
- 시간이 지나며 실제 엣지 케이스로 평가 세트를 강화하고 동일 실패 모드의 재발 방지

**10.6.4 자가 치유 에이전트 (Self-Healing Agents)**
- 모니터링은 실패 탐지를 넘어 회복도 도울 수 있어, 자신의 텔레메트리를 실시간으로 읽는 에이전트는 문제 감지 시 폴백 가동
- 예: 도구 호출이 반복 실패하면 더 단순한 폴백 계획으로 우회하거나 사용자에게 명확화 요청, 지연 급증 시 선택적 추론 단계 생략, 할루시네이션 점수가 높으면 면책 고지나 인간 검토로 이관
- 각 폴백 결정을 로그·트레이스로 남기면 언제·왜 폴백이 발동됐고 효과가 있었는지 분석 가능

### 10.7 사용자 피드백
- 로그·트레이스·지표 외에 사용자 피드백은 에이전트가 인간 기대를 얼마나 충족하는지 직접 보여주는 보완적 렌즈
- 암묵적 피드백(입력 재작성·작업 포기·머뭇거림)과 명시적 피드백(싫어요·별점·자유 텍스트) 모두 실시간 신호로 스택에 통합
- 암묵적 지표(작업 포기율·재질의 빈도)는 Loki에 집계하고 Grafana로 시각화해 마찰·혼란의 조기 지표로 활용, 명시적 이벤트(낮은 평점)는 Tempo 트레이스와 연결하고 불만 급증 시 알림
- 사용자 감정 지표와 트레이스 기반 기술 데이터를 결합한 대시보드로 성능 문제와 사용자 좌절을 상관시켜 에이전트 건강의 전체 그림 확보
- 낮은 평점 트레이스를 평가 세트로 export해 사후 검토하거나, 특정 흐름을 여러 사용자가 포기하면 계획 전략·프롬프트 재검토 — 사용자 신호를 행동 프레임워크에 통합해 사용자 중심성 유지

### 10.8 분포 변화
- 분포 변화는 사용자 언어·신제품 용어·API 응답·모델 업데이트 등으로 환경의 통계적 속성이 시간에 따라 바뀌는 현상으로, 명시적 오류 없이 성능 저하·출력 어긋남·폴백 증가로 나타나는 미묘한 과제
- 모니터링이 이 느린 드리프트에 대한 1차 방어선이며 작업 성공률·도구 실패·토큰 추세·할루시네이션 빈도 대시보드가 조기 신호 표면화
- 정량 탐지에는 통계 검정 활용 — KS 검정은 두 데이터의 누적분포를 비교하는 비모수 검정으로 쿼리 길이·지연 같은 연속 특성의 변화 탐지에 적합(KS > 0.1, p < 0.05면 유의한 발산)
- KL 발산은 한 분포가 다른 분포에서 얼마나 벗어났는지 측정하며 토큰 분포 변화로 컨셉 드리프트를 탐지(비대칭, 임계값 > 0.5면 변화 플래그)
- PSI는 범주형·구간화 변수(도구 사용 범주 등)의 분포 변화를 측정하며 < 0.1 안정, 0.1~0.25 경미(모니터), > 0.25 중대(개입·재학습)
- 정확도 급락(24시간 5~10% 초과), 작업 포기 증가(15% 초과), 재시도 급증(세션 20% 초과)도 드리프트 지표이며 임베딩 코사인 유사도(평균 < 0.8면 검토)도 활용
- 일시적 변화는 임계값 조정·파싱 로직 수정으로, 지속적 변화는 재학습·API 적응으로 대응하되 드리프트 심각도(예: PSI > 0.25가 48시간 지속 시 재학습 우선)에 따라 판단하고 A/B 테스트로 수정 검증

### 10.9 지표 소유권과 기능 간 거버넌스
- 에이전트 배포 시 "어떤 지표를 누가 소유하는가"라는 미묘하지만 중대한 조직적 과제 발생 — 전통 스택의 명확한 분담(인프라=지연·가동, 제품=전환·성공, ML=모델)을 모델 기반 에이전트는 따르지 않음
- 모델 응답은 곧 제품이고, 긴 도구 호출·재시도·폴백·생성 사슬은 곧 사용자 경험이며, 5초 계획 지연은 모델 한계가 아니라 프롬프트·워크플로 설계 결정인 경우가 많음
- 따라서 에이전트의 로그·트레이스·평가 신호는 제품 대시보드나 모델 노트북이 아니라 서비스 건강과 함께 코어 관측 플랫폼에 있어야 전체 그림 확보 가능
- 지연이 대표 사례 — "모델은 원래 느리다"는 통념이 장황한 프롬프트·불필요한 재시도·비대한 계획으로 지연을 당연시하게 만들고, 트레이스 기반 계측 없이는 이 드리프트가 미탐지
- 해법은 소유권을 인프라/제품에 떠넘기는 게 아니라 공유 대시보드 구축 — 제품 리드는 계획 지연·폴백률과 작업 포기의 상관을, ML은 할루시네이션·드리프트를 사용자 피드백과 함께, 인프라/SRE는 토큰 급증·도구 불안정을 관찰
- 각 팀이 텔레메트리 일부를 소유하되 어느 한 팀도 단독으로 해석할 수 없으며, RACI 차트(R 실무·A 책임·C 자문·I 통지)로 역할을 명확히 해 누락과 사일로 방지
- 실천법: 버전 태그·의미적 지표가 담긴 공유 대시보드 사용, 스팬·로그에 제품 맥락(기능 플래그·사용자 등급·워크플로 ID) 태깅, 출시·회귀 후 기능 간 트리아지 의례 마련, 모델 지연에 이중 잣대 금지

### 10.10 마무리
- 모니터링은 안전 점검을 넘어 지능형 시스템이 실세계에서 번성하게 하는 규율이며, 반응적인 것이 아니라 프로덕션에서 학습하고 변화에 적응하며 진척을 가속하는 방식
- OTel 계측 → Loki·Tempo의 실시간 로그·트레이스 수집 → Grafana 대시보드·알림으로 장애 이전에 문제를 표면화하고 모든 배포를 개선 기회로 전환하는 오픈소스 피드백 루프 구성
- 섀도 모드·카나리·폴백 로깅·사용자 감정 추적 같은 실용 기법과 함께, 무엇을 측정할지뿐 아니라 어떻게 행동할지를 강조
- 관측성을 염두에 두고 계측·시각화·학습하는 팀이 더 빨리 반복하고 지표를 신뢰하며 우아하게 회복하는 강력한 우위 확보 — 에이전트가 핵심 인프라가 되는 시대에 견고한 모니터링은 선택이 아닌 필수

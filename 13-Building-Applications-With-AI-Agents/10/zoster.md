# 10. Monitoring in Production

## 10.1 Monitoring Is How You Learn

* 에이전트 시스템은 확률적으로 동작하므로 전통적인 소프트웨어와 달리
  예측 불가능한 환경에서 운영됩니다. 모니터링은
  단순한 문제 탐지를 넘어 시스템의 실패를 분석하고, 성공 사례를 학습하여
  CI/CD 파이프라인을 강화하는 핵심적인 피드백 루프 역할을 합니다.


* 시스템의 무결성을 유지하기 위해 단순 실패와 기대되는 확률적 변이를
  구분해야 합니다. 결정 트리 기반의 판단 과정을 통해 노이즈를
  제외하고 실제 성능 저하나 이상 징후를 조기에 발견하여 안정적인
  운영 환경을 구축하는 것이 중요합니다.



## 10.2 Monitoring Stacks

* 에이전트 모니터링은 인프라 지표와 더불어 할루시네이션, 도구 효용성 등
  의미론적 행동을 포착해야 합니다. 기존의 엔터프라이즈 로깅
  환경을 확장하여 OpenTelemetry와 Grafana, Loki, Tempo 등을 활용하면
  가시성을 확보할 수 있으며, 필요에 따라 Langfuse나 Arize Phoenix 같은
  전용 도구를 병행하여 최적의 관측성을 구현할 수 있습니다.


* Grafana 기반 스택은 높은 구성 가능성을 제공하며, ELK 스택은 강력한
  검색 및 분석 기능을 지원합니다. 한편, Langfuse나
  Phoenix는 에이전트 평가 및 디버깅에 특화되어 있으며, SigNoz는 단일
  플랫폼 형태의 통합된 환경을 제공하여 팀의 성숙도와 요구사항에 맞는
  선택지를 폭넓게 제공합니다.



## 10.3 OTel Instrumentation

* 에이전트 런타임에 고품질 데이터를 직접 삽입하는 계측(Instrumentation)은
  관측성의 첫걸음입니다. LangGraph와 같은 프레임워크는
  함수 호출 단위로 구성되어 있어 각 노드에 OpenTelemetry 스팬을 추가하여
  워크플로우의 성능, 도구 호출 결과, 토큰 사용량 등을 상세히 기록하기에
  매우 용이합니다.


* 계측 시 너무 많은 정보는 노이즈가 되므로, 요청 ID나 세션 메타데이터 등
  루트 원인 분석에 꼭 필요한 정보만 선별적으로 담아야 합니다.
  이렇게 수집된 데이터는 Tempo와 Loki를 통해 추적 및 로그로 저장되며,
  Grafana에서 통합 시각화되어 복잡한 에이전트 시스템의 동작을 투명하게
  드러내줍니다.



## 10.4 Visualization and Alerting

* Grafana는 수집된 로그와 추적 데이터를 시각화하는 핵심 프론트엔드이자,
  사전 정의된 임계값에 따라 경고를 발송하는 운영 플랫폼입니다.
  대시보드를 통해 토큰 사용량, 할루시네이션 비율, 작업 성공률 등을 실시간으로
  파악하여 시스템의 상태를 한눈에 모니터링할 수 있습니다.


* 사용자 피드백이나 도구 실패 등 주요 지표가 설정 범위를 벗어날 경우,
  Grafana는 즉각적인 알림을 통해 담당 팀에 공유합니다.
  필요에 따라 Sentry와 같은 도구를 병행하여 예외 처리를 보강하거나,
  AgentOps.ai를 사용하여 더 특화된 에이전트 관리 기능을 활용할 수
  있습니다.



## 10.5 Monitoring Patterns

* 확률적인 에이전트 시스템을 안전하게 배포하기 위해서는 섀도 모드나
  카나리 배포와 같은 패턴이 필요합니다. 섀도 모드는
  실제 트래픽을 이용해 새로운 에이전트의 동작을 실험하고, 카나리는 일부
  사용자에게만 점진적으로 배포하여 위험을 최소화하며 배포 안전성을
  확보합니다.


* 또한 실패한 실행을 테스트 케이스로 전환하는 회귀 추적 수집과, 스스로
  데이터를 읽어 문제를 복구하는 자가 치유(Self-Healing) 기법은 시스템의
  회복 탄력성을 크게 높여줍니다. 이러한 기법들은
  모니터링 데이터가 단순히 보고용이 아닌 능동적인 개선 도구임을 보여줍니다.



## 10.6 Distribution Shifts

* 에이전트 환경에서 발생하는 데이터 분포 변화인 드리프트는 즉각적인 오류를
  일으키지 않더라도 점진적인 성능 저하를 유발합니다. 이를
  탐지하기 위해 KS 테스트, KL 발산, PSI와 같은 통계적 기법을 활용하여
  입력 데이터나 출력 결과의 편차를 모니터링하고 임계값을 관리해야 합니다.


* 드리프트가 발견되면 즉각적인 조정이나 워크플로우 재학습 등을 통해
  대응해야 합니다. 이러한 과정은 팀이 문제를 인식하고 해결하는
  피드백 루프의 일부로, 실시간 관측성을 통해 드리프트가 시스템 실패로
  번지기 전에 예방하는 효과를 줍니다.



## 10.7 Metric Ownership and Cross-Functional Governance

* 에이전트는 특정 부서의 소유가 아닌 복합적인 제품이므로, 지표 역시
  제품, ML, 인프라 팀이 공유하는 구조가 필수적입니다.
  latency와 같은 지표를 단순히 모델 성능으로만 보지 않고 사용자 경험의
  관점에서 바라보는 공통의 대시보드가 필요합니다.


* RACI 차트 등을 활용하여 책임 소재를 명확히 하고, 정기적인 트리야지
  의식을 통해 부서 간 협업을 강화합니다. 서로 다른
  팀이 동일한 모니터링 데이터를 보고 소통함으로써 에이전트 시스템의
  기술적 안정성과 사용자 가치를 동시에 향상시킬 수 있습니다.



# Appendix

* **OpenTelemetry**: 에이전트 워크플로우를 계측하여 추적, 지표, 로그 데이터를 수집하는 표준 프레임워크.


* **Loki**: 구조화된 로그를 수집, 집계 및 검색할 수 있게 해주는 오픈 소스 로그 관리 플랫폼.


* **Tempo**: 대규모 분산 환경에서 에이전트 시스템의 추적(Tracing) 데이터를 저장하고 쿼리하는 고성능 백엔드.


* **Grafana**: 시각화 및 대시보드 도구로, 다양한 소스의 메트릭을 통합하여 모니터링과 알림을 담당하는 운영 플랫폼.


* **Shadow Mode**: 실제 서비스 트래픽을 새로운 에이전트에 공급하되, 출력값은 사용자에게 전달하지 않고 성능만 평가하는 실험 방식.


* **Canary Deployment**: 새 버전의 에이전트를 일부 사용자에게만 먼저 배포하여 안정성을 검증하는 배포 전략.


* **Drift**: 사용자 언어나 시스템 입력 데이터의 통계적 분포가 시간이 지나면서 변화하여 성능이 저하되는 현상.


* **KS Test**: 두 데이터셋의 분포 차이를 판단하여 드리프트를 탐지하는 비모수 통계 검정법.

# Chapter 1. 에이전트

## 1.1 에이전트의 정의

### 자율 에이전트

- **자율 에이전트** 데이터를 분석하고, 환경을 해석하며, 주어진 컨텍스트에 따라 스스로 판단하고 행동하도록 설계된 지능형 시스템
- **정해진 스크립트만 실행하는지**, 아니면 **상황을 이해하고 다음 행동을 선택**
    - 한 번 답변하고 끝나는 것이 아니라 **추론 → 행동 → 관찰 → 재판단**의 반복 구조

| 구분 | 설명 |
| --- | --- |
| 입력 이해 | 사용자의 요청, 데이터, 환경 정보를 해석한다 |
| 추론 | 현재 상황에서 무엇이 필요한지 판단한다 |
| 행동 선택 | 적절한 도구, API, 검색, 코드 실행 등을 선택한다 |
| 결과 확인 | 실행 결과를 보고 다음 단계를 다시 결정한다 |

### 에이전트 시스템

- 에이전트가 실제로 동작하기 위한 전체 실행 환경
- 모델만 포함되는 것이 아니라, 도구, 메모리, 외부 API, 오케스트레이션, 프롬프트, 평가 체계까지 포함
- MCP model context protocol, A2A agent to agent protocol 같은 프로토콜은 에이전트가 외부 도구나 다른 에이전트와 협력할 수 있게 해줌

| 구성 요소 | 역할 |
| --- | --- |
| 파운데이션 모델 | 자연어 이해, 추론, 생성의 중심 역할 |
| 도구 | 검색, 계산, 파일 처리, API 호출 등 외부 기능 수행 |
| 메모리 | 이전 대화, 작업 상태, 사용자 선호 등을 저장 |
| 오케스트레이션 | 어떤 순서로 어떤 작업을 수행할지 제어 |
| 프로토콜 | 외부 도구나 다른 에이전트와 연결하기 위한 규칙 |

## 1.2 사전학습이 일으킨 혁명

- 기존 AI 시스템은 특정 작업을 위해 별도의 모델을 만들고 학습시키는 방식이 많았다. 반면 **사전학습 모델**은 대규모 데이터로 언어와 지식을 미리 학습해두고, 이후 다양한 작업에 재사용 할 수 있다.
- GPT, BERT, T5 같은 모델은 사전학습을 통해 범용적인 언어 이해 능력을 확보
    - 이후 프롬프트, 파인튜닝, 검색 증강 생성 등을 통해 특정 목적에 맞게 활용 됨.

| 방식 | 특징 |
| --- | --- |
| 기존 모델 | 특정 작업별로 별도 학습이 필요 |
| 사전학습 모델 | 대규모 데이터로 먼저 학습 후 여러 작업에 재사용 |
| 파인튜닝 | 특정 도메인이나 작업에 맞게 추가 학습 |
| 프롬프트 기반 사용 | 자연어 지시만으로 다양한 작업 수행 |
- 사전학습의 등장으로 AI는 단순 분류기나 예측 모델을 넘어, **자연어로 지시받고 복합 작업을 수행하는 시스템**으로 확장되었다.

## 1.3 에이전트 유행

최근 에이전트가 주목받는 이유는 LLM이 단순히 문장을 생성하는 수준을 넘어, **계획 수립, 도구 사용, 중간 결과 해석, 작업 반복**을 수행할 수 있게 되었기 때문이다.

에이전트는 사용자의 요청을 바로 답변으로 바꾸는 것이 아니라, 필요한 경우 작업을 여러 단계로 나누고 외부 도구를 사용한다.

| 일반 챗봇 | 에이전트 |
| --- | --- |
| 질문에 대한 답변 생성 중심 | 목표 달성을 위한 행동 수행 중심 |
| 대화 맥락 위주 | 대화, 도구, 데이터, 상태를 함께 활용 |
| 한 번의 응답으로 끝나는 경우가 많음 | 중간 결과를 보고 다음 행동을 결정 |
| 실행 권한이 제한적 | 검색, API, 코드 실행 등 도구 사용 가능 |

대표적인 흐름으로는 ReAct, AutoGPT, LangChain, LangGraph, CrewAI 같은 접근이 있다. 다만 에이전트라는 이름을 붙인다고 자동으로 좋은 시스템이 되는 것은 아니다. 실제 문제에 맞게 **역할, 책임, 도구, 평가 기준**을 설계해야 한다.

## 1.4 모델 선택

에이전트 시스템에서 어떤 모델을 선택하느냐는 성능, 비용, 속도, 안정성에 직접적인 영향을 준다.

모델을 선택할 때는 단순히 성능이 가장 좋은 모델만 고르는 것이 아니라, **작업 난이도와 비용 구조에 맞게 선택**해야 한다.

| 고려 요소 | 설명 |
| --- | --- |
| 정확도 | 답변과 판단의 신뢰성 |
| 추론 능력 | 복잡한 문제를 단계적으로 해결하는 능력 |
| 컨텍스트 길이 | 긴 문서, 코드, 대화 기록을 처리할 수 있는 범위 |
| 도구 사용 능력 | 함수 호출, API 호출, 검색 결과 활용 능력 |
| 응답 속도 | 사용자 경험과 처리량에 영향 |
| 비용 | 대량 호출 시 운영 비용에 직접 영향 |
- 복잡한 추론, 계획 수립, 중요한 의사결정에는 고성능 모델이 유리
    - 반대로 단순 분류, 요약, 포맷 변환, 라우팅 같은 작업은 더 작고 빠른 모델로도 충분
- 실무에서는 하나의 모델만 쓰기보다 다음과 같은 **모델 라우팅 전략**이 유용

| 작업 유형 | 적합한 모델 전략 |
| --- | --- |
| 단순 분류, 라우팅 | 작고 빠른 모델 |
| 문서 요약, 포맷 변환 | 중간급 모델 |
| 복잡한 추론, 설계, 검증 | 고성능 모델 |
| 대량 배치 처리 | 비용 효율이 높은 모델 |

## 1.5 긴 컨텍스트의 전환

긴 컨텍스트 모델은 많은 문서, 긴 대화 기록, 코드베이스, 로그 등을 한 번에 참고할 수 있게 해준다. 기존에는 필요한 정보를 검색하거나 요약해서 넣어야 했지만, 긴 컨텍스트 모델은 더 넓은 정보를 직접 입력으로 받을 수 있다.

- 컨텍스트가 길다고 항상 좋은 것은 아니다. 정보가 많아질수록 중요한 내용이 묻힐 수 있고, 비용과 지연 시간이 증가할 수 있다.
- **검색 기반 RAG와 함께 사용하는 방식**이 현실적
    - 전체 문서를 모두 넣는 대신 검색으로 관련 후보를 좁힌 뒤 긴 컨텍스트에 구조화해서 넣으면 더 안정적인 결과를 얻을 수 있음

| 장점 | 한계 |
| --- | --- |
| 많은 문서를 한 번에 참고 가능 | 비용 증가 |
| 긴 대화나 코드 흐름 유지 가능 | 중요한 정보가 묻힐 수 있음 |
| 검색 없이도 넓은 범위 분석 가능 | 입력 품질이 낮으면 성능 저하 |
| 복잡한 문맥 이해에 유리 | 모델이 모든 정보를 균등하게 잘 쓰지는 않음 |

## 1.6 활용 사례

에이전트는 단순 질의응답보다 **복합적인 업무 자동화**에 적합

- 특히 중간 판단이 필요하거나 여러 도구를 조합해야 하는 작업에서 효과적

| 활용 영역 | 예시 |
| --- | --- |
| 코드 개발 | 코드 작성, 리팩터링, 테스트 생성, 오류 분석 |
| 문서 처리 | 요약, 분류, 정보 추출, 보고서 작성 |
| 검색/리서치 | 웹 검색, 내부 문서 검색, 근거 정리 |
| 데이터 분석 | 데이터 로딩, 통계 계산, 시각화, 인사이트 도출 |
| 고객 지원 | 문의 분류, 답변 생성, 티켓 라우팅 |
| 업무 자동화 | 이메일 작성, 일정 확인, API 호출, 워크플로 실행 |
- 실제 운영 환경에서는 에이전트가 모든 일을 혼자 처리하게 하기보다, **작은 역할을 가진 구성요소들이 협력하는 구조**가 더 안정적이다.
- 예를 들어 문서 색인 시스템에서는..

| 역할 | 설명 |
| --- | --- |
| 추출기 | PDF, PPTX, Excel 등에서 텍스트와 이미지 추출 |
| 청킹기 | 의미 단위로 문서를 나눔 |
| 임베딩기 | 벡터 생성 |
| 인덱서 | 검색 엔진에 저장 |
| 검증 에이전트 | 추출 품질과 색인 품질 점검 |

## 1.7 워크플로와 에이전트

- **워크플로**는 미리 정해진 절차를 안정적으로 수행하는 구조
- **에이전트**는 상황에 따라 다음 행동을 판단하는 구조다.

| 구분 | 워크플로 | 에이전트 |
| --- | --- | --- |
| 핵심 | 정해진 절차 실행 | 상황 기반 판단 |
| 흐름 | 고정적 | 유동적 |
| 적합한 업무 | 반복적이고 예측 가능한 작업 | 중간 판단이 필요한 작업 |
| 장점 | 안정성, 재현성 | 유연성, 문제 해결 능력 |
| 단점 | 예외 상황 대응이 약함 | 예측 가능성이 낮을 수 있음 |

### 워크플로 예시

문서 업로드 시스템은 다음처럼 고정된 흐름이 많은데

```
파일 업로드 → 텍스트 추출 → 청킹 → 임베딩 → 색인 → 완료
```

이런 경우는 워크플로로 구성하는 것이 좋음

### 에이전트 예시

반면 검색 결과가 부족할 때 다음 검색어를 바꾸거나, 오류가 발생했을 때 다른 도구를 선택하는 작업은 에이전트에 가깝다.

```
질문 분석 → 검색 수행 → 결과 부족 판단 → 검색어 재작성 → 재검색 → 답변 생성
```

실제 시스템에서는 둘 중 하나만 선택하기보다, **워크플로 안에 에이전트 판단을 일부 넣는 혼합 구조**가 많이 사용

## 1.8 효과적인 에이전트 시스템 구축 원칙

- 처음부터 복잡한 구조를 만들기보다 작고 명확한 구조에서 시작하는 것이 좋음

| 원칙 | 설명 |
| --- | --- |
| 단순하게 시작 | 복잡한 멀티 에이전트보다 작은 단일 흐름부터 검증 |
| 역할 제한 | 에이전트가 할 일과 하지 말아야 할 일을 명확히 구분 |
| 도구 분리 | 검색, 계산, 실행, 저장 같은 기능을 분리 |
| 관찰 가능성 확보 | 어떤 판단과 도구 호출이 있었는지 추적 가능해야 함 |
| 실패 대응 | 재시도, 예외 처리, 사람 검토 지점을 설계 |
| 평가 기준 마련 | 성공과 실패를 판단할 수 있는 기준 필요 |
- 특히 중요한 것은 **통제 가능성**
    - 에이전트가 자율적으로 움직이더라도, 시스템은 무엇이 실행되었고 왜 실행되었는지 확인할 수 있어야 한다.
    - 운영 환경에서는 다음 항목을 기록하는 것이 좋다.


        | 로그 항목 | 필요 이유 |
        | --- | --- |
        | 사용자 입력 | 요청의 원본 확인 |
        | 모델 응답 | 판단 과정 추적 |
        | 도구 호출 내역 | 어떤 외부 기능을 사용했는지 확인 |
        | 중간 결과 | 실패 지점 분석 |
        | 최종 결과 | 품질 평가와 재현성 확보 |

## 1.9 에이전트 시스템 구축을 위한 조언

에이전트 시스템을 만들 때는 코드를 먼저 작성하기보다, 해결하려는 문제를 명확히 정의하는 것이 중요하다.

먼저 다음 질문에 답할 수 있어야 한다.

| 질문 | 의미 |
| --- | --- |
| 어떤 문제를 해결하는가? | 시스템의 목적 정의 |
| 어떤 입력을 받는가? | 사용자 요청, 파일, 데이터 형식 정의 |
| 어떤 출력을 만들어야 하는가? | 답변, 보고서, API 결과 등 정의 |
| 어떤 도구가 필요한가? | 검색, DB, API, 코드 실행 등 정의 |
| 어디까지 자동화할 것인가? | 사람 검토가 필요한 지점 정의 |
| 어떻게 평가할 것인가? | 품질 기준과 테스트 방법 정의 |

에이전트 시스템은 모델 성능만으로 완성되지 않는다.

- 프롬프트, 도구, 데이터, 평가, 운영 설계가 함께 맞물려야 함
- 실무적으로는 다음 순서가 좋음

    ```
    문제 정의 → 입력/출력 정의 → 단순 워크플로 구현 → 도구 연결 → 에이전트 판단 추가 → 평가/로그 강화 → 운영 확장
    ```

- 처음부터 완전 자율 에이전트를 만들려고 하기보다, **작은 단위의 자동화부터 검증하고 확장**하는 방식이 안정적

## 1.10 에이전트 프레임워크

에이전트 프레임워크는 에이전트를 개발할 때 반복적으로 필요한 기능을 제공한다. 대표적으로 상태 관리, 도구 호출, 메모리, 라우팅, 워크플로 제어, 오류 처리, 관찰 가능성 등이 있다.

| 프레임워크 기능 | 설명 |
| --- | --- |
| 상태 관리 | 현재 작업 상태와 중간 결과 유지 |
| 도구 호출 | 검색, API, 코드 실행 등 외부 기능 연결 |
| 메모리 | 과거 대화나 작업 이력 저장 |
| 라우팅 | 요청 유형에 따라 다른 노드나 에이전트로 분기 |
| 워크플로 제어 | 단계별 실행 흐름 관리 |
| 오류 처리 | 실패 시 재시도나 대체 경로 실행 |
| 관찰 가능성 | 실행 로그와 트레이싱 제공 |

대표적인 프레임워크로는 LangChain, LangGraph, AutoGen, CrewAI 등이 있음

| 도구 | 특징 |
| --- | --- |
| LangChain | LLM 애플리케이션 구성 요소를 빠르게 연결 |
| LangGraph | 상태 기반 그래프 워크플로와 에이전트 흐름 구성 |
| AutoGen | 여러 에이전트 간 협업 구조 실험에 적합 |
| CrewAI | 역할 기반 멀티 에이전트 구성에 초점 |
- 프레임워크를 사용하면 빠르게 시작할 수 있지만, 내부 동작을 이해하지 못하면 디버깅과 운영이 어려워질 수 있다. 따라서 핵심 흐름을 먼저 이해한 뒤 필요한 부분에 프레임워크를 적용하는 것이 좋다.

## 1.11 마무리

에이전트는 하나의 모델이 아니라 **모델, 도구, 메모리, 실행 구조, 평가 체계가 결합된 시스템**이다.

좋은 에이전트 시스템은 무조건 높은 자율성을 목표로 하지 않는다. 실무에서는 **신뢰성, 관찰 가능성, 통제 가능성**이 더 중요하다.

| 핵심 정리 | 설명 |
| --- | --- |
| 에이전트는 시스템이다 | LLM 하나가 아니라 도구와 실행 구조까지 포함한다 |
| 워크플로와 에이전트는 다르다 | 절차 중심이면 워크플로, 판단 중심이면 에이전트 |
| 긴 컨텍스트는 만능이 아니다 | RAG, 요약, 구조화와 함께 사용해야 한다 |
| 모델 선택은 전략이다 | 작업 난이도에 따라 모델을 나누는 것이 좋다 |
| 운영 설계가 중요하다 | 로그, 평가, 예외 처리, 사람 검토가 필요하다 |

결론적으로 에이전트는 워크플로를 완전히 대체하는 개념이 아니라, **판단이 필요한 지점에서 워크플로를 보완하는 구조**로 이해하는 것이 현실적이다.

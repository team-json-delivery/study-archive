# 1. Agent
## 1.1 AI Agent의 정의
- 자율 agent는 데이터를 스스로 분석하고 환경을 해석하며 컨텍스트에 기반한 결정을 내리도록 설계된 지능형 시스템이다
- 진정한 에이전트를 판단하는 핵심 기준은 정적인 스크립트를 따르는지 실제로 의사결정을 하는지에 달려 있다
  - > Skill과 Agent의 차이로 볼 수 있을지?
- Agent가 효과적으로 동작하도록 돕는 도구, 메모리, 파운데이션 모델, 오케스트레이션, 지원 인프라 전체를 Agentic system 이라고 부르겠다

## 1.2 사전학습이 일으킨 혁명
- 전통적인 머신러닝은 데이터의 양과 질에 많은 영향을 받았다
- 이제는 API 호출 한 번으로 대형 사전학습 생성 모델을 활용할 수 있다
- 파운데이션 모델은 자연어 이해와 생성에 강점을 보이고, 에이전트의 능력을 강화한다
  - 자연어 이해, 컨텍스트 인지 상호작용, 구조화된 생성
- 이것으로도 강력하지만, 정교한 오케스트레이션, 프레임워크와 결합하면 외부 시스템과 직접 상호작용하여 실용적인 작업을 실행할 수 있다
  - 컨텍스트 기반 해석과 의사결정, 도구 사용, 적응적 계획(복잡한 다단계 작업을 계획, 실행), 정보 요약, 비정형 데이터 처리, 코드 생성, 반복 업부 자동화, 멀티모달 통합

## 1.3 Agent 유형
- Information은 Agent를 7가지 유형으로 분류했다
  - 대화형 : 사용자와 상호작용하는 에이전트
  - 리서치
  - 분석
  - 개발
  - 도메인 특화
  - 브라우저 활용
  - 그외
    - 음성 / 비디오
- > Agent와 Skill의 차이는 위의 유형보다는 오히려 Open 탐색이냐, 정형화된 결과 반환이냐에 달린것 아닐까?

## 1.4 모델 선택
- 여러가지 모델이 있고, 각 모델이 잘하는 범위가 다르기 때문에 효율적인 모델을 고를 수 있다
- 비용 효율적이거나, 부분 별로 다른 모델을 사용하거나 등.. 미래에는 멀티모델이 시스템 표준이 될 가능성이 높다

## 1.5 동기에서 비동기로의 전환
- 자율 Agent는 비동기식을 전제로 설계된다. 동시에 여러 작업을 관리하고 새로운 정보에 빨리 적응하며 변화하는 조건에 따라 우선순위를 동적으로 조정한다.
- 여기서 단순히 속도를 높이는 것이 아니라, 여러 세부적인 작업을 동시에 처리하고, 인간은 그걸 감독하는 역할로 넘어가게 된다

## 1.6 활용 사례
- 실제 예제 에이전트들을 보면
  - 고객 지원 에이전트 : 빈번한 문의 처리, 환불, 주문 업데이트 등..
  - 금융 서비스 에이전트 : 계정 관리, 대출, 사기 탐지 등..
  - 의료 접수, 분류 에이전트 : 신규 환자 등록, 보험 확인
  - IT 헬프데스크 에이전트 : 접근 권한 관리, 네트워크/시스템 문제 해결
  - 법률 문서 검토 에이전트 : 계약 검토, 리서치, 접수, 충돌 검사 등..
  - 보안 운영센터(SOC) 분석 에이전트 : 경보 조사, 위협 인텔 수집, 로그 질의 등..
  - 공급망, 물류 에이전트 : 재고 최적화, 선적 추적, 공급업체 평가 등..

## 1.7 워크플로와 에이전트
- 에이전트를 선택 할때는 입력의 가변성, 필요한 추론 복잡도, 성능/컴플라이언스 제약, 유지보수 부담이라는 네 가지 핵심 요소를 고려해야 합니다
- 입력의 형태가 완전히 정해져 있다면 모델보다는 코드를 작성하는 편이 낫다
- 로직을 유한한 단계나 분기로 표현할 수 있고 개입이 필요한 경우 등 모든 케이스가 커버되었다면 워크플로를 사용하는 것이 좋다(Skill)
- 챗봇/RAG : 자연어 이해와 문서 검색 기능을 추가하지만 자율적이고 다단계적인 계획 수립까지는 하지 않음
- 자율 Agent : 입력이 변동성이 크고 상황에 따라 계획이 바뀌거나 지속적 학습이 필요한 경우 자율 에이전트를 사용해야 한다

## 1.8 효과적인 에이전틱 시스템 구축 원칙
- 확장성 : 분산 아키텍처, 클라우드 인프라, 병렬 처리를 지원하는 효율적 알고리즘을 통해 증가하는 부하와 다양한 작ㅇ버을 처리한다
- 모듈성 : 명확한 인터페이스로 연결된 독립적이고 교환 가능한 구성요소로 설계. 유지보수와 변경 적응이 쉬워짐
- 지속 학습 : 인컨텍스트 학습과 같이 경험에서 배우는 메커니즘을 구축하고 사용자 피드백을 통합
- 회복탄력성 : 오류, 보안 위협, 타임아웃등 예기치 못한 상황을 우아하게 처리하는 아키텍처를 갖춰야 한다
- 미래 대비 : 개방형 표준과 확장 가능한 인프라를 중심으로 설계하고 실험 문화를 유지한다

## 1.9 에이전틱 시스템 구축을 위한 조직 전략
- 여러 팀에서 각자 agent를 구축하면서 파편화가 발생하기 쉽다
- 반대로 표준화를 빠르게 하면 창의성을 억누르고 경직된 프레임워크나 특정 벤더 솔루션에 갇힐 수 있다. 이 균형을 잡는 것이 중요
- 부서나 조직 단위에서 송통 도구와 방법론을 표준화해 협업을 간소화하면서 조직 전체의 혁신은 제한하지 않는 방식이 필요하다
- 벤더 종속 역시, 특정 기술이나 공급 업체에 대한 의존도를 낮추는 방향으로 적응력을 높여야 한다
- 효과적인 지식 공유도 핵심, 성공과 실패 사례에서 얻은 교훈을 내부 포럼, 공유 저장소, 포괄적인 문서화를 통해 널리 전파해야 한다
- 마지막으로 프레임워크는 가볍고 유연하게 유지해야 한다

# 2. Agent 시스템 설계
## 2.1 우리의 첫 번째 에이전트 시스템
- 반복되는 고객지원 요청을 프로세스 별로 구분한다음, 어느 도구를 호출할지 결정하고 올바른 파라미터로 해당 도구를 실행한 다음 확인 메시지를 보냄
- 그럼 단순한 CANCEL_ORDER agent를 만들었다.
  - > python으로 ...?
- 너무 넓거나 좁은 범위를 처리하지 않도록 주의해야 한다.
  - > 적절한 수준의 추상화가 필요하다. 프로그래밍과 똑같다
- 평가도 필요하다

## 2.2 에이전트 시스템의 핵심 구성요소
- 사용자 -> 오케스트레이션 -> 모델/도구/메모리/지식 베이스

## 2.3 모델 선택
- 모든 에이전트 기반 시스템의 중심에는 모델이 있음
- 적절한 모델 선택은 작업의 복잡성, 입력 데이터의 특성, 인프라 제약, 일반석-속도-정밀성의 트레이드오프에 달려 있다
- 일반적으로 작업 복잡도 평가에서 시작하면 된다
- GPT나 클로드 같은 대형 모델은 뛰어나지만 느리고 비용이 비싸다
- 파생 모델이나 소형 모델은 정의가 명확한 반복작업에 적합한 경우가 많다
- 최근에는 모달리티가 중요해 졌는데, 텍스트뿐 아니라 이미지, 오디오, 구조화 데이터까지 처리해야 하는 경우가 늘었음
- 오픈 소스 모델은 투명하고 필요에 따라 파인튜닝이나 수정이 가능해 규제 환경이나 보안이 중요한 경우 특히 유리할 수 있다
- GPT나 클로드는 좋지만 비싸고 커스터마이징에 한계가 있다
- 범용 사전학습 모델을 사용할지, 맞춤형 모델을 준비할지는 도메인의 특수성과 민감도에 달려 있다

## 2.4 도구
- 에이전트 기반 시스템에서 도구는 에이전트가 특정 작업을 수행하고 문제를 해결할 수 있게 한다

### 2.4.1 특정 작업을 해결하는 도구 설계
- 도구는 크게 세 가지로 나눌 수 있다
- 로컬 : 외부 의존성 없이 내부 로직과 계산만으로 작업을 수행
- API : 외부 서비스나 데이터 소스와 상호작용
- MCP : 표준 스키마를 통해 사용자 프로필, 대화 이력 등의 구조화된 실시간 컨텍스트를 프롬프트에 직접 전달

### 2.4.2 도구 통합과 모듈성
- 도구는 필요에 따라 쉽게 통합하거나 교체할 수 있도록 독립적인 모듈로 설계해야 한다

## 2.5 메모리
- 메모리는 에이전트가 정보를 저장하고 검색할 수 있게 하는 핵심 구성 요소이다

### 2.5.1 단기 메모리
- 단기 메모리는 에이전트가 현재 작업이나 대화와 관련된 정보를 저장하고 관리하는 능력, 사용자와 상호작용 중 컨텍스트를 유지하는 데 사용되며 실시간으로 일관된 의사결정을 내릴 수 있도록 돕는다
- 단기 모메리는 종종 롤링 컨텍스트 윈도 방식을 통해 구현되어며 최근 데이터를 보고 오래된 세부 정보는 잊는 방식으로 구현된다

### 2.5.2 장기 메모리
- 장기 메모리는 에이전트가 오랜 기간에 걸쳐 지식과 경험을 저장하도록 하여 과거 정보를 바탕으로 향후 행동을 결정할 수 있게 한다
- 시간에 지남에 따라 점차 발전해야 하거나 사용자 선호도에 기반한 경험을 제공해야 하는 경우 특히 중요하다
- 주로 데이터베이스, 지식 그래프, 파인튜닝된 모델을 통해 구현된다

### 2.5.3 메모리 관리 및 검색
- 효과적인 메모리 관리는 저장된 데이터를 체계적으로 구성하고 인덱싱해 필요할 때 쉽게 검색할 수 있도록 하는 것을 의미
- > LLM에서는 context를 적게 소모한다는 의미도 될듯

## 2.6 오케스트레이션
- 오케스트레이션은 개별적인 기능을 E2E 솔루션으로 전환한다
- 여러 스킬을 배치하고 상황에 따라 실행하며 전체 과정을 감독하는 논리
- 견고한 오케스트레이션 계층이 없다면 방향이 어긋나거나 전체 실행이 멈출 위험이 있다

## 2.7 설계 트레이드오프
### 2.7.1 성능: 속도와 정확도의 균형
- 중요한 트레이드오프 하나는 속도와 정확도의 균형이다
- 시스템의 종류에 따라 선택해야 하며, 하이브리드 접근으로 먼저 대략적인 결과를 제시하고 추가 시간을 이용해 보완하는 방식이 있을 수 있다
- > 현재 구현한 analyzer가 이와 유사한 동작

### 2.7.2 확장성: 에이전트 시스템의 엔지니어링적 확장
- 시스템이 커질수록 확장성은 중요한 과제가 된다
- GPU는 에이전트 시스템에서 가장 비싸고 제약이 많은 자원이기 때문에 이를 효율적으로 활용하는 것이 최우선 과제가 된다
- 동적 GPU 할당, 탄력적 GPU 할당, 우선순위 큐잉과 지능형 작업 스케쥴링을 결합해서 효율성을 높일 수 있다

### 2.7.3 신뢰성: 견고하고 일관된 에이전트
- 신뢰성은 에이전트가 일관되고 정확하게 작업을 수행할 수 있는 능력을 의미한다

**장애 허용**
- 시스템이 장애를 허용하도록 설계되어야 한다. 예를 들어, 모델이 실패할 때 대체 모델로 폴백하거나, 도구가 실패할 때 대체 도구로 폴백하는 방식이 있다

**일관성과 견고성**
- 철저한 테스트와 모니터링 그리고 피드백 루프를 통해 시스템이 일관되고 견고하게 동작하도록 해야 한다

### 2.7.4 비용: 성능과 지출의 균형
- ROI를 기준으로 평가해야 한다
- 개발 비용, 운영 비용, 비용 대비 가치를 따져서 적절한 모델을 선택하고 그에 따른 적절한 수준의 시스템 고도화가 필요하다

## 2.8 아키텍처 디자인 패턴

### 2.8.1 단일 에이전트 아키텍처
- 가장 단순하고 직관적인 설계 형태로, 하나의 에이전트가 시스템 내 모든 작업을 관리하고 실행한다

### 2.8.2 멀티 에이전트 아키텍처: 협업, 병렬성, 조율
- 여러 에이전트가 공동의 목표를 달성하기 위해 협력한다
- 협업과 전문화 : 각 에이전트가 특화된 영역이 있도록 설계한다
- 병렬성 : 여러 작업을 동시에 수행할 수 있다
- 확장성 향상 : 새로운 에이전트를 추가해 더 많은 작업을 처리하거나 부하를 분산할 수 있다
- 중복성과 복원력 : 일부 작업에 실패하더라도 다른 에이전트가 대신 수행할 수 있어 시스템의 신뢰성이 높아진다
- 조율과 통신, 복잡성 증가, 효율성 저하

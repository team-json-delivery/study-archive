# 5. 오케스트레이션
- 도구 호출뿐만 아니라 적합한 컨텍스트를 구성해 효과적이고 근거있는 행동을 유도하는 과정을 포함
    - 단순한 작업은 하나의 도구와 최소한의 컨텍스트만으로 충분
    - 복잡한 워크플로는 정교한 계획, 메모리 검색, 동적 컨텍스트 조합 필요
- 시스템이 보유한 리소스를 활용해 요청을 효과적으로 처리하도록 하는 핵심 로직
# 5.1 에이전트 유형
- 구축 가능한 다양한 에이전트 유형을 이해하는 것이 중요
    - 유형에 따라 추론, 계획, 행동의 접근 방식이 달라짐 -> 작업의 분해와 실행 방식에 직접적인 영향을 줌
    - 에이전트 유형의 선택은 시스템의 성능, 비용, 기능을 좌우함
##### 5.1.1 반사형 에이전트
- 입력과 행동을 직접 연결하며 내부 추론 과정을 거치지 않음
    - '조건이 참이면 도구 실행' 규칙을 따르며, 사전 정의된 트리거가 감지되면 즉시 적절한 도구 호출
- 지연시간이 매우 짧고 성능 예측이 가능
    - 키워드 기반 라우팅, 단일 단계 데이터 조회, 간단한 자동화 같은 용도에 적합
    - 다단계 추론이나 입력 외부의 컨텍스트를 요구하는 작업은 처리 불가능
##### 5.1.2 리액트 에이전트
- 추론과 행동을 반복 루프 내에서 교차 수행
    - 모델이 사고를 생성하고, 도구를 선택해 호출한 뒤 결과를 관찰하고 다시 계획을 갱신
    - 복잡한 작업을 여러 관리 가능한 단계로 분할
- 탐색적 시나리오(동적 데이터 분석 등)에 강하며 중간 전략을 조정할 수 있는 유연성이 큰 장점
- 로프 구조로 인해 API 비용과 응답 시간이 늘어남
##### 5.1.3 계획 후 실행 에이전트
- 작업을 두 개의 뚜렷한 단계로 나눔
    - 계획 단계: 다단계 계획을 생성
    - 실행 단계: 계획된 각 단계를 도구 호출을 통해 수행
- 계획이 명시적으로 드러나기에 디버깅과 모니터링이 쉬움
    - 명확한 분해: 복잡한 작업을 하위 작업으로 나눌 수 있음
    - 디버깅 용이성: 계획이 명시적이어서 오류가 어디 발생했는지 드러남
    - 비용 효율성: 실행이 더 작은 모델이나 더 적은 LLM 호출로 처리, 큰 모델은 계획 수립에만 집중
##### 5.1.4 쿼리 분해 에이전트
- 복잡한 질문을 반복적으로 하위 질문으로 나누고 각 하위 질문마다 검색이나 도구 호출한 뒤 최종 답변을 종합
    - '자문 후 검색'이라는 불리며, 스스로에게 묻도록 유도
    - '어떤 후속 질문이 더 필요하지?' -> 검색 호출 -> '다음 질문이 무엇이지?' ->... -> '최종 답은 뭐지?'
- 외부 지식 검색이 필요한 상황에서 효과적
##### 5.1.5 성찰형 에이전트
- 생각과 행동을 교차시키는 것에 그치지 않고, 다음 단계 진행하기 전 과거 단계를 검토 -> 실수를 찾아 수정
    - 리액트 패러다임 확장
    - 현재 상태를 의도한 결과와 비교 -> 어긋남이 생기면 계획을 조정
    - 자신의 추록 과정을 비판적으로 검토하고 논리적 오류를 수정
- 초기 오류가 연쇄적으로 이어져 비용이 큰 실패로 이어질 수 있는 고위험 워크플로에 유용
    - 예) 금융 거래 오케스트레이션, 의료 진단 지원
- 지연시간과 연산 비용이 늘어남
    - 속도 보단 정확성과 신뢰성
##### 5.1.6 심층 리서치 에이전트
- 개방형이고 매우 복잡한 조사 과제를 전문적으로 다룸
    - 예) 문헌 리뷰, 과학적 발견 등 적합
- 다음과 같은 동작을 함
    1. 전체 연구 아젠다를 계획
    2. 하위 토픽을 SELF_ASK 같은 자문 기법을 통해 구체적인 쿼리로 분석
    3. 학술 검색 API부터 도메인 특화 DB까지 다양한 도구 호출, 결과의 관련성과 신뢰성을 성찰
    4. LLM 기반 요약과 비평을 활용하여 점진적 발전하는 보고서나 추천안을 통합
- 강점
    - 역량: 전문 DB나 여러 분야의 자료에 의존하는 고복잡도, 다단계, 조사 처리 가능
    - 적응성: 새로운 증거가 나타날 때마다 연구 방향을 조정
    - 투명성: 계획과 분해 단계가 명시적으로 노출 -> 방법론 감사와 검토가 편리
- 취약점
    - 높은 비용: 광범위한 파운데이션 모델 사용 및 다수의 API 호출 -> 연산 및 토큰 비용 증가
    - 지연시간: 계획, 분해, 성찰 계층으로 인해 추가적인 지연 발생
    - 취약성: 외부 데이터 소스의 품질과 가용성에 크게 의존 -> 오류 처리와 폴백 전략 필요

##### 정리
- 각 유형은 속도, 유연성, 복잡성 측면에서의 트레이드 오프 존재

| 유형      | 강점              | 약점                  | 최적 사용 사례         |
|---------|-----------------|---------------------|------------------|
| 반사형     | 밀리초 단위 응답       | 다단계 추론 불가           | 키워드 라우팅, 단순 조회   |
| 리액트     | 유연한 적응력         | 더 높은 지연시간과 비용       | 탐색형 워크플로, 트러블 슈팅 |
| 계획 후 실행 | 명확한 작업 분해       | 계획 수립 오버헤드          | 복잡한 다단계 프로세스     |
| 쿼리 분해   | 근거가 명확한 검색 정확도  | 다수의 도구 호출 필요        | 리서치, 사실 기반 Q&A   |
| 성찰형     | 초기 오류 감지        | 추가 연산 및 지연시간        | 고위험, 안전이 중요한 작업  |
| 심층 리서치  | 다단계 및 적응형 조사 관리 | 높은 연산 비용과 매우 긴 지연시간 | 장문의 문헌 리뷰        |

# 5.2 도구 선택
- 도구 선택은 고도화된 계획 기법의 기반됨
- 도구 선택 방식에 따라 서로 다른 장점과 고려사항 존재

| 기법  | 장점                                 | 단점                            |
|-----|------------------------------------|-------------------------------|
| 표준  | 구현이 단순함                            | 도구의 수가 많아지면 확장성이 떨어짐          |
| 시맨틱 | 많은수의 도구로 확장성이 뛰어남. 구현 시, 지연 시간이 낮음 | 시맨틱 충돌 때문에 선택 정확도가 떨어지는 경우 존재 |
| 계층적 | 애무 낮은 수의 도구로 확장성이 뛰어남              | 연속으로 파운데이션 모델을 여러 번 호출해야해서 느림 |

##### 5.2.1 표준 도구 선택
- 도구의 정의, 설명을 파운데이션 모델에 함께 제공하여, 컨텍스트에 가장 적합한 도구 선택하도록 요청
  - 구현이 쉽고, 추가적인 학습이나 도구 계층 구조 없이 사용 가능
  - 단점은 지연 시간 -> 파운데이션 모델을 한 번 더 호출해야함
- 인컨텍스트 러닝에도 잘 맞음
  - 소수의 예시(퓨샷)를 제공하면 모델을 직접 학습하거나 파인튜닝하지 않고도 예측 정확도를 끌어올릴 수 있음

| 구분 | 퓨샷 (인컨텍스트 러닝) | 파인튜닝 |
|------|----------------------|----------|
| 가중치 변경 | 없음 | 있음 (영구적) |
| 방식 | 프롬프트에 예시를 넣어 그때그때 유도 | 데이터셋으로 모델을 추가 학습 |
| 지속성 | 그 대화 안에서만 | 모델에 영구히 남음 |
| 비용·난이도 | 낮음, 즉시 가능 | 상대적으로 높음, 데이터·연산 필요 |
| 적합한 경우 | 빠른 시도, 형식·스타일 조정 | 특정 도메인·작업에 지속적으로 특화 |

- 효과적인 도구 선택은 대부분 각 기능을 어떻게 서술하느냐에 달림
  - 간결하고 설명적인 이름 붙이기(process_numbers -> caculate_sum)
  - 도구의 고유 목적을 한 문장으로 요약('두 수의 합을 반환')
  - 설명에는 예시 호출을 포함해 전형적인 입/출력이 어떻게 생겼는지 보여주기
    - 모델이 도구를 보다 명확하게 이해
  - 입력 타입과 봄위를 명시해 제약을 강제('x와 y는 0에서 1,000 사이의 정수여야 합니다')
    - 애매한 매치를 줄이고 파운데이션 모델이 관련 없는 도구를 후보에서 제외
- `대표적인 프롬프트로 반복적으로 테스트하고, 도구 설명을 명료함과 구체성을 기존으로 다듬으면 학습이나 인프라 없이도 도구 선택 정확도가 크게 향상`

```
## 도구 선택 정확도 높이기 (저비용 방법)

**1. 대표 프롬프트로 반복 테스트**
실제 사용자가 할 법한 전형적 질문들을 모아 시스템을 계속 돌려본다.
→ "도구 A를 써야 하는데 자꾸 B를 부른다" 같은 실패 패턴이 드러남.

**2. 도구 설명을 명료·구체적으로 다듬기**
헷갈리는 설명을 고쳐 쓴다.
- ❌ 막연: "데이터를 가져온다"
- ✅ 구체: "주문 ID로 주문 내역 조회. 환불·배송 상태 확인에 사용"

**핵심 메시지**
파인튜닝(추가 학습)이나 새 인프라 구축 없이도, **테스트 + 설명문 다듬기**만으로 도구 선택 정확도가 크게 향상된다
```
```
# 대표 프롬프트 테스트 방법에 대한 의문과 답변

## ❓ 질문: 사람 발화는 다양한데 대표 프롬프트가 의미 있나?

**답: 절반은 맞지만, 목적을 오해한 비판.**

- 대표 프롬프트의 목적은 *모든 경우 커버*가 아니라 *체계적 실패 패턴 발견*.
- 발화는 표면적으로 무한히 다양하지만 **의도(intent)는 그렇지 않음**.
  - "환불해줘" / "돈 돌려받고 싶은데" / "결제 취소돼요?" → 표현은 달라도 같은 의도.
- 그래서 실무에서는 **의도별로 표현 변형(paraphrase)을 묶은 평가 세트**로 구성.
  - 같은 의도를 여러 어투로 테스트 → 하나라도 엉뚱한 도구로 가면 그게 잡을 결함.
- 평가의 핵심 효용은 절대 점수가 아니라 **변경 전후 비교 / 회귀 탐지**.
  - 설명문 수정 전후를 같은 세트로 비교하면 개선·악화가 분명히 보임.
```

- 작은 도구 모음에서는 잘 확장되지만,도구 라이브러리가 커질수록 정확도를 유지하고 오선택을 피하기 위해서는 도구 설명을 정교하게 설계하는 작업이 필수

##### 5.2.2 시멘틱 도구 선택
- 모든 사용 가능한 도구를 시맨틱 표현으로 인덱싱하고, 가장 관련성이 높은 도구를 가져오기
  - 선택해야할 도구 수가 줄어들고, 훨씬 더 작은 후보 집합 안에서 파운데이션 모델이 올바른 도구와 파라미터를 선택
  - 인코더 전용 모델을 통해 도구 이름과 설명을 숫자 벡터로 표현 -> 도구가 시맥틱 유사도 기반 검색 될수 있도록 임베딩 효과
  - 런타임에는 현재 컨텍스트를 같은 임베딩 모델로 임베딩 하고 데이터베이스에서 검색해 상위 도구들을 선택해 조회함
- 표준 도구 선택보다 빠르고 성능도 우수하며, 확장성도 좋음
##### 5.2.3 계층적 도구 선택
- 도구 수가 매우 많다면 계층적 도구 선택 고려
  - 특히 시맨틱하게 서로 비슷하고 더 높은 지연시간과 복잡성을 감수하더라도 정확도를 개선하고 싶을때 활용
  - 도구를 어려 그룹으로 조직하고 그룹에 대한 설명을 제공
    - 도구 선택이 먼저 그룹을 선택하고, 그룹 안에서 2차 검색을 수행 -> 도구를 고름
- 더 느리고 병렬화 비용도 많이 듬 <-> 복잡성을 두어 도구 선택 정확도를 더 높임
  - 그룹을 설계하고 유지하는 데에는 시간과 노력이 필요
- 매우 많은 수의 도구를 보유한 경우가 아니라면 권장하지 않음
# 5.3 도구 실행
- 파라미터화: 도구 실행을 안내할 파라미터를 정의하고 설정하는 과정
  - 모델이 작업을 어떻게 해석하고 특정 요구사항을 충족하도록 응답을 어떻게 조정할지 결정
  - 입력이 데이터 타입의 기본 조건을 만족하는지 확인하기 위해 기본 파서를 사용하고, 검사를 통과하지 못하면 파운데이션 모델이 형식을 수정하도록 지시하는것이 좋음
- 파라미터가 설정되면 도구 실행 단계 시작
  - 외부 데이터 소스와 도구를 통합하면 에이전트 출력의 유용성과 정확성을 크게 높일 수 있음
  - 타임아웃과 재시도 로직은 사용 사례의 지연시간 및 성능 요구사항에 맞게 조정 필요
# 5.4 도구 토폴로지
- 에이전트에 충분한 범위의 도구를 제공하면 에이전트가 도구들을 유연하게 배열 및 올바른 순서로 적용 -> 다양한 문제 해결 가능
  - 소프트웨어 설계자들은 어떤 단계들이 어떤 순서와 제어 흐름 실행 구현 -> 에이전트가 작동할 수 있는 도구와 도구 토폴로지를 구현
##### 5.4.1 단일 도구 실행
- 정확히 하나의 도구만 필요로 하는 작업
  - 작업을 해결하는데 가장 적합한 단일 도구를 선택
  - 계획의 최소 정의지만 더 복잡한 패턴을 구축하는 기반
- 고급 에이전트 시스템에서 더 복잡한 다단계 계획 및 도구 오케스트레이션 전략을 구축하는 데 기반
##### 5.4.2 병렬 도구 실행
- 여러 도구를 병렬로 실행하면 복잡성이 증가
  - 몇 개의 도구를 실행해야 할지가 명확하지 않기 때문에 문제의 복잡성이 증가
- 일반적인 접근 방식은 시맨틱 도구 선택을 사용해 실행될 수 있는 도구의 최대 개수를 먼저 가져오는 것
  - 반복 호출 및 추가 컨텍스트를 통해 더 이상 도구 추가하지 않기록 결정할때까지 도구 추가
  - 도구가 선택되면 각 독립적으로 파라미터를 설정하고 실행 -> 완료되면 사용자에게 제공할 최종 응답 작성
- 전체 지연 최소화하면서 더 풍부하고 정보에 기반한 출력을 제공
##### 5.4.3 체인
- 일련의 작업을 순차적으로 실행하는 패턴
  - 이전 작업이 완료되어야만 이어서 수행 가능
  - 체인 계획은 특정 목표를 달성하기 위해 어떤 작업을 어떤 순서로 수행해야 하는지 결정하고 작업이 끊김 없이 다음 작업으로 이어지도록 보장하는 과정
  - 단계별 절차나 선형 워크플로가 필요한 작업에서 흔히 사용
- 체인 길이가 길어질수록 오류가 누적될 수 있으므로 도구 체인에는 최대 길이를 설정할 것을 강력히 권한
##### 5.4.4 그래프
- 체인이나 트리보다 훨씬 더 풍부하게 복잡한 비계층형 플로를 모델링함
  - 조건부 엣지와 병합 엣지를 모두 정의할 수 있으며 병렬 경로가 다시 공유 노드로 합쳐지게 만들 수 있음
- 복잡하고 비선형적인 워크플로를 모델링하는 데 있어 최상의 유연성 제공
  - 여러 도구 실행 경로를 분기하고 다시 합치고 통합해 하나의 워크플로로 엮을 수 있음
- 더 많은 LLM 호출, 더 깊은 라우팅 로직 등 추가 오버헤드가 따름
  - 사용 사례의 요구사항을 단단히 기반을 두고 불필요하게 구조를 복잡하게 만들고 싶은 유혹을 경계 필요

##### 요약
- 체인: 작업이 업격하게 선형적(프롬프트 -> 모델 -> 파서)일 경우
  - 추론과 디버깅이 쉬움
- 그래프: 여러 정보 스트림을 분기했다가 나중에 다시 합쳐야하는 상황(하나의 요약으로 모이는 병렬 분석 단계들)
- 실계 설계 시 토폴로지를 스케치하는 것이 좋음
  - 전이 화살표를 그리며 분기들이 다시 합쳐지는 지점을 표시
  - 점진적으로 구현 -> 각 라우터에 대해 단위 테스트 작성
- 구조를 가능한 한 단순하게 유지 필요
  - 노드나 엣지가 추가될 때마다 잠재적인 실행 경로와 오류 모드가 기하 급수적으로 늘어남
  - 그래프 패턴은 정말로 복잡한 시나리오를 위해 아껴두는 편이 좋음

# 5.5 컨텍스트 엔지니어링
- 에이전트 계획의 각 단계가 효과적으로 작동하는 데, 올바른 정보와 지침을 갖추도록 보장
  - 프롬프트 엔지니어링은 효과적인 지침을 작성에 집중
  - 컨텍스트 엔지니어링은 모든 입력을 동적으로 조합해 작업 성능 최대화 하는 구조적이고 토큰 효율적인 컨텍스트 윈도를 만드는 작업
  - 계획과 실행을 이어줌, 에이전트 워크플로가 일관, 현실에 기반하며 사용자의 정렬된 상태를 유지하도록 도움
- 핵심
  - 어떤 정보를 포함할지
  - 최대한 명료하고 관련성 있게 보이도록 어떻게 구조화 할지
  - 토큰 한도 안에 얼마나 효율적으로 담을지
- 에이전트가 다단계 워크플로를 오케스트레이션하거나 복잡한 작업 처리하기 시작하면 일관성, 정확성, 유용성 유지를 위해 동적 컨텍스트 구성이 필수
- 효과적인 컨텍스트 엔지니어링 핵심 실천 사항
  - 가장 유용한 정보만 검색하고 선택해 관련성 우선
  - MCP 같은 구조화된 포멧을 사용하여 상태와 검색된 지식을 해결 가능한 방식으로 모델에게 전달하여 명료성 유지
  - 히스토리가 길어지면 요약을 적용해 핵심 정보를 보존, 불필요한 표현 제거로 토큰 사용량 최적화
  - 각 추론 스텝마다 에이전트의 목표, 워크플로 단계, 사용자 입력을 반영하도록 동적 조립
- 각 단계가 효과적으로 실행되는 데 필요한 올바른 정보 갖추도록 책임
  - 부실한 컨텍스트는 성능도 약화시킬수 있음

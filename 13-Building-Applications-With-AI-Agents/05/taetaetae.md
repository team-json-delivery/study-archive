# 5장: 오케스트레이션

**개요**

* **핵심 역할**: 에이전트가 보유한 도구들을 조율하여 실제 작업을 해결하는 과정.
* **주요 요소**: 단순히 호출할 도구를 결정하는 것을 넘어, 각 모델 호출 시 효과적이고 근거 있는 행동을 보장하기 위한 **적절한 컨텍스트 구성** 포함.

---

## 5.1 에이전트 유형

### 5.1.1 반사형 에이전트 (Reflex Agents)

* **동작 방식**: 사전 정의된 규칙("조건-행동")에 따라 입력값을 즉각적으로 도구와 매핑.
* **특징**: 내부 추론 과정 생략. 최저 지연 시간(Low latency) 및 예측 가능한 성능.
* **적합한 작업**: 키워드 기반 라우팅, 단일 단계 데이터 조회, 단순 자동화. 복잡한 다단계 추론 불가.

### 5.1.2 리액트 에이전트 (ReAct Agents)

* **동작 방식**: 추론(Reasoning)과 행동(Action)을 반복하는 루프 (생각 → 도구 선택/호출 → 결과 관찰 → 반복).
* **특징**: 작업 중단 및 방향 수정 등 동적 적응력 우수. '사고의 사슬(Chain of thought)'을 통한 투명성 제공.
* **적합한 작업**: 탐색적 시나리오(동적 데이터 분석, 트러블슈팅 등). 단, API 비용 및 응답 시간 증가 단점 존재.

### 5.1.3 계획 후 실행 에이전트 (Planner-Executor Agents)

* **동작 방식**: 다단계 계획을 생성하는 **계획(Planning) 단계**와 도구를 호출해 처리하는 **실행(Execution) 단계**로 분리.
* **특징**: 명확한 작업 분해, 쉬운 디버깅.
* **장점**: 실행 단계에 소형 모델을 사용하여 비용 효율성 극대화. 복잡한 다단계 프로세스에 적합.

### 5.1.4 쿼리 분해 에이전트 (Query-Decomposition Agents)

* **동작 방식**: 복잡한 질문을 여러 하위 질문으로 쪼개고, 각각 도구(검색 등)를 호출한 뒤 최종 답변을 합성 ("self-ask with search").
* **적합한 작업**: 정확한 사실 관계가 중요한 외부 지식 검색 및 Q&A.

### 5.1.5 성찰형 에이전트 (Reflection Agents)

* **동작 방식**: ReAct 패턴의 확장판. 목표 상태와 현재 상태를 비교하며 과거 단계를 검토하고 실수를 스스로 수정.
* **특징**: 인간의 자기 평가(Self-assessment) 모방. 높은 연산량과 지연 시간 발생.
* **적합한 작업**: 금융 거래, 의료 진단 등 초기 오류가 치명적인 **고위험/안전 필수 워크플로**.

### 5.1.6 심층 리서치 에이전트 (Deep Research Agents)

* **동작 방식**: 계획-실행, 쿼리 분해, ReAct 패턴의 결합. 광범위한 정보 수집, 가설 검증, 합성 반복.
* **특징**: 고도의 복잡성 처리 및 적응력 우수. 투명한 방법론 감사 가능.
* **단점**: 극도로 높은 컴퓨팅 비용과 긴 지연 시간, 외부 데이터 의존성(취약함).
* **적합한 작업**: 학술 문헌 조사, 기술 실사 등 속도보다 **깊이와 정확성**이 중요한 장문/전문가 수준의 작업.

## 5.2 도구 선택

### 5.2.1 표준 도구 선택 (Standard Tool Selection)

* **방식**: 모든 도구의 정의와 설명을 파운데이션 모델에 제공하여 문맥에 맞는 도구를 직접 선택하게 함.
* **장점**: 구현이 단순하고 별도의 인프라나 학습 불필요. 퓨샷(Few-shot) 예시를 통한 성능 향상 용이.
* **단점**: 도구 수가 많아질수록 모델 호출로 인한 지연 시간(Latency) 증가 및 설명 중복으로 인한 선택 오류 발생. (정밀한 도구 메타데이터 작성 필수).

### 5.2.2 시맨틱 도구 선택 (Semantic Tool Selection)

* **방식**: 도구 설명을 임베딩하여 벡터 DB에 저장 후, 사용자 쿼리와의 의미론적 유사도(Semantic search)를 기반으로 상위 도구를 먼저 검색 → 이후 모델이 최종 선택.
* **장점**: 대규모 도구 세트에서도 뛰어난 확장성, 표준 선택 대비 빠른 속도. **대부분의 사례에 권장되는 패턴**.

### 5.2.3 계층적 도구 선택 (Hierarchical Tool Selection)

* **방식**: 도구를 논리적 그룹으로 분류 → 1차로 적합한 '그룹' 선택 → 2차로 그룹 내에서 특정 '도구' 선택.
* **장점**: 시맨틱 충돌을 줄여 선택 정확도 극대화. 수많은 도구를 가진 환경에 적합.
* **단점**: 순차적인 LLM 호출로 인한 높은 지연 시간과 복잡도 증가.

---

## 5.3 도구 실행

* **매개변수화(Parametrization)**: 도구 정의에 따라 모델이 인풋 데이터 타입을 올바르게 채워 넣는 과정. 현재 에이전트 상태, 시간, 위치 등 추가 컨텍스트 주입 가능.
* **유효성 검사**: 기본 파서(Parser)를 사용해 입력값이 데이터 형식 기준을 충족하는지 확인 및 자동 수정 지시.
* **실행 및 예외 처리**: 로컬 또는 원격 API 실행. 사용 사례의 성능 요구사항에 맞춘 타임아웃 및 재시도(Retry) 로직 필수.

---

## 5.4 도구 토폴로지

### 5.4.1 단일 도구 실행 (Single Tool Execution)

* **구조**: 단 하나의 도구만 선택 → 매개변수화 → 실행 → 최종 응답 구성.
* **특징**: 가장 단순한 형태. 낮은 지연 시간. 복잡한 시스템의 기초.

### 5.4.2 병렬 도구 실행 (Parallel Tool Execution)

* **구조**: 다수의 출처에서 정보를 수집해야 할 때, 여러 도구를 동시에 실행.
* **특징**: 모델이 필요한 도구 목록을 한 번에 확정하고 병렬 처리 후 결과를 통합. 전체 지연 시간을 최소화하면서 풍부한 결과물 도출.

### 5.4.3 체인 (Chains)

* **구조**: 이전 행동의 성공적 완료에 의존하여 도구들이 순차적으로 실행되는 선형 워크플로.
* **특징**: LangChain의 LCEL(선언적 구문) 등을 활용해 구축. 오류 누적 방지를 위해 체인의 최대 길이 제한 권장.

### 5.4.4 그래프 (Graphs)

* **구조**: 다중 의사결정 지점, 조건부 엣지(Conditional edges), 병렬 경로의 병합(Consolidation)을 지원하는 비계층적/비선형 흐름.
* **특징**: LangGraph 등을 활용하여 가장 복잡한 분기 및 통합 로직 표현 가능.
* **단점 및 유의사항**: 비용 및 지연 시간 급증, 무한 루프 등 잠재적 오류 증가. 반드시 필요한 복잡한 시나리오에만 도입하고 점진적 설계 권장.

---

## 5.5 컨텍스트 엔지니어링

* **개념**: 단순한 프롬프트 작성을 넘어, 사용자 메시지, 검색된 지식, 워크플로 상태, 시스템 프롬프트를 **토큰 효율적으로 조립**하는 과정.
* **목적**: 계획과 실행의 가교 역할. 에이전트가 정확하고 일관성 있게 작업에 임하도록 정보 제공.
* **핵심 실천 방안**:
  1. 대량의 텍스트 단순 추가 지양, **가장 관련성 높은 정보만 검색/우선순위화**.
  2. MCP(Model Context Protocol) 등 스키마를 활용한 **구조화 및 명확성 유지**.
  3. 과거 이력의 **요약(Summarization)** 을 통한 토큰 최적화.
  4. 현재 목표와 진행 단계에 맞춘 **동적 조립**.



---

## 5.6 마무리

* 에이전트의 성공은 적절한 **오케스트레이션 전략(지연 시간 vs. 정확도의 트레이드오프)** 에 달려 있음.
* **권장 접근법**:
  * 사용 사례의 복잡도(필요 행동 수, 계획 변경 빈도 등)를 평가하여 **가장 단순한 계획 방식부터 시작**.
  * 체계적인 테스트 케이스를 통해 점진적으로 복잡성 확장.

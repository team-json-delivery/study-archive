# Chapter 5. Orchestration (오케스트레이션)

## 1. 오케스트레이션이란

- 도구를 “가지는 것”과 “잘 쓰는 것”은 다르다. **오케스트레이션**은 에이전트가 가진 자원(도구·기억·컨텍스트)을 **언제·어떤 순서로·어떻게 조합**해 사용자 요청을 해결할지를 다룬다.
- 단순 작업: 도구 하나 + 최소 컨텍스트면 충분
- 복잡한 작업: 계획(planning) + 기억 검색(memory retrieval) + 동적 컨텍스트 조립이 필요
- 이 장의 범위: 에이전트 유형 → 도구 선택 → 도구 실행 → 토폴로지 → 컨텍스트 엔지니어링

## 2. 에이전트 유형 (Agent Types)

복잡도·자율성에 따라 여러 패턴이 있다. 아래로 갈수록 더 정교/자율적.

### 2-1. Reflex Agents (반사형)

- 입력 → 정해진 규칙/매핑으로 **즉각 반응**
- 추론·계획 거의 없음, 가장 단순·빠름
- 단순 분류·라우팅 같은 작업에 적합

### 2-2. ReAct Agents (Reasoning + Acting) ⭐

- **추론(Thought) → 행동(Action) → 관찰(Observation)** 루프를 반복
- 도구 호출 결과를 보고 다음 단계를 다시 추론 → 동적 문제 해결
- 가장 널리 쓰이는 기본 패턴

```text
Thought: 사용자 질문에 답하려면 날씨 정보가 필요
Action: get_weather("Seoul")
Observation: 맑음, 22도
Thought: 정보를 얻었으니 답변 가능
Answer: 오늘 서울은 맑고 22도입니다.
```

### 2-3. Planner-Executor Agents (계획-실행 분리)

- **Planner**가 전체 계획(단계 목록)을 먼저 세우고, **Executor**가 단계별로 실행
- 복잡·다단계 작업에서 일관성↑, 중간 점검 용이
- 계획과 실행 책임을 분리 → 디버깅·재시도 쉬움

### 2-4. Query-Decomposition Agents (질의 분해)

- 큰 요청을 **여러 하위 질문/작업으로 분해**해 각각 처리 후 결과를 합침
- 예: “A와 B를 비교해줘” → A 조사 + B 조사 + 비교 종합

### 2-5. Reflection Agents (자기 점검)

- 자신의 출력/중간 결과를 **스스로 평가·비판하고 개선**하는 루프
- (Reflexion 등) 오류 자기수정으로 품질↑, 단 비용/지연↑

### 2-6. Deep Research Agents (심층 리서치)

- 여러 출처를 **반복 검색·교차검증·종합**해 장문의 리포트를 생성
- 다단계 계획 + 도구 다중 호출 + 메모리/컨텍스트 관리가 모두 결합된 고급 형태

## 3. 도구 선택 (Tool Selection)

도구가 많아질수록 “어떤 도구를 쓸지” 고르는 게 병목이 된다. 도구 수에 맞는 선택 전략이 필요.

### 3-1. Standard Tool Selection (표준/생성형 선택)

- 모든 도구 명세를 프롬프트에 넣고 **모델이 직접 선택**
- 도구가 적을 때 간단·효과적
- 도구가 많아지면 컨텍스트 폭증 + 정확도 저하

### 3-2. Semantic Tool Selection (의미 기반 선택)

- 사용자 질의와 도구 설명을 **임베딩**해, 의미적으로 유사한 도구 후보만 추려서 모델에 제시
- 벡터 검색으로 후보를 줄여 토큰·정확도 개선
- 도구 수가 많을 때 핵심 전략

### 3-3. Hierarchical Tool Selection (계층적 선택)

- 도구를 **카테고리/그룹으로 묶고** 상위 그룹 → 하위 도구 순으로 단계적 선택
- 수백 개 도구도 트리 구조로 효율적 탐색
- 대규모 도구 레지스트리에 적합

## 4. 도구 실행 (Tool Execution)

선택한 도구를 실제로 호출하는 단계에서 고려할 점.

- **인자 파싱·검증**: 모델이 만든 인자가 스키마에 맞는지 확인
- **에러 핸들링·재시도**: 실패 시 재시도, 폴백, 오류를 모델에 피드백해 재추론
- **타임아웃·관찰값 정리**: 결과(Observation)를 모델이 읽기 좋게 가공
- **비용·지연 관리**: 불필요한 호출 줄이기

## 5. 도구 토폴로지 (Tool Topologies)

도구 호출들을 **어떤 구조로 연결**하는가.

### 5-1. Single Tool Execution (단일 실행)

- 한 번에 도구 하나 호출. 가장 단순.

### 5-2. Parallel Tool Execution (병렬 실행)

- 서로 독립적인 도구들을 **동시에** 호출 → 지연 단축
- 예: 여러 소스를 동시에 검색
- 주의: 결과 병합, 부분 실패 처리

### 5-3. Chains (체인 / 순차)

- 한 도구의 **출력이 다음 도구의 입력**이 되는 순차 연결
- 단계적 파이프라인에 적합 (예: 검색 → 요약 → 번역)
- 단점: 앞 단계 실패가 전체에 전파

### 5-4. Graphs (그래프)

- 분기·병합·반복이 가능한 **유연한 DAG/그래프** 구조
- 조건부 흐름, 병렬+순차 혼합, 루프 표현 가능 (LangGraph 등)
- 가장 표현력이 높지만 설계·디버깅 복잡

|토폴로지    |구조      |적합 상황           |
|--------|--------|----------------|
|Single  |단일 호출   |단순 작업           |
|Parallel|동시 호출   |독립적 다중 작업, 지연 단축|
|Chain   |순차 연결   |단계적 파이프라인       |
|Graph   |분기·병합·반복|복잡한 조건/멀티스텝 워크플로|

## 6. 컨텍스트 엔지니어링 (Context Engineering) ⭐

모델의 컨텍스트 윈도우는 유한 → **무엇을 넣을지 동적으로 잘 조립**하는 것이 성능을 좌우.

- 매 단계 필요한 정보(관련 기억·도구 결과·지시문)만 **선별·조립**
- 불필요한 내용은 빼서 토큰·노이즈·비용 절감 → 정확도↑
- 너무 많은 컨텍스트 = 혼란/지연/비용, 너무 적은 컨텍스트 = 정보 부족
- 동적 컨텍스트 조립 = 좋은 오케스트레이션의 핵심

## 7. 핵심 정리 (Key Takeaways)

1. 오케스트레이션 = 자원(도구·기억·컨텍스트)을 **언제·어떻게 조합**할지의 문제.
1. 에이전트 유형: Reflex → **ReAct** → Planner-Executor → Query-Decomposition → Reflection → Deep Research (자율성·복잡도 증가).
1. 도구가 많아지면 Standard → **Semantic(임베딩)** → **Hierarchical(계층)** 선택으로 확장.
1. 토폴로지: Single / Parallel / Chain / **Graph** — 작업 구조에 맞게 선택.
1. **컨텍스트 엔지니어링**(필요한 것만 동적으로 조립)이 안정적 멀티스텝 수행의 열쇠.

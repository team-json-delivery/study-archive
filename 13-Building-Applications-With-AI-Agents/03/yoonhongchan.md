# 3. 에이전트 시스템을 위한 UX디자인
- 에이전트 시스템이 자리잡으면서 이들이 제공하는 UX는 성공을 좌우하는 핵심 요소가 됨
    - 파운데이션 모델과 에이전트 아키텍처는 뛰어난 기술적 능력을 구현
    - 효과성, 신뢰성, 도입을 결정하는건 사용자가 에이전트와 상호작용하는 방식
- 잘 설계된 에이전트 경험은 사용자의 역량 강화, 신뢰를 쌓으며 불편을 해소, 에이전트 능력과 한계를 명확히 드러냄
- UX 고려사항은 시간일 길어질 경우 컨텍스트를 어떻게 관리하냐가 관건 -> 과거 컨텐스트 유지 시, UX 상향
    - 일부는 메모리를 단기간만, 일부는 시간이 지나도 메모리 유지해 과거 상호 작용 기억
- 에이전트 능력, 한계, 불확실성을 명확히 알리는 일은 현실적인 기대치를 설정하고 오해를 방지하는 데 필수
- 신뢰와 투명성은 에이전트 시스템의 긍정적 사용자 경험을 떠받치는 토대
    - 예측 가능한 행동과 명확한 행위 설명은 에이전트를 신뢰하고 의존하도록 도움

# 3.1 모달리티
- 에이전트 시스템은 다양한 모달리티로 사용자와 상호작용
    - 모달리티마다 고유한 강점, 한계, 설계 고려 사항이 존재
        - 텍스트, 그래픽 인터페이스, 음성, 비디오 중 선택은 상호작용하는 방식을 좌우
##### 3.1.1 텍스트 기반 인터페이스
- 가장 흔하고 범용적인 방식
    - 단순성, 친숙함, 기존 워크플로에 쉽게 통합되는 점이 채택 이유
    - 동기식 및 비동기식 상호작용 모두 지원
    - 컨텍스트 유지 및 상호작용 교환 내역이 명확함
    - 투명성, 책임성, 트러블 슈팅이 쉬움
- 개발자가 자연어 프롬프트로 복잡한 문법이나 플래그를 위우지 않고 생성, 실행, 디버깅 가능
    - 기존 터미널은 정확한 문법 지식과 플래그 암기 요구
    - AI 터미널은 사용자 의도를 해석하고 모범 사례를 제안하며 실시간으로 오류를 디버깅
- 텍스트 기반 인터페이스 한계
    - 사용자가 에이전트가 지원하는 능력이나 효과적인 명령 표현 방식을 모름 -> 버튼이나 메뉴로 알려주지 못함
        - 요청 지원 범위를 벗어날 때 사용자의 불편함이 커짐
- 텍스트 기반 에이전트 효과적인 설계
    - 동적 제안을 통해 지원 기능을 선제적으로 알려줘야함
        - '오늘 무엇을 도와 드릴까요?' -> '주문 취소, 배송 상태 확인, 계정 정보 업데이트를 도와드릴 수 있어요' 라고 해야 작동 범위를 이해하도록 해 시행착오를 줄임
    - 명확성, 컨텍스트 유지, 오류 관리에 세심한 주의 필요
        - 모호하지 않고 간결하게 소통
        - 대화기 길어지면 컨텍스트 유지
        - 명확한 오류 메시지와 폴백 매커니즘 제공
            - 요청을 수행할 수 없으면 인간 상담원에게 전달하거나 대안 제시
- 신중하게 설계된 텍스트 기반 에이전트는 다양한 컨텍스트에서 신뢰할 수 있고 적응력이 높이며 매우 유용함

##### 3.1.2 그래픽 인터페이스
- 텍스트, 버튼, 아이콘 등 여러 그래픽 요소를 결합해 사용자와 에이전트 시스템이 시각적인 상호작용
    - 순수 텍스트나 음성만으로 부족할 수 있는 작업에 효과적
    - 대시보드 기반 AI 도구, 그래픽 채팅 인터페이스
- 강점으로 정보를 시각적으로 제시 -> 인지 부하를 줄임
    - 잘 설계한 인테페이스는 직관적이고 이해하기 쉬운 형식으로 보여줌
        - 진행률 막대 표시, 경고 아이콘 등
    - 예) 워크플로 관리 하는 에이전트, 최신 AI 지원 IDE 인터페이스
- 그래픽 에이전트 인터페이스에서 새롭게 떠오르는 분야는 생성형 UI의 등장
    - 정적인 대시보드나 사전 설계된 레이아웃에만 의존하지 않는 생성형 UI
        - 사용자의 질의에 따라 인터페이스 요소, 데이터 시각화, 구조화된 출력을 동적으로 만듬
        - 미리 정의된 워크플로를 넘어 시각적 구조화가 이해를 높여주는 개방형 과제로 확장
            - `사용 가능하고 미적으로 일관되며 부실하거나 과도한 정보로 사용자를 압도하지 않도록 해야함` -> 신중한 디자인 패턴, 레이아웃 제약, 우선순위 부여 로직 중요
- 그래픽 에이전트 설계 과제는 화면의 물리적 공간이 한정적이라는 과제 존재
    - `중요한 정보를 노이즈 속에 파묻지 않도록 우선순위를 정해야함`
    - 비동기적 실시간 업데이트 및 그래픽 요소가 화면 크기에 맞게 우아하게 적응해 일관성 유지 필요
- 자동화와 사용자 통제의 균형도 중요

##### 3.1.3 음성 인터페이스
- 말을 주된 소통 방식으로 상호작용(시리 같은 가상 비서)
    - 수동입력이 비현실적이거나 불가능한 상황에 특히 강함
- 지연시간이 큰 장애물
    - 실시간 처리를 위해 인식, 의도 인식, 적절한 응답 생성 필요 -> 대화 흐름을 때는 지연 발생 이슈
- 중단을 우아하게 처리하는 능력이 매우 중요
    - 말을 멈추거나 설명을 덧붙이는 등 사용자가 중간에 멈춰도 혼란 없이 받아들이는 능력 중요
- 음성 에이전트 워크플로 통합이 큰 도약
    - 정적 답을 반환하는데 끝나지 않고 여러 액션 수행
- 별도의 조작이 필요 없는 작업, 빠른 질의, 액션 중심 워크플로에서 가장 효과적임

##### 3.1.4 비디오 기반 인터페이스
- 시각, 청각, 텍스트 요소를 하나의 일관된 경험으로 결합
    - 대면 대화를 모사하는 비디오 아바타
- 시각적 단서, 음성, 텍스트 오버레이, 애니매이션 등 여러 감각 채널을 결합 해 풍부하고 표현력 있는 상호 작용 제공
- 기술적, 디자인적 과제
    - 상당한 연산 성능과 대역폭 필요 -> 랙이나 화질 거하고 나쁜 UX 제공
    - 렌더링, 실시간 애니매이션, 대역폭 최적화가 해결되면 크게 성정

##### 3.1.5 모달리티 결합을 통한 매끄러운 경험
- 가장 매력적인 에이전트 UX는 여러 모달리티를 하나의 응집된 사용자 여정으로 결합
    - 모달리티를 넘나들면서 상태와 컨텍스트를 유지하는 능력이 중요
- 기술이 빠르게 확장되더라도 핵심 UX와 제품 원칙은 변하지 않음
    - 성공적인 에이전트 구축은 최신 모달리티 통합이나 생성형 UI를 과시하는데 있지 않음
    - 사용자를 깊이 이해하고 직관적이며 즐거운 경험을 제공해 실제 문제를 해결하는 것
- 최고의 제품은 기술을 통해 인간의 역량을 우아하고 눈에 거슬리지 않게 증폭하는 제품

##### 3.1.6 자율성 슬라이더
- UX 설계에서 에이전트 자율성 수준 부여는 매우 중요
    - 완전 수동 제어, 부분 자동화, 완전 자율 운영까지 부드럽게 조정할 수 있어야함
- `자율성 슬라이더`는 사용자가 특정 시점에 유지할 통제와 위임할 범위를 선택할 수 있게 함
    - 슬라이더 바를 통해 수동, 요청, 완전 자율 -> 선호, 작업 복잡도, 컨텍스트와 맞춰 사용자 신뢰 구축
- 사용자, 작업, 컨텍스트에 따라 요구되는 에이전트 자율성의 정도는 다름
    - 케이스 별 정밀함을 완전 수동 제어 선호, 반복적이고 복잡한 작업은 아예 위임
        - `이런 선호가 고정되지 않음` -> 사용자 신뢰, 작업 친숙도, 위험도, 업무량에 따라 변화
            - 수동: 모든 코드 직접 작성, 문법 하이라이트와 린팅만 제공
            - 요청(보조): 코드 자동 완성, 리팩토링 등 선제적 제안하지만 적용 전에는 개발자가 각 제안을 검토
            - 완전 자율: 개별 승인 없이 자율적으로 수행, 변경 사항을 알림 받지만 승인은 하지 않음
- 단일 인터페이스에 개발자가 통제와 효율의 균형을 잡도록 도움
- 효과적으로 통합하려면 다음 설계 원칙 고려
    - 자율성의 단계를 명확히 하라
        - 수동, 보조, 자동 처럼 직관적인 용어를 통해 사용자가 단계 이해
    - 매끄러운 전환을 가능하게 하라
        - 사용자 신뢰, 컨텍스트, 업무량이 변할 때 자율성 수준을 손쉽게 변경 가능
            - 검토 모드에서 자동 승인 모드로 빠르게 변경 가능
    - 각 수준에서 예측 가능하고 투명한 기능을 제공하라
        - 수준마다 작동 과정을 잘 정의해야함
    - 각 수준의 위험과 이점을 소통하라
        - 자율성을 높일 때 얻는 이점과 감수하는 위험을 사용자가 인지
    - 사용자 신뢰와 역량에 따라 자율성을 조절하라
        - 사용자의 신뢰가 쌓일수록 자율성 수준을 높이도록 제안
- 자율성 슬라이더는 신뢰를 구축하는 메커니즘(단순 기능 x)

# 3.2 동기식 대 비동기식 에이전트 경험
- 동기식 또는 비동기식 모드는 각 장단점 존재
    - 동기식
        - 실시간 상호작용으로 즉각적 대화 -> 흐름과 몰입을 유지하려면 빠른 응답 필수
            - 음성 대화, 채팅 인터페이스
    - 비동기식
        - 시간 간격을 두고 이따금 소통
            - 작업 알림, 처리 완료 후 전달되는 에이전트 생성 보고서
- 작업의 성격, 사용자의 기대, 운영 컨텍스트에 따라 방식 선택
    - 동기식
        - 즉각적 피드백이나 실시간 의사결정이 필요
    - 비동기식
        - 시간이 오래 걸리거나 백그라운드 처리가 필요
        - 사용자의 지속적인 주의가 요구되지 않는 워크플로
- 두 패턴 모두 유용하지만 어떤 경험이 어느 범주에 속하는지 미리 정해두는 것 권장

##### 3.2.1 동기식 경험을 위한 디자인 패턴
- 즉시성, 명확성, 반응성이 중요
    - 신속히 응답하고 눈에 띄는 지연 없이 대화 흐름과 컨텍스트 유지
    - 낮은 지연과 컨텍스트 유지는 필수
- 응답에서 명료함과 간결함이 우선
    - 장황한 설명이나 과도하게 복잡한 출력은 실시간 상호작용을 깨뜨림
- 오류 처리가 동일하게 중요함
    - 중단된다고 해도 상호작용을 망치지 않고 우아하게 회복
    - 불확실성이 생기면 위험한 추정보단 명확한 질문을 하거나 부드럽게 방향을 전환
##### 3.2.2 비동기식 경험을 위한 디자인 패턴
- 유연성, 지속성, 장기적 명료성 우선
    - 장시간 실행 작업 처리 등 즉각 응답이 필요하지 않은 컨텍스트에서 자주 발생
- 작업 상태와 결과를 명확히 전달
    - 사용자는 에이전트가 무엇을 하고 있는지, 작업이 어느 단계인지 항상 이해할 수 있어야함
- 컨텍스트 관리도 중요
    - 상호 작용에 긴 지연이 생길 수 있으므로 과거 컨텍스트를 끊김 없이 보존하고 참조
- 명확한 일정, 진행 지표, 후속 알림 등 가시성 부족으로 인한 좌절을 예방 필요
##### 3.2.3 능동적 개입과 침해적 행동 사이의 균형
- 에이전트가 언제, 어떻게 사용자를 능동적으로 참여 시켜야하는지 결정하는건 어려운 일
    - 잘하면 큰 도움이 되지만, 안좋으면 완전히 이탈하게 만듬
- 능동성의 균형은 컨텍스트인식과 사용자 통제에 달림
    - 사용자의 현재 집중 대상, 긴급도, 커뮤니케이션 선호 이해 필요
- 능동적으로 연락할 때는 관련성을 최우선해야 함
    - 알림과 제안은 문제를 해결하거나 인사이트를 제공해 도움이 되어야 함

# 3.3 컨텍스트 유지와 연속성
- 컨텍스트 유지와 연속성을 보장하는 것은 중요
    - 중단된 대화를 이어가거나 과거 상호작용에 대한 컨텍스트 유지는 사용성, 효율, 신뢰에 영향
- 컨텍스트 유지는 기술적 능력이지만 UX 관점의 문제
    - 주의 깊은 협력자로 느끼게 해야함 -> 반복을 강요하는 단절된 도구 느끼게 하면 안됨
- 구현 방법은 UX를 직접 규정 -> 사용자 여정, 프라이버시 요구사항, 의도화 개인화 수준에 따라 달림
    - 클라이언트 컨텍스트는 빠르게 느껴지지만 로그인 간 연속성을 잃어 매끄러운 UX를 해침
    - 서버 컨텍스트는 장기 메모리와 크로스 디바이스 경험을 가능 -> 지연시간이나 프라이버스 고려 사항 초래
    - 하이브리드 접근은 대체로 최적의 UX 균형을 달성
    - `궁국적으로 컨텍스트는 UX`
- 효과적인 컨텍스트 유지를 위해 단기와 장기 메모리를 모두 적절히 관리 필요
    - 단기메모리: 방금 전 질문의 세부나 지침을 유지
    - 장기 메모리는: 여러 세션에 걸쳐 선호, 과거 상호작용, 더 넒은 사용자 패턴을 보전
- 컨텍스트 관리는 신경 쓸 요소가 많음
    - 데이터 지속성, 프라이버시 문제, 메모리 한계 등 고려

##### 3.3.1 상호작용 간 상태 유지
- 상태 관리는 컨텍스트 연속성을 떠받치는 중요한 역할
    - 에이전트는 지금까지 무엇이 일어났는지 사용자가 무엇을 달령하려는지 정확히 추적 필요
    - 멀티턴 대화, 작업 인계, 중간 상태가 있는 워크플로에서 특히 중요
        - 컨텍스트를 잃으면 좌절, 비효율, 작업 포기
- 효과적인 상태 관리는 사용자나 세션을 어떻게 식별, 추적하느냐에 달려 있음
    - 기기와 세션을 넘나드는 메모리에 지속성 제공
    - 익명의 사용자는 보통 쿠키나 토큰 같은 세션 식별자가 필요
- 대규모로 확장되면 세션 상태를 메모리에만 두어서는 안됨
    - 데이터베이스나 분산 캐시로 유지 -> 서버 재시작 이후에도 연속성 보장, 로드밸런스 지원, 멀티디바이스 경험
- 사용자 기반 메모리(지속적)와 세션 기반 메모리(휘발성) 간 선택은 프라이버시 요구, 사용자 기대, 운영 아키텍처에 좌우됨
- 효과적인 상태 유지는 세션 경계, 데이터 검증, 폴백 매커니즘 요구
    - 컨텍스트를 잊는다면 잘못된 추정보단 명확한 질문으로 우아하게 회복
    - 개인 식별 가능 정보는 보안과 책임 있는 관리 필수
- 상태 관리를 잘 구현하면 불필요한 반복 없이 복잡한 작업을 안내하고 인지 부하를 줄이며 협업 지속 느낌을 줌
##### 3.3.2 개인화의 적응성
- 단순한 컨텍스트 기억을 넘어 과거 상호작용과 선호를 사용해 에이전트의 행동, 응답, 추천을 사용자별로 조정
    - 이전 교환에서 학습해 점점 더 정교하고 관련성 높은 결과 제공
        - 선호 보존: 자주 선택하는 사용자 설정 기억
        - 행동 적응: 사용자 패턴을 기반해 응답 스타일이나 상호 작용 흐름 조정
        - 능동적 지원: 과거 행동을 바탕으로 사용자 필요를 예측하고 제안
- 어떤 데이터를 저장하고 어떻게 사용하는지에 대한 투명한 소통 등 프라이버시 문제 따름
- 적응성과 과도한 지속성 사이에 균형 필요
    - 언제든 개인화 설정 초기화하거나 재정의를 할수 있어야함
# 3.4 에이전트 능력 커뮤니케이션
- 사용자가 에이전트가 무엇을 할 수 있으며, 어떻게 상호작용하는지 명확히 이해하게 하는 일이 설계에서 중요
    - 발견 가능성, 직관성, 실사용성을 갖추는지는 사용자의 경험이 좌우
    - 에이전트는 눈에 보이는 행동유도성이 부족해 사용자가 에이전트 가능, 불가능을 짐작해야함
- 효과적인 에이전트 UX는 선세적으로 커뮤니케이션 해 문제를 해결
    - 입력창 아래에 제안 액션 버튼(주문 추적, 요약 생성)을 제공
        - 시각적 행동을 유도해 사용자가 특정 명령을 기억하거나 가능한 범위를 추측하지 않아도 지원 워크플로로 안내
- 또 다른 유용한 패턴은 확장형 메뉴나 '능력 카드'로 제공 기능을 구조적으로 열거
    - 기존 앱 메뉴에 익숙하게 사이드바를 데이터 조회, 분석, 요약으로 구성
- 개방형 텍스트 입력 에이전트는 대화속에서 스스로 능력을 분명히 알릴수 있어야함
    - 세션 시작 시, '~~ 도와드릴 수 있습니다. 무엇을 해볼까요?' 처럼 주도적으로 소개 필요
    - 능력을 넘어서는 기능을 요청할 때는 거절 대신 대안을 제시
- 한번에 너무 많은 옵션을 제시해 사용자를 압도하지 않는 것이 중요
    - 점진적 공개를 우선하여 초기에는 핵심 기능만 보여줌 -> 추후 고급 기능을 드러냄
- 위 원칙은 다양한 모달리티에 적용
    - 텍스트 기반: 빠른 응답 버튼과 예시 프롬프트 -> 명료성을 높임
    - 그래픽 대시보드: 능력 메뉴와 툴팁 -> 기능 전달
    - 음성 에이전트: 간결한 안내 -> 인지 부하 방지
    - 생성형 UI: 자연어 + 시각적 출력 -> 사용가능한 능력 즉시 시각화 및 실행
- 능력 나열 보단 사용자가 그 능력을 효율적으로 활용하도록 경험을 설계
    - 에이전트의 범위와 한계를 이해하도록 설계 필요
##### 3.4.1 신뢰도와 불확실성 커뮤니케이션
- 에이전트는 종종 결정적 규칙이 아닌 통계 모델 기반한 확률적 환경에서 작동
    - 모든 응답이나 행동이 동일한 신뢰도를 갖는 건 아님
- 불확실성을 효과적으로 전달하는 일은 사용자 신뢰를 쌓고 근거한 결정을 내리도록 돕는데 필수
    - 명시적 진술: '정답일 확률 90%입니다'
    - 시각적 신호: 아이콘, 색상 기반 경고 그래피 인터페이스 신뢰도 미터
    - 행동 조정: 신뢰도가 낮을 때 단정적 권고 대신 제안을 제공
- 사용자와 상호작용의 중요도에 맞게 응답을 구성하는게 중요
    - 불확실성이 높은 사황에서 과도하게 자신감을 보여선 안됨
    - 위험이 낮은 상호작용에서 지나치게 모호하게 말해서도 안됨
##### 3.4.2 사용자 지침과 입력 요청
- 고도화된 에이전트라도 모호하거나 애매한 사용자 입력을 완벽히 해석할 수 없음
    - 위험한 추정 보단 명확한 질문을 하거나 지침을 구할 시점을 알아야함 -> 잠재적 오류를 협업의 기회로 전환
- 모호성을 직면했을 때 초점을 맞춘 유용한 질문을 하도록 설계
- 질문 방식도 중요
    - 질문은 명확하고 공손하며 컨텍스트 인식적이어야 함 -> 반복 지향
    - 왜 명확화가 필요한지 투명하게 설명 필요
    - 한번에 너무 많은 질문 지향
##### 3.4.3 우아한 실패
- 에이전트 시스템이 항상 제대로 작동한다는 법은 없음
    - 실패를 다루는 방식은 성공만큼이나 중요
    - 우아하게 실패해 좌절을 최소화
- 우아한 실패 핵심은 문제를 투명하게 인정하고 유용한 설명을 제공해 실행 가능한 다음 단계 제시
    - '찾으시는 정보를 확인하지 못했습니다. 사람 상담원으로 전달해 드릴까요?'
- 흔한 실패 지점을 예측해 사전에 폴백 메커니즘을 갖춰야함
    - '요청을 이해하는 데 어려움이 있습니다. 표현을 바꿔주시거나 질문을 직업 입력해주시겠어요?'
- 다단계 작업에서는 실패 시, 진행 상태 보존이 동일하게 중요
    - 문제가 해결되면 계속 진행할 수 있도록 진행 사항 유지 -> 반복과 좌절 예방
- 사과와 공감의 언어
    - 문제가 발생하면 인간적이고 배려 있는 방식으로 실패를 인정
        - '죄송합니다. 요청 처리 중 문제가 발생했습니다. 담당자와 연결해 드릴께요'
- 명확한 해결 경로를 제공
    - 단계별 문제 해결 안내, 사람 상당원에게 전달 등
- 가능하다면 실패로부터 학습 필요
    - 실패 지점을 기록하고 반복 이슈를 분석해 유사 실패 빈도를 줄이기
# 3.5 상호작용 설계에서의 신뢰
- 신뢰를 얻는데 오래걸려도 잃기는 쉬움 -> 에이전트도 동일
- 투명성과 예측 가능성은 신뢰를 구축하는데 가장 강력한 도구
    - 투명성
        - 에이전트의 능력과 제약을 명확히 전달
    - 예측 가능성
        - 다양한 상황에서의 일관된 행동을 통해 투명성을 보완
        - 과거 상호 작용을 바탕으로 에이전트 반응을 예측할 수 있어야함
- 에이전트가 협력자로서 거듭나는 신뢰 구축의 두가지 핵심 요소
    - 행동의 예측 가능성
        - 일관되게 행동하고 적절히 응답, 오류를 우아하게 처리하리라 기대
    - 신뢰성
        - 출력의 일관성에서 출발
        - 동일한 조건에서 같은 질문을 하면 같은 응답이 나와야함
        - 일관성 유지할 수 없는 경우(예: 언어 모델의 확률적 출력)
        - 에이전트 엣지 케이스도 신중히 처리 필요
        - 시스템 복원력
            - 오류에서 회복하고 중단 상황에서도 상태 유지
        - 기대를 일관되게 설정하고 충족하는 일

# 2장: 에이전트 시스템 설계

## 2.1 우리의 첫 번째 에이전트 시스템
- 반복적 룰 기반 작업(이커머스 CS의 환불·취소 등)은 에이전트로 자동화하기 좋은 후보다.
- 첫 예제는 LangChain + LangGraph 기반 `cancel_order` 에이전트로, 도구 호출 후 확인 메시지를 보내는 2-패스 LLM 패턴이다.
- 스코핑은 균형이 핵심 — 너무 좁으면 영향력이 작고, 너무 넓으면 엣지 케이스에 잠식되며, 모호하면 성공을 정의할 수 없다.
- 좋은 슬라이스는 구체적 입력 + 구조화된 출력 + 빠른 피드백 루프를 갖추며, 평가는 도구·파라미터·메시지 품질을 측정한다.

## 2.2 에이전트 시스템의 핵심 구성요소
- 에이전트 시스템은 **모델·도구·메모리·오케스트레이션** 네 부품이 유기적으로 결합된 구조다.
- 모델은 두뇌, 도구는 행동, 메모리는 맥락, 오케스트레이션은 흐름 제어를 담당하며 부품 간 상호작용 설계가 핵심이다.

## 2.3 모델 선택
- 모델 선택은 작업 복잡도·모달리티·개방성·비용/지연 등 여러 축에서 지속 재평가할 전략적 선택이다.
- 큰 모델은 개방형·다단계 작업에 강하나 비용·지연 부담이 크고, 작은 모델은 정형 작업에 효율적이다.
- ~14B 모델은 컨슈머급 GPU(RTX 3090)로, 그 이상은 A100 같은 서버급 GPU가 필요하다.
- 성능과 가격은 정비례하지 않으므로 자기 작업으로 직접 비교해야 하며, 추세는 **하이브리드/동적 라우팅**이다.

## 2.4 도구

### 2.4.1 특정 작업을 해결하는 도구 설계
- 도구는 세 종류 — **로컬**(내부 로직), **API 기반**(외부 서비스), **MCP**(컨텍스트 주입).
- MCP는 별도 호출 왕복 없이 동적 컨텍스트를 모델 추론에 직접 넣어 중복 호출을 줄인다.

### 2.4.2 도구 통합과 모듈성
- 각 도구는 자기 완결적 모듈로 설계해 핵심 운영을 흔들지 않고 무중단 확장·교체가 가능해야 한다.

## 2.5 메모리

### 2.5.1 단기 메모리
- 현재 작업·대화 정보를 보관하며, 주로 **롤링 컨텍스트 윈도우**로 구현해 최근 정보만 유지한다.

### 2.5.2 장기 메모리
- 시간에 따른 개선과 개인화를 위해 **DB·지식 그래프·파인튜닝된 모델**로 지식과 경험을 저장한다.

### 2.5.3 메모리 관리 및 검색
- 효과적 메모리는 인덱싱·빠른 검색뿐 아니라 **잊기(forgetting)** 능력도 갖춰야 한다.
- 최근 데이터에 우선순위를 두어 변화하는 맥락을 반영한다.

## 2.6 오케스트레이션
- 오케스트레이션은 도구·스킬 시퀀스를 평가하고 최적 경로를 선택해 다단계 작업을 묶는 로직이다.
- 환경 변화(새 정보, 우선순위 변화 등)에 맞춰 진행을 모니터링하고 필요 시 일시정지·재라우팅한다.
- **점진적 계획 수립**이 일반적 — 몇 단계 실행 후 결과를 재평가하고 남은 흐름을 업데이트한다.

## 2.7 설계 트레이드오프

### 2.7.1 성능: 속도와 정확도의 균형
- 속도와 정확도는 본질적 트레이드오프 — 도메인 우선순위(자율주행 vs 의료 진단)에 따라 선택한다.
- **하이브리드 접근**(빠른 근사 → 추후 정밀)이 추천·진단 시스템에서 흔히 쓰인다.

### 2.7.2 확장성: 엔지니어링적 확장
- 확장의 핵심 자원은 **GPU**이며, 단순 추가가 아닌 **활용도 극대화**가 핵심이다.
- 동적 할당·우선순위 큐·비동기 실행·동적 부하 분산으로 유휴 시간과 병목을 줄인다.
- 수평 확장·하이브리드 클라우드·버스트 스케일링으로 비용 효율을 확보한다.

### 2.7.3 신뢰성: 견고하고 일관된 에이전트
- 결함 허용성과 일관성이 핵심 — 중복성 도입과 다양한 환경에서의 일관된 성능을 보장해야 한다.
- 광범위 테스트(엣지 케이스·적대적 조건 포함) + 모니터링·피드백 루프 조합으로 달성한다.

### 2.7.4 비용: 성능과 지출의 균형
- 개발 비용(인력·인프라)과 운영 비용(GPU·저장·유지보수) 모두를 시스템 목표·수명에 비추어 정당화해야 한다.
- 린 모델·클라우드 자원·오픈소스 활용으로 비용을 최적화한다.

## 2.8 아키텍처 디자인 패턴

### 2.8.1 단일 에이전트 아키텍처
- 한 에이전트가 모든 작업을 단독 처리하며, **단순함**(설계·배포 용이)이 강점이다.
- 잘 정의된 좁은 작업(FAQ 챗봇, 코드 생성 등)에 최적이다.

### 2.8.2 멀티 에이전트 아키텍처: 협업, 병렬성, 조율
- 협업·병렬성·확장성·회복탄력성이 강점이지만, 조율 복잡성과 토큰 소비 증가가 도전 과제다.
- 금융 거래·사이버보안·협력형 AI처럼 복잡·분산·전문화된 환경에 적합하다.

## 2.9 모범 사례

### 2.9.1 점진적 설계
- 완벽 대신 작은 프로토타입을 빠르게 만들어 반복 개선하는 것이 핵심이다.
- **빠른 프로토타입 → 테스트·피드백 → 정제·반복** 사이클을 돈다.

### 2.9.2 평가 전략
- 기능 테스트(정확성·엣지 케이스), 일반화 능력, UX 메트릭(NPS·완료율·신호 분석)을 종합한다.
- Human-in-the-Loop 검증으로 자동 평가를 보정하고, 실세계 시뮬레이션 환경에서 종단간 평가한다.

### 2.9.3 실환경 테스트
- 통제 환경만으로는 부족하며, 실 사용자에게 배포해 예측 불가능한 조건과 부하를 검증해야 한다.
- **단계적 배포 → 모니터링 → 피드백 수집 → 인사이트 기반 반복** 사이클을 돈다.

## 2.10 마무리
- 좋은 에이전트는 설계서 없이 작은 슬라이스부터 시작해 즉시 가치를 만든다.
- 강한 아키텍처·규율 있는 엔지니어링·긴밀한 피드백 루프가 프로토타입을 신뢰할 수 있는 프로덕션 시스템으로 바꾼다.

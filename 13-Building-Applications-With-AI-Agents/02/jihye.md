# Chapter 2. 에이전트 시스템 설계

## 2.1 우리의 첫 번째 에이전트 시스템

- 첫 번째 예제는 **고객 지원 메일을 처리하는 에이전트 시스템**
    - 사용자가 보낸 메시지를 읽고, 내부 도구를 사용해 주문 정보를 조회한 뒤, 상황에 따라 답변을 생성하거나 주문 취소 같은 작업을 수행하는 흐름
    - 핵심은 단순히 LLM에게 “답변해줘”라고 하는 것이 아니라, **LLM이 도구를 호출하고, 그 결과를 다시 판단해서 최종 행동을 결정하게 만드는 구조**

### 전체 흐름

| 단계 | 역할 | 설명 |
| --- | --- | --- |
| 1 | 메시지 입력 | 고객이 보낸 이메일/문의 내용을 받음 |
| 2 | LLM 판단 | 문의 의도를 분석하고 필요한 도구를 결정 |
| 3 | 도구 호출 | 주문 조회, 주문 취소, 환불 처리 등의 함수 실행 |
| 4 | 결과 반영 | 도구 실행 결과를 LLM에게 다시 전달 |
| 5 | 최종 응답 | 고객에게 보낼 답변 생성 |

정리하면, **LLM은 판단을 담당하고 실제 작업은 도구가 담당**

---

## 2.1.1 시스템 프롬프트와 도구 정의

에이전트가 제대로 동작하려면 먼저 **시스템 프롬프트**와 **도구 목록**을 정의해야 한다.

- 시스템 프롬프트
    - 에이전트의 역할, 사용할 수 있는 도구, 응답 방식, 제한 사항을 알려주는 설정

```python
system_prompt = """
당신은 고객 지원 에이전트입니다.
고객 문의를 읽고 필요한 경우 도구를 사용하세요.
	주문 취소가 필요한 경우 cancel_order 도구를 호출하고,
고객에게는 처리 결과를 친절하게 설명하세요.
"""
```

- 도구
    - LLM이 직접 실행하는 것이 아니라, **LLM이 호출 요청을 만들고 프로그램이 실제 함수를 실행**하는 방식

```python
def get_order(order_id: str) -> str:
    """주문 정보를 조회한다."""
    return f"주문 {order_id} 정보를 조회했습니다."

def cancel_order(order_id: str) -> str:
    """주문을 취소한다."""
    return f"주문 {order_id}를 취소했습니다."
```

- 중요한 포인트는 LLM이 스스로 DB를 뒤지거나 API를 직접 실행하는 게 아니라는 점
    - LLM은 “어떤 도구가 필요하다”를 판단하고, 실제 실행은 애플리케이션 코드가 담당

---

## 2.1.2 메시지 구조와 대화 상태

- 에이전트 시스템에서는 대화가 단순 문자열 하나로 끝나지 않고
    - **system, user, assistant, tool** 같은 메시지 역할을 구분해서 관리한다.

| 메시지 타입 | 의미 |
| --- | --- |
| SystemMessage | 에이전트의 역할과 규칙 |
| HumanMessage | 사용자의 입력 |
| AIMessage | LLM의 응답 또는 도구 호출 요청 |
| ToolMessage | 도구 실행 결과 |

대화 상태는 보통 `messages` 리스트로 누적된다.

```python
messages = [
    SystemMessage(content=system_prompt),
    HumanMessage(content="주문 #7893793을 취소해주세요."),
]
```

LLM이 도구 호출을 요청하면, 애플리케이션은 해당 도구를 실행하고 결과를 다시 메시지에 추가한다.

```python
messages.append(ai_message)
messages.append(ToolMessage(content=tool_result, tool_call_id=tool_call_id))
```

결국 에이전트는 **현재 메시지 목록 전체를 보고 다음 행동을 결정**한다.

→ 그래서 메시지 히스토리 관리가 중요함. 대충 하면 맥락 날아감.

---

## 2.1.3 그래프 기반 에이전트 구성

노드는 특정 작업을 수행하고, 엣지는 다음에 어떤 노드로 이동할지 결정한다.

```
사용자 입력
   ↓
에이전트 호출
   ↓
도구 호출 필요 여부 판단
   ├─ 필요함 → 도구 실행 → 다시 에이전트 호출
   └─ 필요 없음 → 최종 응답
```

- 그래프 기반으로 만들면 좋은 점
    - 실제 서비스에서는 성공/실패/재시도/검증/승인 같은 분기가 계속 생기기 때문임.

| 장점 | 설명 |
| --- | --- |
| 흐름이 명확함 | 어떤 단계에서 무엇을 하는지 분리됨 |
| 조건 분기가 쉬움 | 도구 호출 여부에 따라 다음 노드를 다르게 선택 가능 |
| 확장성이 좋음 | 도구, 검증, 재시도 노드를 추가하기 쉬움 |
| 디버깅이 편함 | 어느 노드에서 문제가 생겼는지 추적 가능 |

---

## 2.1.4 조건부 엣지

- **현재 상태를 보고 다음 노드를 결정하는 함수**다.
    - 예를 들어 LLM 응답에 tool call이 있으면 도구 실행 노드로 보내고, 없으면 종료한다.

```python
def should_continue(state):
    messages = state["messages"]
    last_message = messages[-1]

    if last_message.tool_calls:
        return "tools"

    return "end"
```

- 이런 식으로 조건을 나누면 에이전트가 다음 행동을 스스로 선택하는 것처럼 동작

| 조건 | 다음 단계 |
| --- | --- |
| tool_calls 있음 | 도구 실행 노드로 이동 |
| tool_calls 없음 | 최종 응답 후 종료 |
| 오류 발생 | 재시도 또는 실패 처리 노드로 이동 가능 |
- 즉, 조건부 엣지는 에이전트 시스템의 **흐름 제어 장치**

---

## 2.1.5 도구 실행 노드

도구 실행 노드는 LLM이 요청한 tool call을 실제 함수 호출로 바꾸는 부분

```python
def tool_node(state):
    messages = state["messages"]
    last_message = messages[-1]

    tool_messages = []

    for tool_call in last_message.tool_calls:
        tool_name = tool_call["name"]
        tool_args = tool_call["args"]

        result = tools[tool_name].invoke(tool_args)

        tool_messages.append(
            ToolMessage(
                content=str(result),
                tool_call_id=tool_call["id"],
            )
        )

    return {"messages": tool_messages}
```

- 핵심은 LLM은 호출 명세를 만들고, 런타임이 안전하게 실행함.

| 구성 요소 | 설명 |
| --- | --- |
| `tool_call["name"]` | 호출할 도구 이름 |
| `tool_call["args"]` | 도구에 전달할 인자 |
| `tools[tool_name].invoke()` | 실제 도구 실행 |
| `ToolMessage` | 실행 결과를 LLM에게 돌려주는 메시지 |

## 2.2 에이전트 시스템의 핵심 구성요소

에이전트 시스템은 보통 다음 구성요소로 이루어진다.

| 구성요소 | 역할 |
| --- | --- |
| 모델 | 판단, 추론, 응답 생성 담당 |
| 도구 | 외부 시스템/API/DB 작업 수행 |
| 메모리 | 이전 대화, 사용자 정보, 작업 상태 저장 |
| 지식 베이스 | 검색 가능한 문서, FAQ, 매뉴얼 등 |
| 오케스트레이션 | 전체 흐름 제어, 조건 분기, 반복 수행 |
- 에이전트는 모델 하나만으로 완성되지 않고
    - 모델은 “머리”에 가깝고, 도구와 메모리, 지식 베이스, 오케스트레이션이 붙어야 실제 시스템이 됨.

### 에이전트 시스템 구조

```
사용자
  ↓
에이전트 시스템
  ├─ 모델
  ├─ 도구
  ├─ 메모리
  └─ 지식 베이스
```

- 실무에서는 이 구성요소를 분리해서 설계해야 함
    - 안 그러면 프롬프트 하나에 모든 걸 때려 넣는 구조가 되고, 유지보수가 어려워짐.

---

## 2.3 모델 선택

- 에이전트 시스템의 성능과 비용을 크게 좌우한다.
    - 무조건 가장 큰 모델을 쓰는 것이 정답은 아니다.

### 모델 선택 기준

| 기준 | 설명 |
| --- | --- |
| 성능 | 추론, 도구 사용, 긴 문맥 처리 능력 |
| 비용 | 호출당 비용, 토큰 비용 |
| 속도 | 응답 지연 시간 |
| 컨텍스트 길이 | 긴 문서나 긴 대화를 처리할 수 있는지 |
| 도구 호출 능력 | function calling/tool calling 품질 |
| 배포 환경 | 클라우드 API, 온프레미스, 로컬 실행 가능 여부 |

### 모델 유형별 특징

| 모델 유형 | 장점 | 단점 | 적합한 경우 |
| --- | --- | --- | --- |
| 대형 상용 모델 | 성능 좋고 도구 호출 안정적 | 비용 높음, 외부 API 의존 | 복잡한 에이전트, 고객 응대, 고품질 답변 |
| 오픈소스 모델 | 커스터마이징 가능, 내부 배포 가능 | 운영 난이도 높음 | 보안 요구, 비용 최적화, 자체 인프라 |
| 소형 모델 | 빠르고 저렴함 | 복잡한 추론 약함 | 단순 분류, 라우팅, 전처리 |
| 멀티모달 모델 | 이미지/문서 이해 가능 | 비용과 지연 증가 | 문서 분석, 이미지 기반 작업 |

정리하면, **복잡한 추론은 큰 모델**, **단순 분류나 라우팅은 작은 모델**을 쓰는 식으로 나누는 게 좋다.

→ 한 모델로 다 하려 하면 비용 터질 수 있음.

## 2.4 도구

- 에이전트가 외부 세계와 상호작용할 수 있게 해주는 인터페이스
    - 예를 들어 검색, 주문 조회, 주문 취소, 메일 발송, 데이터베이스 조회, 계산기 같은 기능이 도구가 된다.

### 도구가 필요한 이유

- LLM은 자체적으로 최신 정보나 내부 시스템 상태를 알 수 없으므로 실제 데이터를 확인하거나 작업을 수행하려면 도구가 필요하다.

| 도구 예시 | 역할 |
| --- | --- |
| 검색 도구 | 문서, 웹, DB 검색 |
| 주문 조회 도구 | 고객 주문 상태 확인 |
| 취소/환불 도구 | 실제 비즈니스 작업 수행 |
| 계산 도구 | 정확한 수치 계산 |
| 메일 도구 | 이메일 작성/전송 |
| 코드 실행 도구 | 파이썬 코드 실행 및 결과 확인 |
- 도구는 강력하지만 위험할 수도 있는데
    - 특히 결제, 주문 취소, 데이터 삭제 같은 도구는 반드시 제한과 검증이 필요

---

## 2.4.1 특정 작업을 실행하는 도구 설계

- 도구는 보통 **명확한 이름, 설명, 입력 스키마**를 가져야 한다.
    - 그래야 LLM이 언제 어떤 도구를 써야 하는지 판단하기 쉽다.

```python
@tool
def get_order(order_id: str) -> str:
    """주문 ID를 받아 주문 정보를 조회한다."""
    return order_db.get(order_id, "주문을 찾을 수 없습니다.")

@tool
def cancel_order(order_id: str) -> str:
    """주문 ID를 받아 해당 주문을 취소한다."""
    if order_id not in order_db:
        return "주문을 찾을 수 없습니다."

    order_db[order_id]["status"] = "cancelled"
    return f"주문 {order_id}가 취소되었습니다."
```

- 도구 설명은 대충 쓰면 안 된다.
    - LLM은 설명을 보고 도구를 고르기 때문에, 설명이 애매하면 잘못된 도구를 호출할 수 있음.

### 좋은 도구 설계 기준

| 기준 | 설명 |
| --- | --- |
| 이름이 명확해야 함 | `do_stuff` 같은 이름은 피함 |
| 입력값이 구체적이어야 함 | `order_id`, `customer_id` 등 명확히 정의 |
| 반환값이 예측 가능해야 함 | 성공/실패 메시지 형식이 일정해야 함 |
| 부작용이 명확해야 함 | 취소, 삭제, 결제 등은 특히 주의 |
| 권한 검사가 필요함 | 사용자가 실행 가능한 작업인지 확인 |

---

## 2.4.2 도구 통합과 관리

- 도구가 많아지면 단순히 리스트에 넣는 것만으로는 관리가 어려움
    - 도구를 그룹화하고, 어떤 에이전트가 어떤 도구를 사용할 수 있는지 제한해야 함

```python
tools = [get_order, cancel_order]
llm_with_tools = llm.bind_tools(tools)
```

- 도구 통합 시 중요한 포인트

| 항목 | 설명 |
| --- | --- |
| 도구 등록 | LLM이 사용할 수 있는 도구 목록을 제공 |
| 도구 라우팅 | 요청에 맞는 도구를 선택 |
| 권한 제어 | 특정 사용자/상황에서만 도구 실행 허용 |
| 실행 로그 | 어떤 도구가 어떤 입력으로 실행됐는지 기록 |
| 예외 처리 | 실패 시 재시도하거나 사용자에게 안내 |
- 실무에서는 도구 실행 로그가 매우 중요
    - 나중에 “왜 주문이 취소됐지?” 같은 문제가 생기면 tool call 로그를 봐야 함.

---

## 2.5 메모리

- 에이전트가 이전 대화나 사용자 정보를 기억하게 해주는 구성요소
    - 메모리가 없으면 매 요청이 독립적으로 처리되어 맥락을 유지하기 어려움

### 메모리의 종류

| 종류 | 설명 | 예시 |
| --- | --- | --- |
| 단기 메모리 | 현재 대화 세션 안에서 유지되는 정보 | 직전 메시지, 현재 작업 상태 |
| 장기 메모리 | 여러 세션에 걸쳐 유지되는 정보 | 사용자 선호, 과거 이력 |
| 작업 메모리 | 현재 작업을 수행하기 위한 임시 상태 | 주문 ID, 처리 단계 |
| 검색 기반 메모리 | 필요할 때 검색해서 가져오는 기억 | 과거 상담 내역, 문서 |

메모리는 많을수록 좋은 게 아니다.

불필요한 기억은 오히려 혼란을 만들고, 개인정보/보안 문제도 생길 수 있음.

---

## 2.5.1 단기 메모리

단기 메모리는 현재 대화 안에서 맥락을 유지하기 위한 메모리다.

보통 메시지 리스트나 상태 객체에 저장한다.

```python
state = {
    "messages": messages,
    "order_id": "7893793",
    "current_step": "checking_order",
}
```

단기 메모리는 다음과 같은 상황에서 유용하다.

| 상황 | 예시 |
| --- | --- |
| 앞선 질문 참조 | “그 주문 취소해줘”에서 `그 주문`이 무엇인지 기억 |
| 작업 단계 유지 | 주문 조회 후 취소 단계로 이동 |
| 사용자 의도 보존 | 고객이 원하는 최종 목표 기억 |

하지만 대화가 길어지면 컨텍스트가 커지고 비용도 늘어난다.

그래서 오래된 메시지는 요약하거나 필요한 정보만 남기는 전략이 필요함.

---

## 2.5.2 장기 메모리

장기 메모리는 세션이 끝나도 유지되는 정보다.

사용자의 선호, 과거 행동, 반복되는 요청 패턴 등을 저장할 수 있다.

예를 들어 고객 지원 에이전트라면 다음과 같은 정보를 기억할 수 있다.

| 장기 메모리 예시 | 활용 |
| --- | --- |
| 사용자가 선호하는 언어 | 한국어/영어 응답 선택 |
| 자주 문의하는 제품 | 답변 개인화 |
| 이전 환불 이력 | 정책 판단에 참고 |
| 고객 등급 | 응대 우선순위 결정 |

다만 장기 메모리는 조심해야 한다.

개인정보, 민감정보, 잘못 저장된 정보는 문제를 만들 수 있음.

따라서 저장 기준, 삭제 기준, 사용자 동의가 필요하다.

---

## 2.5.3 메모리 관리 및 검색

메모리는 저장만 해서는 의미가 없다.

필요한 순간에 관련 정보를 잘 찾아와야 한다.

일반적으로는 벡터 검색, 키워드 검색, 메타데이터 필터링을 함께 사용한다.

| 방식 | 설명 |
| --- | --- |
| 키워드 검색 | 정확한 단어 기반 검색 |
| 벡터 검색 | 의미적으로 비슷한 과거 정보 검색 |
| 메타데이터 필터 | 사용자 ID, 날짜, 카테고리 기준 필터링 |
| 요약 저장 | 긴 대화를 짧게 요약해서 저장 |

메모리 검색은 RAG와 비슷하지만 목적이 약간 다르다.

RAG는 외부 지식 검색에 가깝고, 메모리는 사용자/작업 맥락을 되살리는 데 가깝다.

## 2.6 오케스트레이션

- 에이전트 시스템의 전체 실행 흐름을 제어
    - 어떤 노드를 실행할지, 언제 반복할지, 언제 종료할지 결정

### 오케스트레이션이 필요한 이유

| 이유 | 설명 |
| --- | --- |
| 복잡한 흐름 관리 | 모델 호출, 도구 실행, 검증, 재시도 등을 연결 |
| 조건 분기 처리 | 도구 필요 여부, 오류 여부에 따라 경로 변경 |
| 반복 제어 | 에이전트가 여러 번 도구를 호출할 수 있음 |
| 안전장치 적용 | 무한 루프 방지, 승인 단계 추가 |
| 관찰 가능성 확보 | 실행 흐름을 로그로 남기고 디버깅 가능 |
- 에이전트는 한 번 호출하고 끝나는 구조가 아니라, 상황에 따라 여러 번 판단하고 도구를 호출한다.
    - 오케스트레이션이 없으면 흐름이 꼬이기 쉬움.

---

## 2.7 설계 트레이드오프

- 성능을 높이면 비용이 증가하고, 자유도를 높이면 통제가 어려워질 수 있음

### 주요 트레이드오프

| 선택지 | 장점 | 단점 |
| --- | --- | --- |
| 큰 모델 사용 | 추론 성능 좋음 | 비용과 지연 증가 |
| 작은 모델 사용 | 빠르고 저렴함 | 복잡한 작업에 약함 |
| 도구 많이 제공 | 다양한 작업 가능 | 잘못된 도구 선택 가능성 증가 |
| 도구 제한 | 안정적이고 통제 쉬움 | 해결 가능한 작업 범위 축소 |
| 장기 메모리 사용 | 개인화 가능 | 개인정보/오염 문제 |
| 매번 검색 | 최신성, 정확성 증가 | 속도와 비용 증가 |
- 에이전트는 “많이 붙이면 좋다”가 아니라 필요한 기능만 붙이고, 위험한 행동은 제한하는 쪽이 실무적으로 안전함.

---

## 2.7.1 성능-속도와 정확도의 균형

- 빠른 응답이 중요한 시스템과 정확한 답변이 중요한 시스템은 설계가 달라야 한다.
    - 예를 들어 고객 상담에서는 어느 정도 빠른 응답이 중요하지만, 환불/결제/계약 같은 작업은 정확도가 더 중요하다.

| 상황 | 우선순위 |
| --- | --- |
| FAQ 답변 | 속도, 비용 |
| 주문 취소 | 정확도, 권한 검증 |
| 법률/의료/금융 답변 | 정확도, 근거, 안전성 |
| 내부 문서 검색 | 검색 품질, 출처 |
| 단순 분류 | 속도, 비용 |
- 정리하면, **모든 요청을 같은 방식으로 처리하면 비효율적**이다.
    - 요청 난이도에 따라 모델과 도구, 검증 강도를 다르게 가져가는 게 좋음

---

## 2.7.2 확장성: 에이전트 시스템 엔지니어링 관점

- 서비스 규모가 커지면 에이전트 시스템도 확장성을 고려해야 한다.
    - 단순 데모에서는 잘 되던 구조가 실제 운영에서는 병목이 될 수 있다.

### 확장성에서 고려할 점

| 항목 | 설명 |
| --- | --- |
| 동시 요청 처리 | 여러 사용자의 요청을 동시에 처리 가능해야 함 |
| 모델 호출 비용 | 호출량 증가에 따른 비용 관리 필요 |
| 캐싱 | 반복되는 검색/응답은 캐싱 가능 |
| 비동기 처리 | 오래 걸리는 작업은 비동기로 분리 |
| 큐 기반 처리 | 작업량이 많을 때 안정적으로 처리 |
| 모니터링 | 지연, 실패율, 도구 호출 수 추적 |
- 운영 환경에서는 “답변이 잘 나옴”만으로 부족하다.
    - 얼마나 안정적으로, 얼마나 싸게, 얼마나 빠르게 처리할 수 있는지가 중요함.

---

## 2.7.3 신뢰성: 고견고한 에이전트

- 에이전트가 잘못된 행동을 하지 않도록 만드는 설계
    - 특히 실제 비즈니스 작업을 수행하는 에이전트에서는 매우 중요

### 신뢰성을 높이는 방법

| 방법 | 설명 |
| --- | --- |
| 입력 검증 | 사용자 요청이 유효한지 확인 |
| 도구 권한 제한 | 위험한 도구는 특정 조건에서만 실행 |
| 실행 전 확인 | 주문 취소/환불 전 사용자 확인 받기 |
| 결과 검증 | 도구 실행 결과가 정상인지 확인 |
| 재시도 정책 | 일시적 오류는 재시도 |
| 휴먼 인 더 루프 | 중요한 결정은 사람이 승인 |
| 로그 기록 | 모든 판단과 도구 호출 기록 |
- 에이전트는 확률적으로 동작하는 모델을 포함하기 때문에, 완전히 믿고 맡기면 위험
    - 특히 돈, 개인정보, 삭제, 외부 전송이 걸린 작업은 반드시 안전장치 필요

---

## 2.7.4 비용-성능의 균형

- 에이전트 시스템은 비용 관리가 중요하다.
    - LLM 호출, 긴 컨텍스트, 도구 호출, 검색, 재시도 모두 비용으로 이어진다.

### 비용을 줄이는 전략

| 전략 | 설명 |
| --- | --- |
| 모델 라우팅 | 쉬운 요청은 작은 모델, 어려운 요청은 큰 모델 사용 |
| 캐싱 | 반복 질문/검색 결과 재사용 |
| 컨텍스트 압축 | 오래된 대화는 요약해서 유지 |
| 도구 호출 최소화 | 불필요한 API 호출 방지 |
| 배치 처리 | 여러 작업을 묶어서 처리 |
| 사전 필터링 | LLM 호출 전에 규칙 기반으로 단순 요청 처리 |
- 가장 흔한 실수는 모든 요청을 고성능 모델로 보내는 것
    - 초기에는 편하지만 사용량이 늘면 비용이 급격히 증가

## 2.8 이커머스 디자인 패턴

- 이커머스 환경에서는 에이전트가 상품 탐색, 주문 조회, 취소, 환불, 고객 응대 등을 수행할 수 있는데 실제 거래와 연결되기 때문에 안전성과 상태 관리가 중요하다.

### 이커머스 에이전트에서 자주 나오는 패턴

| 패턴 | 설명 |
| --- | --- |
| 고객 문의 응대 | 배송, 주문, 반품 문의 처리 |
| 주문 상태 조회 | 주문 ID 기반 상태 확인 |
| 취소/환불 처리 | 정책 확인 후 작업 실행 |
| 상품 추천 | 사용자 조건에 맞는 상품 검색 |
| FAQ + RAG | 정책/매뉴얼 검색 후 답변 |
| 상담원 연결 | 해결 불가능하거나 위험한 요청은 사람에게 전달 |

이커머스 에이전트는 “말 잘하는 챗봇”보다 **정확한 상태 확인과 안전한 작업 실행**이 더 중요하다.

---

## 2.8.1 단일 에이전트 아키텍처

- 단일 에이전트 아키텍처는 하나의 에이전트가 여러 도구를 가지고 대부분의 작업을 처리하는 구조다.

```
사용자 요청
  ↓
단일 에이전트
  ├─ 주문 조회 도구
  ├─ 취소 도구
  ├─ 환불 도구
  └─ FAQ 검색 도구
```

### 장점과 단점

| 구분 | 내용 |
| --- | --- |
| 장점 | 구조가 단순하고 구현이 빠름 |
| 장점 | 초기 데모나 작은 서비스에 적합 |
| 단점 | 도구가 많아지면 선택 오류 가능성 증가 |
| 단점 | 프롬프트가 길어지고 유지보수 어려움 |
| 단점 | 역할이 섞여 디버깅이 어려울 수 있음 |
- 단일 에이전트는 시작하기 좋지만, 시스템이 커지면 한계가 온다.
    - 처음엔 이걸로 가도 되지만, 운영 규모가 커지면 분리 필요함.

---

## 2.8.2 멀티 에이전트 아키텍처: 협업 방식으로 확장

- 멀티 에이전트 구조는 역할별로 에이전트를 나누고, 서로 협업하게 만드는 방식.

```
사용자 요청
  ↓
라우터 에이전트
  ├─ 주문 에이전트
  ├─ 환불 에이전트
  ├─ 상품 추천 에이전트
  └─ 고객 응대 에이전트
```

### 멀티 에이전트의 장점

| 장점 | 설명 |
| --- | --- |
| 역할 분리 | 각 에이전트가 특정 업무에 집중 |
| 프롬프트 단순화 | 에이전트별 지시문이 짧아짐 |
| 도구 제한 가능 | 필요한 도구만 제공 가능 |
| 확장성 증가 | 새로운 역할을 추가하기 쉬움 |
| 안전성 향상 | 위험한 작업을 별도 에이전트로 제한 가능 |

### 멀티 에이전트의 단점

| 단점 | 설명 |
| --- | --- |
| 구조 복잡도 증가 | 라우팅, 상태 공유, 실패 처리 필요 |
| 비용 증가 가능 | 여러 모델 호출이 발생할 수 있음 |
| 디버깅 어려움 | 어느 에이전트가 문제인지 추적 필요 |
| 일관성 관리 필요 | 에이전트 간 답변 톤과 정책 일치 필요 |
- 멀티 에이전트는 멋있어 보이지만 무조건 정답은 아니다.
    - 작은 시스템에 과하게 넣으면 오히려 복잡해짐.

---

## 2.8.3 멀티 에이전트 아키텍처 설계 포인트

- 멀티 에이전트를 설계할 때는 다음을 정해야 한다.

| 설계 요소 | 질문 |
| --- | --- |
| 라우팅 기준 | 어떤 요청을 어떤 에이전트에게 보낼 것인가? |
| 상태 공유 | 에이전트 간 어떤 정보를 공유할 것인가? |
| 권한 관리 | 어떤 에이전트가 어떤 도구를 사용할 수 있는가? |
| 종료 조건 | 언제 작업을 끝낼 것인가? |
| 실패 처리 | 특정 에이전트가 실패하면 어떻게 할 것인가? |
| 인간 개입 | 언제 상담원/운영자가 개입할 것인가? |
- 멀티 에이전트에서 가장 중요한 건 “역할 경계”다.
    - 역할이 겹치면 서로 같은 일을 하거나, 책임이 불명확해짐.

## 2.9 마법 사례

- 사용자 입장에서는 에이전트가 알아서 판단하고 처리하는 것처럼 보이지만, 내부적으로는 **명확한 흐름, 도구 호출, 상태 관리, 평가 전략**이 필요

| 관점 | 설명 |
| --- | --- |
| 사용자 경험 | 사용자는 자연어로 요청하고, 에이전트가 알아서 처리하는 것처럼 느껴야 함 |
| 시스템 설계 | 실제로는 프롬프트, 도구, 메모리, 라우팅, 검증 로직이 필요함 |
| 안정성 | 모든 요청을 자동 처리하면 위험하므로 승인/검증/제한이 필요함 |
| 평가 | 잘 동작하는지 확인하려면 테스트 케이스와 평가 기준이 있어야 함 |

즉, 좋은 에이전트는 “마법”이 아니라 **마법처럼 보이도록 잘 설계된 시스템**

---

## 2.9.1 전면 사례

- 사용자가 에이전트 시스템을 실제로 마주하는 화면이나 흐름을 의미한다.
    - 예를 들어 고객 지원 에이전트라면 사용자는 단순히 문의를 입력하지만, 내부적으로는 여러 단계가 진행된다.

```
사용자 문의
  ↓
에이전트가 의도 파악
  ↓
필요한 정보 조회
  ↓
도구 실행 또는 답변 생성
  ↓
사용자에게 결과 제공
```

### 전면 사례에서 중요한 점

| 항목 | 설명 |
| --- | --- |
| 명확한 입력 | 사용자가 무엇을 요청했는지 이해 가능해야 함 |
| 적절한 응답 | 답변, 작업 결과, 추가 질문 중 무엇이 필요한지 판단해야 함 |
| 신뢰성 | 실제 작업 결과와 답변 내용이 일치해야 함 |
| 투명성 | 필요한 경우 어떤 작업을 수행했는지 사용자에게 알려야 함 |
| 실패 처리 | 정보 부족, 권한 부족, 도구 실패 시 자연스럽게 안내해야 함 |
- 중요한 건 “에이전트가 똑똑해 보이는 것”보다 **사용자가 안심하고 쓸 수 있는 흐름**을 만드는 것임.
    - 특히 주문 취소, 환불, 계정 변경처럼 실제 영향이 있는 작업은 바로 실행하지 말고 확인 절차를 두는 게 좋음.

### 사용자 중심 흐름 예시

| 상황 | 에이전트 동작 |
| --- | --- |
| 주문 상태를 알고 싶음 | 주문 조회 도구 호출 후 상태 안내 |
| 주문을 취소하고 싶음 | 주문 상태 확인 → 취소 가능 여부 판단 → 사용자 확인 후 취소 |
| 환불을 요청함 | 정책 확인 → 주문 정보 확인 → 환불 가능 여부 안내 |
| 애매한 요청 | 추가 질문으로 의도 명확화 |
| 위험한 요청 | 자동 처리하지 않고 사람에게 넘김 |
- 이런 구조가 있어야 실제 서비스에서 사고가 덜 남.
    - 그냥 LLM에게 “알아서 해”라고 하면 운영에서 터질 수 있음.

## 2.9.2 평가 전략

- 에이전트 시스템은 단순한 챗봇보다 평가가 어려움
    - 답변 텍스트만 보는 게 아니라, **올바른 도구를 선택했는지, 인자를 정확히 넣었는지, 최종 행동이 안전했는지**까지 봐야 한다.

### 평가 대상

| 평가 대상 | 확인할 내용 |
| --- | --- |
| 의도 분류 | 사용자의 요청을 제대로 이해했는가 |
| 도구 선택 | 필요한 도구를 정확히 골랐는가 |
| 인자 추출 | 주문 ID, 사용자 ID, 날짜 같은 값을 정확히 추출했는가 |
| 도구 실행 흐름 | 호출 순서가 맞고 누락이 없는가 |
| 최종 응답 | 도구 결과를 바탕으로 정확히 답했는가 |
| 안전성 | 위험한 작업에 확인 절차를 거쳤는가 |
| 실패 대응 | 실패/정보 부족 상황을 적절히 처리했는가 |

### 평가 방식

| 방식 | 설명 |
| --- | --- |
| 정적 테스트 | 미리 만든 입력/정답 세트로 동작 확인 |
| 회귀 테스트 | 수정 후 기존 기능이 깨지지 않았는지 확인 |
| 시나리오 테스트 | 실제 사용자 흐름처럼 여러 턴을 이어서 테스트 |
| 로그 기반 평가 | 실제 실행 로그를 보고 실패 패턴 분석 |
| 사람 검토 | 중요한 케이스는 사람이 직접 판단 |
| LLM-as-judge | 보조 평가자로 LLM을 사용하되 맹신하지 않음 |
- 평가는 “답변이 그럴듯한가”에서 끝나면 안 됨.
    - **정확한 도구 호출 + 안전한 실행 + 사용자에게 맞는 응답**까지 같이 봐야 함.

### 평가 체크리스트

| 체크포인트 | 질문 |
| --- | --- |
| 목적 달성 | 사용자의 원래 요청을 해결했는가 |
| 불필요한 작업 방지 | 하지 않아도 되는 도구 호출을 하지 않았는가 |
| 데이터 정확성 | 조회한 정보와 답변이 일치하는가 |
| 권한/정책 준수 | 허용되지 않은 작업을 하지 않았는가 |
| 설명 가능성 | 왜 그런 응답을 했는지 추적 가능한가 |
| 재현성 | 같은 입력에서 비슷한 품질의 결과가 나오는가 |
- 실무에서는 평가셋이 중요
    - 에이전트는 프롬프트 조금만 바꿔도 행동이 달라질 수 있어서, 회귀 테스트 없으면 품질 관리 어려움.

## 2.9.3 실험의 힌트

- 에이전트 시스템은 처음부터 완벽하게 설계하기 어렵다.
    - 그래서 작은 실험을 반복하면서 어떤 구조가 잘 맞는지 확인해야 한다.

### 실험할 때 볼 것

| 실험 항목 | 확인할 내용 |
| --- | --- |
| 프롬프트 변경 | 역할 설명, 제약 조건, 출력 형식 변경에 따른 차이 |
| 도구 설명 변경 | 도구 설명이 명확할수록 선택 정확도가 올라가는지 |
| 도구 분리 | 하나의 큰 도구보다 작은 도구 여러 개가 나은지 |
| 메모리 사용 | 대화 기록이나 사용자 정보가 성능에 도움 되는지 |
| 라우팅 방식 | 단일 에이전트와 멀티 에이전트 중 어느 쪽이 안정적인지 |
| 모델 변경 | 작은 모델과 큰 모델의 비용/성능 차이 |
| 재시도 전략 | 실패 시 재시도, 재질문, 사람 개입 중 무엇이 적절한지 |

### 실험 설계 포인트

| 포인트 | 설명 |
| --- | --- |
| 한 번에 하나씩 바꾸기 | 여러 요소를 동시에 바꾸면 원인 파악이 어려움 |
| 로그 남기기 | 입력, 모델 응답, 도구 호출, 최종 결과를 저장해야 함 |
| 실패 케이스 모으기 | 성공 사례보다 실패 사례가 개선에 더 중요함 |
| 기준선 만들기 | 비교할 기본 버전이 있어야 개선 여부를 알 수 있음 |
| 운영 케이스 반영 | 예쁜 데모보다 실제 사용자 요청을 기준으로 실험해야 함 |
- 에이전트 실험은 감으로 하면 안 됨.
    - 프롬프트 바꿨더니 좋아진 것 같다는 느낌만으로는 부족하고, **테스트셋과 로그로 확인해야 함**.

### 자주 나오는 실험 질문

```
이 요청은 단일 에이전트가 처리하는 게 나은가?
아니면 라우터가 전문 에이전트에게 넘기는 게 나은가?

도구 설명을 더 자세히 쓰면 tool calling 정확도가 올라가는가?

사용자 확인을 어느 시점에 받아야 안전한가?

메모리를 넣었을 때 답변 품질이 좋아지는가?
아니면 오히려 오래된 맥락 때문에 혼동이 생기는가?
```

이 질문들을 하나씩 실험하면서 구조를 잡아가야 함.

## 2.10 마무리

- 핵심은 LLM 자체보다 **LLM을 둘러싼 실행 구조**다.

### 2장의 핵심 정리

| 주제 | 핵심 내용 |
| --- | --- |
| 에이전트 시스템 | 모델, 도구, 메모리, 오케스트레이션이 결합된 실행 시스템 |
| 도구 | LLM이 외부 세계와 연결되는 통로 |
| 메모리 | 이전 대화와 사용자 정보를 유지하는 장치 |
| 오케스트레이션 | 흐름 제어, 조건 분기, 반복, 종료를 관리 |
| 단일 에이전트 | 단순하고 빠르게 시작 가능하지만 커지면 복잡해짐 |
| 멀티 에이전트 | 역할 분리가 가능하지만 라우팅과 상태 관리가 어려움 |
| 평가 | 답변뿐 아니라 도구 호출, 안전성, 재현성까지 봐야 함 |
- 좋은 에이전트 시스템은 단순히 모델을 호출하는 코드가 아니다.
    - **입력 → 판단 → 도구 실행 → 결과 반영 → 응답 → 평가**까지 연결된 하나의 시스템임.

```
사용자 입력
  ↓
모델 판단
  ↓
도구 / 메모리 / 지식베이스 활용
  ↓
결과 검증
  ↓
최종 응답 또는 작업 실행
  ↓
로그와 평가로 개선
```

- 결국 운영 가능한 에이전트를 만들려면 아래가 필요

| 필요 요소 | 이유 |
| --- | --- |
| 명확한 역할 정의 | 에이전트가 무엇을 해야 하는지 알아야 함 |
| 제한된 도구 권한 | 아무 도구나 쓰면 위험함 |
| 상태 관리 | 여러 턴의 작업을 이어가야 함 |
| 안전장치 | 실제 변경 작업에는 확인과 승인 필요 |
| 관찰 가능성 | 로그가 있어야 문제를 추적할 수 있음 |
| 평가 체계 | 품질이 좋아졌는지 나빠졌는지 판단해야 함 |
- 한 줄로 정리하면, **에이전트는 모델 하나가 아니라 모델을 안전하게 실행시키는 제품 구조**.
    - 데모는 금방 만들 수 있지만, 서비스로 쓰려면 설계·평가·운영이 같이 들어가야 함.

# 2. 에이전트 시스템 설계
- 빠르게 에이전트 시스템을 설계하고 실행하도록 실용적인 내용 소개
# 2.1 우리의 첫 번째 에이전트 시스템
- 반복적인 프로세스는 자동화에 매우 적합하지만 그 범위를 명확하게 해야함
    - 사람이 키보드를 치고 버튼을 클릭하는 것처럼 특정 규칙과 가이드라인을 따르는 작업 ->  파운데이션 모델을 기반으로 시스템을 설계하는 정도로 자동화 가능
    - 만들 에이전트는 고객 메시지와 주문 기록을 받아 어떤 도구를 호출할지 결정
        - 올바른 파라미터로 해당 도구를 실행한 다음 간단한 확인 메시지를 보냄
- 에이전트의 작업 범위를 설정할 때는 늘 균형에 신경써야함
    - 너무 좁히면('취소만 처리') 자주 생기는 다른 요청을 놓쳐 큰 효과를 보지 못함
    - 너무 넒히면('모든 지원 문의 자동화') 수많은 엣지케이스를 대응하도록 준비해야 해서 작업 기간이 오래 걸림
- 운영 환경에서는 에이전트가 그저 실행만 하는 정도로 충분하지 않음
    - 성능, 장단점, 실패 지점을 알야아함
    - 다음 사항을 위주로 평가
        - 올바른 도구를 호출했는가?
        - 올바른 파라미터를 전달했는가?
        - 고객에게 명확하고 정확한 확인 메시지를 보냈는가?
    - 실제 평가는 더 깊이 들어감 -> 테스트 하지 않은 에이전트는 신뢰할 수 없음
        - 도구 정밀도, 파라미터 정확도, 전체 작업 성공률을 측정하여 엣지 케이스를 잡아냄

# 2.2 에이전트 시스템의 핵심 구성 요소
- 효과적인 에이전트 기반 시스템을 설계하려면 핵심 구성 요소를 깊이 이해 필요
    - 구성요소는 에이전트의 능력, 효율, 적응성을 결정
    - 적절한 모델 선택부터 도구, 메모리, 계획 능력 부여 등 많은 요소가 함께 작동 필요

# 2.3 모델 선택
- 시스템 중심에는 모델이 존재
    - 에이전트의 의사결정, 상호작용, 학습 역량을 결정
    - 에이전트 시스템 구축은 모델을 선택하는데서 시작
- 모델 선택은 작업의 복잡성, 입력 데이터의 특성, 인프라 제약, 일반석-속도-정밀성의 트레이드 오프에 달려 있음
    - 작업 복잡도 평가
        - 대형 파운데이션 모델(GPT, 클로드): 컨텍스트 이해, 유연한 추론, 창의적 생성이 필요한 에이전트 적합
            - 일반화가 뛰어나고, 모호성, 문맥적 뉘앙스, 다단계 작업에 강함
            - 높은 연산 자원, 클라우드 인프라, 더 큰 지연 요구
        - 소형 파운데이션 모델(ModernBERT, Phi-4): 정의가 명확한 반복 작업에 적합
            - 로컬 하드웨어에서 실행, 응답이 빠르며, 배포 및 운영 비용이 낮음
            - 고객 지원, 정보 검색, 데이터 라벨링처럼 구조화된 환경에서 유용
    - 모달리티(modality)가 중요
        - 텍스트뿐만 아니라 이미지, 오디오, 구조화 데이터까지 처리
        - 멀티모달 모델: 의료, 로보틱스, 고객지원처럼 여러 형태의 정보를 통합한 의사 결정이 필요한 도메인에 적합
        - 텍스트 중심 모델: 텍스트 중심 과제(단순성과 추론 속도)에 적합
    - 개방성과 커스터마이징 가능성도 중요
        - 오픈 소스 모델(라마, 딥시크)은 투명하고 필요에 따라 파인튜닝이나 수정 가능
            - 규제 및 프라이버시가 중요한 도메인에 특히 유리
            - 라이센스 비용은 없지만 엔지니어링적인 노력이 듬
        - 상용 모델은 API를 통해 강력한 능력과 관리형 인프라 및 모니터링 제공
            - 커스터 마이징에 대한 한계 존재
            - 사용량에 따른 비용 증가
    - 도메인의 특수성과 민감도에 따라 모델 고민(범용 사전학습 모델 or 맞춤형 모델)
        - 범용 사전학습 모델
            - 일반적인 언어 과제 도메인 특화 정밀도가 중요하지 않은 상황에 적합
        - 맞춤형 모델
            - 의료, 법률 등 특수 도메인에서는 맞춤 학습 모델이 상당한 이점 제공
    - 비용과 지연시간도 모델 선택의 결정적 요인
        - 대형 모델
            - 높은 성능 제공
            - 운영 비용 높고, 응답 지연 초래 -> 감내가 어려우면 소형 모델이나 대형 모델 압축형 사용
        - 하이브리드 전략
            - 복잡한 요청은 대형 모델, 일상적인 요청은 소형 모델
            - 동적 모델 라우팅을 통해 각 요청을 평가하고 복잡도에 따라 가장 적합한 모델로 라우팅 -> 품질/비용 최적화
- 일반적으로 더 큰 모델이 더 나은 성능을 보이지만 일관적이지는 않음 -> 높은 성능을 얻기 위해 많은 연산 자원 필요
    - 중간 수준의 성능만 필요하면 훨씬 작은 비용을 달성 가능
    - 자신의 작업을 기준으로 가성비 가장 좋은 모델을 고르는 편이 좋음
- 모델 선택은 지속적으로 재검토해야함
    - 에이전트 역량, 사용자 요구, 인프라 진화가 지속적으로 발생
    - 일반성 vs 전문성, 성능 vs 비용, 단순성 vs 확장성 등 트레이드 오프 비교를 통해 결정

# 2.4 도구
- 에이전트가 특정 작업을 수행하고 문제를 해결할 수 있게 함
- 에이전트 효과는 도구의 범위와 정교함에 달려 있음
### 2.4.1 특정 작업을 해결하는 도구 설계
- 도구를 설계할 때 다양한 조건과 컨텍스트에서 에이전트가 어떻게 작업을 수행할지 고려
- 도구는 크게 세가지로 나눌수 있음
    - 로컬 도구
        - 외부 의존성 없이 내부 로직과 계산만으로 작업 수행
        - 주로 규칙 기반이거나 사전 정의된 함수 실행
    - API 기반 도구
        - 외부 서비스나 데이터 소스와 상호 작용
        - 실시간 데이터를 조회하거나 외부 시스템을 활용해 로컬 환경 너머의 기능 수행
    - MCP
        - 표준 스키마를 통해 사용자 프로필, 대화 이력 같은 구조화된 실시간 컨텍스트를 프로픔트에 직접 전달
        - MCP 스키마를 통해 구조화된 컨텍스트를 제공하고 도구 호출 표준 방식 지원
```
API 기반 도구와 MCP 차이
- API 기반 도구: 각 앱마다 외부 호출에 대한 하드 코딩 후 호출
- MCP: 독립적으로 만들어 표준 스키마를 통해 연결
```
### 2.4.2 도구 통합과 모듈성
- 도구 개발에서 모듈형 설계는 필수 -> 에이전트 기능 확장과 업데이트 고려
    - 쉽게 통합하거나 교체할 수 있도록 설계

# 2.5 메모리
- 에이전트가 정보를 저장하고 검색할 수 있게 하는 핵심 구성 요소
    - 컨텍스트 유지
    - 과거 상호작용에서 학습하여 더 나은 의사결정 내림
### 2.5.1 단기 메모리
- 에이전트가 현재 작업이나 대화와 관련된 정보를 저장하고 관리하는 능력
    - 컨텍스트를 유지하는데 사용 -> 사용자 UX 향상
- 종종 롤링 컨텍스트 윈도 방식을 통해 구현
    - 오래된 데이터를 버릴수 있음
    - 세부 정보는 잊어도 되는 애플리케이션에 유용
### 2.5.2 장기 메모리
- 에이전트가 오랜 기간 걸쳐 지식과 경험을 저장하도록 하여 과거 정보를 바탕으로 향후 행동 결정
    - 사용자 선호도에 기반한 개인화된 경험 제공에 유용
- 주로 데이터베이스, 지식 그래프 또는 파인튜닝된 모델을 통해 구현
    - 사용자 선호도 같은 구조화된 데이터를 저장하고 필요할 때 이를 검색
### 2.5.3 메모리 관리 및 검색
- 효과적인 메모리는 저장된 데이터를 체계적으로 구성하고 인덱싱해 필요할 때 쉽게 검색할 수 있게 하는 것
    - 에이전트가 관련 있는 데이터와 관련 없는 데이터를 구분하고 필요한 정보를 신속하게 찾아야 원활한 성능 유지
    - 경우에 따라 오래되거나 불필요한 세부정보가 메모리에 쌓이는 것을 방지

### 2.6 오케스트레이션
- 개별 기능을 엔드투엔드 솔루션으로 전환
    - 여러 스킬을 배치하고 상황에 따라 실행
    - 단계별 자연스럽게 이어지며 명확한 목표를 이루어지게 전체 과정을 감독
- 오케스트레이션 핵심은 도구나 스킬을 호출 가능한 순서를 평가하고 결과를 예측하며 여러단계를 거치는 작업에서 성공 가능성이 높은 경로를 선택

### 2.7 설계 트레이드오프
- 에이전트 기반 시스템은 아래 조건을 다양한 트레이드오프로 균형 있게 조정 필요
    - 성능
    - 확장성
    - 신뢰성
    - 비용
##### 2.7.1 성능: 속도와 정확도의 균형
- 중요 트페이드오프 하나는 속도와 정확도의 균형
    - 자율주행 차량 또는 트레이딩 시스템은 실시간 의사결정 중요
    - 법률 분석이나 의료 진단은 정확성이 중요
- 하이브리드 접근 방법도 존재
    - 빠르고 대략적인 결과를 제시한 뒤, 추가로 시간과 데이터를 활용하여 정교하게 보완하는 방식
        - 추천 시스템이나 진단 시스템에서 해당 방식 활용
##### 2.7.2 확장성: 에이전트 시스템의 엔지니어링적 확장
- 딥러닝 모델과 실시간 처리에 크게 의존하기에 확장성은 중요 과제
    - 시스템이 커질수록 데이터량, 동시 처리 작업 수, 연산 리소스(GPU)의 효율적인 관리가 핵심
        - GPU는 학습과 추론을 가속화하는 기반이지만 효율적인 확장에서 병목 현상, 자원 낭비, 비용 상승을 방지하는 정교한 엔지니어링 필요
- GPU는 가장 비싸고 제약이 많은 자원이기에 효율적으로 활용하는 것이 최우선 과제
    - 동적 GPU 활동: 실시간 수요에 따라 GPU 배정 -> 유휴시간 줄이고 활용도 극대화
    - 탄력적인 GPU 프로비저닝: 클라우드 GPU 클러스트를 활용하여 리소스 확장/축소
    - 우선순위 큐잉: 중요 작업에 GPU 권한 부여
- 대규모 에이전트의 지연시간 이슈
    - 실시간 또는 준실시간 환경에서 최소 지연이 되도록 GPU를 효율적 스케쥴링 필요
- 효과적인 전력에는 비동기 작업 실행
    - GPU 작업을 병렬 처리해 GPU 자원 활용도 극대화
- 동적 로드 밸런싱
    - 작업을 분산하여 병목현상이 안나게 처리
- 수평 확장
    - 시스템 처리 능력을 높이기 위해 GPU 노드를 추가
- 하이브리드 클라우드 방식
    - 온프레미스 GPU와 클라우드 GPU를 결합하여 버스트 스케일링 처리
- GPU 리소스에 크게 의존하는 에이전트 시스템의 확장은 GPU 효율을 극대화, 지연시간 최소화, 동적인 워크로드를 안정적으로 처리할 수 있게 균형 잡힌 접근 필요
##### 2.7.3 신뢰성: 견고하고 일관된 에이전트
- 신뢰성은 일정 기간 동안 일관되고 정확하게 작업을 수행할 수 있는 능력 의미
    - 신뢰성 향상은 시스템 복잡성, 비용, 개발 기간 측면에서 절충 필요
- 장애 허용
    - 신뢰성의 핵심 요소
    - 에이전트가 오류나 예기치 못한 사건을 감지하고 비정상 종료나 오작동 없이 이를 처리 필요
    - 중복 구조를 활용 -> 일부 고장나더라도 전체 성능이나 서비스가 중단되지 않도록 보장
- 일관성과 견고성
    - 다양한 시나리오, 입력, 환경에서 일관된 성능 유지 필요
        - 철저한 테스트: 단위 테스트. 통합 테스트, 실제 환경 시뮬레이션 테스트 등 다양한 검증 과정 필요
        - 모니터링과 피드백 루프: 이상 징후 지속적 탐지, 피드백 루프로 학습하고 성능 향상
##### 2.7.4 비용: 성능과 지출의 균형
- 자주 간과 되지만 매우 중요한 고려 요소
- 기대되는 이익과 투자 수익률을 기준으로 평가
- 개발 비용
    - 고도화된 에이전트를 개발하는 과정은 비용이 많이 듬
        - 복잡한 에이전트 구축에 데이터 과학자, 머신러닝 엔지니어 등 구성된 팀 필요
        - 방대한 테스트 인프라 필요
- 운영 비용
    - 배포 이후 운영 비용 고려
        - 특히 실시간 의사결정이나 지속적인 데이터가 필요한 시스템 경우 고성능 연산 필요
    - 방대한 데이터를 처리하거나 대규모 메모리를 유지하는 에이전트는 비용을 많이 잡아 먹음
    - 장기적으로 시스템의 신뢰성과 성능 보장을 위한 인적, 기술적 자원 투입이 불가피
- 비용 대비 가치
    - 시스템의 비용은 가치로 정당화 되어야 함
        - 중요도가 낮은 작업은 저비용 에이전트
        - 멈추거나 중단되면 안되는 작업이라면 복잡하고 고성능의 에이전트
    - 비용에 대한 모든 의사결정은 시스템의 전체 목표와 예상 수명을 고려
    - 비용 최적화 전략
        - 경량 모델
            - 규칙 기반 시스템이 유사 성능을 낼수 있다면 단순한 접근이 훨씬 비용 효율적
        - 클라우드 기반 리소스
            - 클라우드 컴퓨팅 활용하여 초기 구축비용 줄이고, 사용량 기반 과금 모델을 통해 확장성과 유연성 확보
        - 오픈소스 모델과 도구
            - 오픈소스 활용으로 개발 비용을 줄이면서 높은 품질의 에이전트 구현 가능
- 에이전트 시스템 설계는 여러 중요한 트레이드오프를 조율하는 과정
    - 성능 <-> 정확도, 멀티에이전트 구조 확장 <-> 일관성
    - 신뢰성 확보를 위한 철저한 테스트와 모니터링 <-> 개발 시간과 복잡도 높아짐

### 2.8 아키텍처 디자인 패턴
- 아키텍처 선택은 확장성, 유지보수성, 유연성에 직접적인 영향을 미침
##### 2.8.1 단일 에이전트 아키텍처
- 가장 단순하고 직관적인 설계로 하나의 에이전트가 시스템 내 모든 작업을 관리하고 실행
- 명확히 정의된 좁은 범위의 작업에 적합, 하나의 엔티티로 처리할 수 있는 업무에 이상적
    - 분산 처리가 필요하지 않고, 좁은 범위의 업무에 유용
    - 문제 도메인이 명확하고 작업이 단순하며, 대규모 확장이 필요하지 않은 환경에서 효과적
    - 예) 고객 지원 챗봇, 범용 어시스턴트, 코드 생성 에이전트
##### 2.8.2 멀티 에이전트 아키텍처: 협업, 병렬성, 조율
- 여러 에이전트가 공동의 목표를 달성하기 위해 협력
- 작업의 성격에 따라 독립적, 병렬적, 조율된 방식으로 작동
- 복잡한 환경에서 유용
    - 작업의 여러 측면에서 전문화된 에이전트가 분담하거나 병렬 처리를 통해 효율성과 확장성을 높일 수 있음
    - 작업이 복잡하거나 분산되어 있거나, 구성 요소별 전문화가 필요한 환경에 적합
- 장점
    - 협업과 전문화: 각 에이전트는 특정 작업이나 영역에 특화되도록 설계 가능
    - 병렬성: 여러 에이전트로 여러 작업을 동시에 수행 가능
    - 확장성 향상: 새로운 에이전트 추가를 통한 부하 분산
    - 중복성과 복원력: 여러 에이전트를 통해 일부 작업이 실패하더라도 전체 시스템 중단 되지 않음
- 단점
    - 조율과 통신의 복잡성: 에이전트 간의 통신 효율적 관리 필요
    - 복잡성 증가: 설계, 개발, 유지보수가 훨씬 어려움
    - 효율성 저하: 토큰 소비가 많아 효율이 낮아질 수 있음
### 2.9 모범 사례
- 알맞는 모델, 스킬, 아키텍쳐만으로 충분하지 않음 -> 개발 라이프사이클 전반에 걸쳐 모범 사례에 따라야 함
##### 2.9.1 점진적 설계
- 초기부터 완벽한 결과물을 목표 하기보다 작은 프로토타입을 만들고 피드백을 반영해 여러 차례 개선
    - 문제의 조기 발견: 성능 병목을 빠르게 찾아 개발비용을 줄이고 대규모 리팩토링을 줄임
    - 사용자 중신 설계: 빈번한 피드백을 받아 사용자 요구와 기대에 잘 맞춤
    - 확장성: 관리 가능한 담위로 점진적 성장 가능
- 다음과 같은 원칙 추천
    - 프로토타입을 빠르게 개발: 초기 핵심 기능 구현 집중
    - 테스트하고 피드백을 수집: 피드백 수집
    - 개선하고 반복: 피드백과 성능 데이터로 개선 사이클
##### 2.9.2 평가 전략
- 견고한 평가 전략 필요
    - 정확도, 효율성, 견고성, 확장성 등 여러 차원에서 에이전트를 체계적으로 테스트 하고 검증
    - 에이전트 기능의 모든 픅면을 포괄하는 종합 테스트 프레임워크를 수립하는 일로 시작 -> 예상치 못한 시나리오 철저히 검증
- 주요 초점
    - 정확성: 일관되게 정확하고 기대한 출력 제공 유무
    - 경계 테스트: 엣지 케이스를 에이전트가 어떻게 처리하는 평가
    - 작업별 지표: 도메인의 정확도와 컴플라이언스 요구사항을 충족하는지 확인
- 실제 사용자로 피드백 수집하면 중요한 인사이트를 얻을 수 있음
    - 사용자 만족도 점수: 사용자 상호작용에 대한 만족도 측정
    - 작업 완료율: 에이전트 도움으로 작업을 성공적으로 완료하는 비율 측정
    - 명시적 신호: 엄지 올림/내림, 별점 등 사용자가 피드백을 제공할 기회를 만듬
    - 암묵적 신호: 오해, 지연, 감정, 부적절한 응답 등 공통 문제 지점 식별
##### 2.9.3 실환경 테스트
- 실제 사용자나 환경에 상호작용할 때, 기대한 대로 작동하는지 확인 필요
    - 에이전트가 라이브 환경의 예측 불가능성과 복잡성을 처리할 수 있도록 보장 필수
    - 엣지 케이스, 예상치 못한 사용자 입력, 높은 부하에서의 성능을 드러냄
- 현실 조건에서의 성능을 검증해 에이전트의 배포 준비 상태를 보장
    - 단계적 배포: 소규모 테스트로 전체 배포로 확장
    - 에이전트 작동 방식 모니터링: 작동 방식, 응답, 성능 지표 추적
    - 사용자 피드백 수집: 상호작용 경험에 대한 피드백 수집
    - 인사이트에 따른 반복: 인사이트를 개발 주기로 돌림

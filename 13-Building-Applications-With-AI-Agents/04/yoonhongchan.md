# 4. 도구
- AI 에이전트가 추가 정보와 컨텍스트를 검색하고 작업을 수행 -> 환경과 의미 있는 상호작용
    - 에이전트가 원하는 결과를 달성하기 위해 수행하는 특정 기능 또는 일련의 작업
- 도구에 대한 종합적인 이해를 제공하는게 해당 장의 목표
- 도구의 설계와 구현은 AI 에이전트의 전반적인 기능과 효과를 좌우
# 4.1 랭체인 기초
- 랭체인 중심에는 프롬프트를 처리하고 응답을 생성하는 파운데이션 모델을 감싼 채팅 모델이 존재
- init_chat_model()를 통해 채팅 모델 인터페이스로 초기화 가능
    - `llm = init_chat_model(model="gpt-5-mini", temperature=0)`
- `@tool`을 통해 도구 정의
- `.bind_tools()`를 통해 도구를 에이전트에 바인딩
    - `llm_with_tool = llm.bind_tools(tools=[tool_1, tool_2])`
- `invoke()`함수를 통해 모델에게 메시지 전달
    - `llm_with_tool.invoke("사용자 발화")`
##### 4.1.1 로컬 도구
- 로컬에서 실행하는 도구, 특정 작업에 맞춘 사전 정의 규칙과 로직을 따라 만듬
    - 수정하기 쉽고 에이저트와 함께 배포
    - 산술, 시간대 변환, 캘린더 작업, 지도와의 상호작용
    - 로직이 명시적으로 정의되므로 대체로 예측 가능하고 신뢰할 수 있음
- 로직만큼 메타데이터(도구 이름, 설명, 스키마)도 중요
    - 해당 메타데이터로 어떤 도구를 호출할지 결정
        - 범위를 좁게 잡고 정확한 이름 선택 -> 이름이 일반적이면 필요하지 않은 상황에서도 LLM이 도구 선택
        - 명확하고 구별되는 설명 작성 -> 여러 도구에 걸쳐 지나치게 넓거나 겹치는 설명은 혼란 및 성능 저하
        - 업격한 입력/출력 스키마 정의 -> 도구를 언제, 어떻게 사용할지 정확히 이해하도록 도움
- 단점
    - 확장성
        - 설계, 구축, 배포가 번거롭고 전반에 걸쳐 공유하기 어려움 -> 대규모로는 어려움
    - 중복
        - 모든 팀이나 에이전트에서 함계 같은 라이브러리 배포 필요
    - 유지보수
        - 환경이나 요구사항이 바뀌면 업데이트와 조정이 필요
- 실행될 도구만 바인딩하는 것은 에이전트를 구축하는 개발자에게 중요한 책무
##### 4.1.2 API 도구
- 에이전트가 외부 서비스와 상호작용하도록 하여 로컬에서 수행하기 어려운 데이터 처리나 액션 실행
    - API를 통해 공용 또는 사설 서비스와 통신
    - 기능을 동적으로 확장하고 확장성을 높이는 수단을 제공
    - 내부에서 처리하기엔 자원 소모가 큰 복잡한 연산을 수행해야 하는 상황에서 특히 유용
- API 도구를 설계할 때는 신뢰성, 보안성, 우아한 실패 중요
    - 폴백 매커니즘이나 명확한 오류메시지 제공 필요
    - 보안 프로토콜 준수
    - 요청 제한을 주의하여 서비스 중단 방지
    - 개인 정보 보호법을 준수하기 위해 마스킹 필요
##### 4.1.3 플러그인 도구
- 모듈화되어 있으며 최소한의 커스터마이징만으로 AI 에이전트 프레임워크에 통합 가능한 구성 요소
    - 기존 라이브러리, API, 서드파트 서비스를 활용해 개발 작업 없이 에이전트 기능 확장
    - AI 주요 플랫폼과 오픈소스 커뮤니티는 플러그인 도구 통합 기능을 표준 기능으로 제공
    - 대규모 맞춤 개발 없이도 AI 에이전트 역량을 확장하는 강력한 수단
    - 예) 클로드는 도구 사용 기능 전체를 직접 제공 -> 커스텀 툴을 등록하거나 엔트로픽이 제공하는 도구 사용 가능
- 모델 실행 계층에서의 통합 가능성
    - 기존 워크플로에 거의 영향을 주지 않고 AI 모델 추가 가능 -> 새로운 기능을 신속히 도입해야할 때 유리함
- 단점
    - 범용적으로 설계되었기 때문에, 맞춤형 툴처럼 높은 수준의 커스터마이징과 적용성을 제공하지 않음
- 주요 플랫폼이 제공하는 도구 카탈로그는 빠르게 확장되고 있음
    - 추후 플러그인 도구가 더욱 전문화되고 고도화된 기능을 포함
- 사용 예제
    - 자연어 이해 및 감정 분석 툴을 통해 고객의 문제를 파악하고 적절히 대응 -> 고객 만족도와 응답 속도 개선
##### 4.1.4 MCP
- 앤트로픽이 처음 제안한 표준으로 독립적인 통합 표준으로 LLM과 외부 시스템 연결하는 프로토콜
- 구성 요소
    - MCP 서버
        - JSON-RPC 2.0 인터페이스를 통해 데이터나 서비스를 노출하는 웹 서버
    - MCP 클라이언트
        - MCP를 사용하는 에이전트 또는 LLM 애플리케이션
- 동작 방식
    - 내부적으로 HTTPS나 WebSocket에서 Json-RPC 2.0을 사용
    - 서버는 자신이 제공하는 메소드와 입출력 스키마는 공개
    - 클라이언트는 서버에게 '메서드 카탈로그' 조회하여 어떤 메소드와 어떤 파라미터를 호출할 수 있는지 추론
- MCP 이전에는 각 대상 시스템 별로 커스텀 어댑터 작성 -> 통합코드가 어려워지고 유지보수가 힘듬
- 아직까지는 인증, 접급 제어, 잠재적인 공격 벡터 이슈 존재
##### 4.1.5 상태 유지 도구 [* DB와 같은 상태를 조작할 수 있는 도구를 의미하는 듯함]
- 지속적인 상태를 직접 조작할 권한을 부여하면 모델이 파괴적인 실수를 하거나 악성 사용자가 악용할 수 있음
    - 임의에 SQL을 노출하는 대신 범위를 좁게 정의된 작업만 도구로 등록 필요
    - 만약 자유형 쿼리가 필요하다면 철저한 데이터 정화와 접근 제어를 구현 필요
        - 프롬프트 인젝션에 DROP이나 ALTER 같은 위험한 구문이 끼어들수 있다고 경고 필요
- 궁극적으로 '최소 권한 원칙'이 설계의 기준이 되어야함
# 4.2 도구 개발 자동화
- AI 에이전트가 자율적으로 코드를 작성해 애플리케이션 개발과 유지보수하는데 시간과 노력을 크게 줄이는 기술
    - 모델을 학습시켜 언어와 패턴, 모범 사례를 이해하도록 함
##### 4.2.1 파운데이션 모델을 활용한 도구 개발
- 파운데이션 모델은 도구 사용을 넘어서 도구를 생성
- 복잡한 API 환경을 다룰 때 특히 유용
    - 수십개의 마이크로서비스 클라이언트를 직접 작성하는 것보다 모델을 통해 처리
- 명확한 검증 기준(테스트, 응답 검증, 스키마 강제)과 개발자의 감독이 필수
##### 4.2.2 실시간 코드 생성
- AI 에이전트가 필요에 따라 코드를 작성하고 실행하는 기능
    - 에이전트가 현재 작업을 분석하고 이를 수행하기 위한 필요한 단계를 결정
- 즉각적인 요구사항을 처리할 수 있어 프로세스 속도를 높이고 다운타임을 줄이며 전체 효율성을 크게 향상 시킴
- 단점
    - 품질 관리: 품질과 보안을 보장하지 못하면 시스템 오류, 보안 침해 등 심각한 문제를 발생
    - 재현성: 프롬프트나 모델 업데이트의 미세한 변화로도 전혀 다른 코드 경로가 만들어질 수 있음
    - 리소스 소비: 많은 연산 자원과 메모리를 요구함
# 4.3 도구 사용 설정
- 'tool-choice'를 통해 유연한 호출 방식에서 결정적 기능으로 전환 가능 -> 유연성 <-> 구조적 제약을 통한 안전성과 예측 가능성
    - auto 모드: 도구 호출 여부를 스스로 결정
    - any 또는 required 모드: 최소 한개의 도구를 반드시 호출하도록 강제
    - none 모드:모든 도구 호출이 차단
- 도구 호출을 건너뛰거나, 오류가 발생한 도구를 실행하는 등 오작동을 보일수 있으므로 폴백 매커니즘이나 후처리 체계 필요
    - 모델의 응답 후 문제 발견 시 다음과 같은 교정 절차 적용
        - 스키마 검증: jsonschema를 사용해 누락된 필드나 잘못된 구조 탐지, 모델에 수정 프롬프트를 보내기
        - 지능적 재시도: 일시적 오류는 지수 백오프 같은 재시도 로직 적용 또는 부분 재시도
        - 안정적인 폴백: 재시도 실채하면 백업 모델이나 서비스를 사용, 안전한 기본 값을 반환
        - 로깅: 검증 오류, 재시도, 폴백 등 관측 가능성 확보 필요

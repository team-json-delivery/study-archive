# CHAPTER 4 Tool Use
### 1. 도구(Tool)의 정의와 중요성
* **도구의 개념**: AI 에이전트가 추가적인 정보와 컨텍스트를 검색하고, 작업을 수행하며, 환경과 의미 있는 방식으로 상호작용할 수 있도록 돕는 빌딩 블록입니다.
* **행동의 매개체**: 에이전트가 단순한 정보 검색을 넘어 실질적인 변화를 외부 환경에 적용하고자 할 때, 도구는 그 변화를 실행하는 핵심 수단이 됩니다.
* **모듈식 구성**: 도구는 독립적으로 개발, 테스트, 최적화가 가능한 모듈식 컴포넌트이며, 이들이 결합되어 복잡한 동작을 수행하는 시스템을 형성합니다.

### 2. LangChain Fundamentals (LangChain 기본 개념)
* **핵심 구성 요소**: 프롬프트를 처리하고 응답을 생성하는 기반 모델(Foundation Models)과 대화 모델(Chat Models)이 중심을 이룹니다.
* **메시지 구조**: 대화의 컨텍스트를 유지하기 위해 사용자 입력을 나타내는 `HumanMessage`와 모델의 응답을 나타내는 `AIMessage` 구조를 사용합니다.
* **도구 정의 및 바인딩**: `@tool` 데코레이터를 사용하여 함수를 도구로 정의하고 스키마를 자동 생성하며, `.bind_tools()` 메서드를 통해 모델에 도구를 결합합니다.
* **실행 흐름**: 모델이 `invoke()` 과정을 통해 도구 호출(`tool_calls`)을 결정하면, 에이전트는 해당 함수를 실행하고 그 결과를 대화에 다시 추가하여 최종 응답을 생성합니다.

### 3. Local Tools (로컬 도구)
* **특징**: 에이전트와 함께 배포되어 사전 정의된 규칙과 로직에 따라 로컬에서 실행되는 도구입니다.
* **장점**: 산술 연산, 시차 변환, 달력 작업, 지도 상호작용 등 LLM이 취약한 전통적인 프로그래밍 영역을 정밀하고 예측 가능하게 보완합니다.
* **메타데이터의 중요성**: 모델이 도구 사용 여부를 정확히 판단할 수 있도록 명확하고 좁은 범위의 이름, 구체적인 설명, 엄격한 입출력 스키마를 정의해야 합니다.
* **단점**:
    * **확장성(Scalability)**: 여러 유스케이스 간에 도구를 공유하고 대규모로 관리하기가 어렵고 번거롭습니다.
    * **중복(Duplication)**: 도구 변경 시 각 에이전트 서비스의 배포를 조율해야 하므로, 많은 팀이 독립적으로 도구를 재구현하는 비효율이 발생합니다.
    * **유지보수(Maintenance)**: 환경 변화에 따라 빈번한 업데이트가 필요하며, 이때마다 에이전트 서비스를 재배포해야 합니다.

### 4. API-Based Tools (API 기반 도구)
* **특징**: 외부 또는 내부의 API를 레버리지하여 실시간 데이터에 접근하고 복잡한 연산을 수행합니다.
* **장점**: 모델을 재학습시키지 않고도 날씨, 금융 데이터, 번역 서비스 등 방대한 외부 서비스를 연동하여 에이전트의 작업 범위를 극적으로 확장할 수 있습니다.
* **설계 시 고려사항**: 외부 서비스 장애에 대비한 폴백(Fallback) 마련, HTTPS 및 강력한 인증을 통한 보안 확보, API 속도 제한(Rate limits) 관리, 데이터 프라이버시 법률 준수를 위한 데이터 익명화 등이 필수적입니다.

### 5. Plug-In Tools (플러그인 도구)
* **특징**: 기존 라이브러리나 제3자 서비스를 활용하여 최소한의 맞춤화 작업만으로 신속하게 통합할 수 있는 사전 설계된 모듈입니다.
* **주요 플랫폼별 특성**:
    * **OpenAI**: 풍부한 플러그인 생태계를 갖추고 있으나 ChatGPT 제품 내부에서만 사용 가능하며, 퍼블릭 API를 통해서는 직접 호출할 수 없습니다.
    * **Anthropic (Claude)**: API(Messages API) 레이어에서 직접 커스텀 도구 사용 기능을 노출하는 'API 퍼스트' 접근 방식을 취합니다.
    * **Google (Gemini)**: Vertex AI API의 `FunctionCallingConfig`를 통해 함수를 선언하고 구조화된 호출을 처리할 수 있도록 지원합니다.
    * **Microsoft (Phi)**: Azure AI Foundry를 통해 인지 검색, 문서 처리 등 다양한 Azure 서비스와 매끄럽게 결합됩니다.
* **트레이드 오프**: 모델 실행 레이어에 간편하게 결합되어 빠른 배포가 가능하지만, 자체 개발 도구에 비해 세부적인 커스터마이징이나 애플리케이션 특화 조율이 어렵다는 한계가 있습니다.
* **오픈소스 생태계**: Hugging Face Transformers 라이브러리 등을 통해 오픈소스 기반 모델에서도 플러그인 도구를 활발히 활용하고 있으며, Glama.ai나 mcp.so와 같은 플랫폼을 통해 도구 검색 및 발견이 용이해지고 있습니다.

### 6. Model Context Protocol (모델 컨텍스트 프로토콜, MCP)
* **개념**: Anthropic이 도입하고 주요 AI 기업들이 채택한 개방형 표준으로, LLM을 외부 시스템에 연결하는 유연하고 모델 아그노스틱(Model-agnostic)한 인터페이스입니다. 에이전트와 도구를 연결하는 'AI의 USB-C 포트' 역할을 합니다.
* **핵심 역할 모델**:
    * **MCP 서버**: 표준화된 JSON-RPC 2.0 인터페이스를 통해 데이터나 서비스(데이터베이스, 클라우드 스토리지 등)를 노출하는 웹 서버입니다.
    * **MCP 클라이언트**: MCP 규격을 따르는 에이전트 혹은 LLM 애플리케이션으로, 서버에 구조화된 JSON-RPC 요청을 보냅니다.
* **장점**: 과거처럼 각 대상 시스템마다 불안정하고 유지보수가 어려운 커스텀 어댑터를 개별적으로 작성할 필요 없이, 한 번 노출된 MCP 서비스를 여러 에이전트가 재사용할 수 있어 개발 비용이 대폭 절감됩니다.
* **과제**: 인증, 접근 제어, 페이로드 주입 공격 방지, 감사 로그 유지 등 보안 관점의 표준화 솔루션은 아직 완벽히 정립되지 않아 활발한 연구와 엔지니어링이 진행 중인 영역입니다.

### 7. Stateful Tools (상태 저장 도구)
* **위험성**: 영구적인 데이터 상태를 변경할 수 있는 직접적인 권한을 모델에 부여할 경우, 모델의 오작동이나 프롬프트 주입(Prompt Injection) 공격 등으로 인해 데이터가 영구 삭제되거나 훼손되는 파괴적인 실수가 발생할 수 있습니다.
* **방어 대책**:
    * **최소 권한 및 범위 제한**: 임의의 SQL을 실행하는 엔드포인트를 열어두지 말고, `get_user_profile`처럼 단일 기능만 수행하도록 도구의 범위를 좁히고 읽기 권한 위주로 제한해야 합니다.
    * **입력값 검증 및 보안**: 프리폼 쿼리가 불가피하다면 DROP, ALTER와 같은 위험 구문을 거르는 엄격한 입력값 검증을 시행하고, 매개변수 바인딩(Prepared Statements)을 사용해야 합니다.
    * **관찰 가능성(Observability)**: 모든 도구 호출을 로깅하고, 비정상적으로 큰 규모의 삭제나 스키마 변경 명령이 감지되면 실시간 알림을 보내 인간이 개입할 수 있는 시스템을 갖춰야 합니다.

### 8. Automated Tool Development (자동화된 도구 개발)
* **Foundation Models as Tool Makers (도구 제작자로서의 기반 모델)**: 모델에 API 규격이나 샘플 입력을 제공하여 스스로 래퍼 함수를 초안하게 하고, 샌드박스 환경에서 실행한 뒤 에러 피드백을 통해 코드를 스스로 정제하도록 만드는 방식입니다. 인간 리뷰어의 검증을 거쳐 CI/CD 파이프라인에 반영하면 개발 효율을 극대화할 수 있습니다.
* **Real-Time Code Generation (실시간 코드 생성)**: 에이전트가 미지의 API나 새로운 문제를 만났을 때 가동 중에 즉석에서 코드를 작성하고 실행하는 방식입니다. 뛰어난 적응성과 효율성을 제공하지만 다음과 같은 명확한 리스크가 있습니다:
    * **품질 및 보안**: 저품질 코드로 인한 시스템 실패나 해커가 악성 코드를 주입할 수 있는 보안 취약점이 발생할 수 있습니다.
    * **반복 가능성(Repeatability) 저하**: 매번 코드를 새로 작성하므로 성공과 실패의 예측 가능성이 떨어지고, 프롬프트나 모델 업데이트에 따라 완전히 다른 코드 경로를 타게 되어 디버깅과 컴플라이언스 준수가 어려워집니다.
    * **자원 소비**: 비효율적인 코드가 생성되고 실행될 경우 상당한 연산 능력과 메모리를 소모하게 됩니다.

### 9. Tool Use Configuration (도구 사용 설정 및 오류 관리)
* **도구 선택 파라미터 제어**: 모델 API의 `tool-choice` 매개변수를 제어하여 유연성과 예측 가능성의 균형을 맞춥니다.
    * **auto**: 컨텍스트에 따라 모델이 자율적으로 도구 호출 여부를 결정합니다.
    * **any / required**: 모델이 최소 하나 이상의 도구를 강제로 호출하도록 제한합니다.
    * **none**: 도구 호출을 차단하여 텍스트만 출력하게 만듭니다(테스트 환경에 유용).
* **오류 관리 및 폴백(Fallback)**: 프로덕션 등급의 안정적인 에이전트를 빌드하기 위해 사후 프로세싱 메커니즘을 두어야 합니다.
    * **검증(Validate)**: Pydantic 등을 이용해 도구가 출력한 결과가 올바른 스키마를 가졌는지 검증하고, 잘못되었다면 모델에게 수정을 요청합니다.
    * **지능적 재시도(Retry)**: 일시적인 네트워크 실패 등의 문제에는 지수 백오프(Exponential backoff) 등의 논리를 사용해 지능적으로 재시도합니다.
    * **우아한 폴백(Fall back)**: 재시도가 최종 실패할 경우 백업 모델로 전환하거나, 유저에게 재질의를 하거나, 캐시된 데이터를 사용하거나, 안전한 디폴트 값을 반환합니다.
    * **로깅(Log)**: 디버깅과 지속적인 개선을 위해 프롬프트, 도구 호출, 에러, 재시도 등 모든 과정을 투명하게 기록합니다.

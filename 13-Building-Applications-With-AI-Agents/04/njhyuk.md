# Chapter 4. Tool Use (도구 사용)

## 1. 도구(Tool)란 무엇이고 왜 필요한가

- 파운데이션 모델은 “대화”는 잘하지만, 실제 정보를 가져오거나 행동을 실행하려면 **도구(Tool)**가 필요하다.
- 도구 = 에이전트가 원하는 결과를 얻기 위해 수행할 수 있는 **특정 기능/행동의 집합**
- 단순 1-step 작업부터, 추론이 필요한 multi-step 작업까지 범위가 넓다.
- 특히 단순 검색·답변을 넘어 “실제 변경(action)“을 일으키려면 반드시 도구를 거친다.
- 비유: 의사가 진단·치료를 위해 다양한 도구가 필요하듯, 에이전트도 여러 작업을 처리하려면 도구 레퍼토리가 필요.
- 도구는 **모듈 단위**로 독립 개발·테스트·최적화한 뒤 통합 → 복잡한 행동을 만든다.

## 2. LangChain 기초 (Tool 정의 방식)

에이전트가 도구를 쓰려면 모델이 이해할 수 있는 **명세(이름·설명·입력 스키마)**가 필요하다. LangChain은 이를 표준화해준다.

핵심 구성요소

- **name**: 도구 식별자
- **description**: 모델이 “언제 이 도구를 쓸지” 판단하는 근거 → 가장 중요
- **args schema**: 입력 파라미터(타입·필수 여부)

```python
from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """주어진 도시의 현재 날씨를 반환한다."""
    return weather_api.fetch(city)
```

포인트

- description과 파라미터 설명을 명확히 작성할수록 모델의 도구 선택 정확도가 올라간다.
- 도구 = “함수 + 모델이 읽는 설명서”라고 생각하면 됨.

## 3. 도구의 종류

### 3-1. 로컬 도구 (Local Tools)

- 에이전트와 **같은 환경(프로세스)** 안에서 실행되는 함수
- 예: 계산, 문자열 처리, 로컬 파일 읽기, 내부 DB 조회
- 장점: 빠름, 외부 의존성 낮음, 디버깅 쉬움
- 단점: 실행 환경에 묶임, 보안/격리(sandboxing) 고려 필요(특히 코드 실행류)

### 3-2. API 기반 도구 (API-Based Tools)

- 외부 서비스의 **HTTP/REST API**를 호출하는 도구
- 예: 검색 API, 결제, 캘린더, 사내 마이크로서비스
- 고려사항: 인증(키/OAuth), 레이트 리밋, 타임아웃·재시도, 에러 핸들링, 응답 파싱
- 외부 API는 사람(개발자) 기준으로 설계돼 있어, 모델이 쓰기 좋게 **래핑(wrapping)**이 필요한 경우가 많다.

### 3-3. 플러그인 도구 (Plug-In Tools)

- 표준화된 규약에 따라 **꽂아서(plug-in)** 쓰는 외부 기능
- 여러 도구를 일관된 인터페이스로 노출 → 재사용·확장 용이
- 생태계(마켓플레이스) 형태로 도구를 공유·발견할 수 있음

### 3-4. Model Context Protocol (MCP) ⭐

- LLM ↔ 외부 도구/데이터 소스를 연결하는 **개방형 표준 프로토콜**
- 문제의식: API마다 매번 보일러플레이트·프롬프트·툴 정의를 새로 짜야 함 → 비효율
- MCP는 도구·리소스·프롬프트를 **표준 방식으로 서버가 노출**하고, 클라이언트(에이전트)가 일관되게 소비
- 효과: 한 번 MCP 서버로 만들어두면 여러 에이전트/프레임워크에서 재사용 가능
- 구조: MCP Server(도구 제공) ↔ MCP Client(에이전트 측)

### 3-5. 상태 유지 도구 (Stateful Tools)

- 호출 간 **상태(state)**를 유지하는 도구 (vs. 매번 독립 실행되는 stateless 도구)
- 예: 로그인 세션 유지, 장바구니, 다단계 트랜잭션, 페이지네이션 커서
- 주의: 세션 관리, 동시성, 상태 일관성, 만료/정리(cleanup) 설계가 필요

## 4. 자동 도구 개발 (Automated Tool Development)

도구를 사람이 일일이 만드는 대신, **모델이 도구를 만들거나 즉석에서 코드를 생성**하게 하는 접근.

### 4-1. Foundation Models as Tool Makers (도구를 만드는 모델)

- 강력한 모델이 새 도구(함수)를 **설계·생성**하고, 더 가볍고 저렴한 모델이 그 도구를 **사용**하는 분업 패턴
- 장점: 도구 제작 비용 절감, 반복되는 작업의 자동 도구화
- “Tool Maker(제작) ↔ Tool User(사용)” 역할 분리

### 4-2. Real-Time Code Generation (실시간 코드 생성)

- 미리 정의된 도구가 없을 때, 에이전트가 **그 자리에서 코드를 작성·실행**해 작업 수행
- 예: 데이터 분석에서 즉석 파이썬 코드 생성·실행
- 매우 유연하지만 **샌드박싱·보안·검증**이 필수 (임의 코드 실행 위험)

## 5. 도구 사용 설정 (Tool Use Configuration)

모델이 도구를 호출하는 **방식 자체를 제어**하는 설정들.

- **tool_choice**: 도구를 자동(auto)으로 선택할지 / 강제(required)할지 / 특정 도구만 쓰게 할지
- **병렬 호출 허용 여부**: 한 턴에 여러 도구를 동시에 부를지
- **structured output / 스키마 강제**: 인자 형식을 엄격히 통제
- 목적: 안정성·예측가능성 확보, 불필요한 도구 호출/환각 호출 방지

## 6. 핵심 정리 (Key Takeaways)

1. 도구는 에이전트가 “말”을 넘어 “행동·검색”을 하게 만드는 핵심 구성요소다.
1. 도구의 **description / 입력 스키마**가 모델의 선택 정확도를 좌우한다.
1. 도구 유형: Local → API → Plug-in → **MCP(표준화)** → Stateful 순으로 결합도/복잡도가 올라간다.
1. MCP는 도구 재사용성을 크게 높이는 표준이라 특히 중요.
1. 자동 도구 개발(Tool Maker, 실시간 코드 생성)은 강력하지만 **보안·샌드박싱**이 전제.
1. tool_choice 등 설정으로 도구 호출의 **안정성·예측가능성**을 확보한다.
